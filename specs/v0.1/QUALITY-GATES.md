# v0.1 质量门

## 1. 来源忠实门

### 必须通过

- 原文、用户解释、系统推导和外部材料被分开。
- 原材料的重要结论、论证顺序和边界没有因抽取机制而丢失。
- 事实断言没有仅因作者写出就被当成外部事实。
- 文本矛盾和残缺没有被静默修复。

### 一票否决

- 把用户解释写成作者原意。
- 把现代机制倒灌给古代材料。
- 为了形成漂亮机制而删除反例或限定。

## 2. 机制门

### 必须通过

- 有输入、作用、中介过程和可观察结果。
- 成立条件能够帮助判断何时适用。
- 失效边界包含具体场景和原因。
- 新场景测试改变了领域和角色，不只是换名词。

### 一票否决

- 名词串代替因果链。
- “经历痛苦就会成长”等无条件一般化。
- 只写建议，没有结果与机制。
- 卡片只能解释原案例。

## 3. 命名门

### 必须通过

- 标题通过三秒测试。
- 已有术语优先，且与完整机制一致。
- 技术术语有白话解释。
- 设计提案明确标识。

### 一票否决

- 标题读完正文才懂。
- 内部工程术语冒充用户标题。
- 用大词覆盖一个片段。
- 仅靠造名字制造创新感。

## 4. 迁移门

逐项比较：

| 检查项 | 原领域 | 新领域 | 是否一致 |
|---|---|---|---|
| 输入或触发 |  |  |  |
| 参与角色 |  |  |  |
| 中介过程 |  |  |  |
| 因果方向 |  |  |  |
| 直接输出 |  |  |  |
| 成立条件 |  |  |  |
| 失效方式 |  |  |  |

只要关键输出或中介过程不同，默认是近邻机制，不得声称完全同构。

## 5. 关系门

每条正式关系必须具备：

- 合法类型；
- 明确目标；
- 建立理由；
- “不代表什么”或适用边界；
- 证据角色；
- 必要时的反向记录。
- 每条关系只指向一张目标卡，避免一条理由覆盖多个不同连接。

删除理由后仍不影响理解的链接，大概率是噪声。

禁止用“相关”“类似”“让我想到”作为正式关系理由。

## 6. 组合门

### 必须通过

- 组合机制来自可区分的功能角色。
- 说明系统接口，而非只列概念。
- 明确原系统的盲点或矛盾。
- 组合产生新的问题解法、判断或设计能力。
- 说明每个必要组件不可替代的职责。
- 给出应用、失效边界和可推翻条件。

### 一票否决

- 新系统只是组件标题的拼接。
- 删除任意组件后能力不变。
- 只产生新名字，没有新规则或新判断。
- 把待验证设计写成成熟理论。

## 7. 可读性门

抽查一张卡，让审核者只看标题、一句话和机制链：

- 能否复述用途？
- 能否说出什么时候失效？
- 是否必须查阅内部规范才能看懂？

任一答案不合格，卡片退回改写。

## 8. Obsidian 机械门

- 文件名与 H1 一致。
- 标题在对应类型目录中唯一。
- 所有 wikilink 能解析到唯一文件。
- frontmatter 字段合法。
- 必填章节存在且非空。
- 正式关系类型属于允许集合。
- retired 卡保留去向和原因。
- 不强制给真实孤立卡制造链接。

## 9. 回归测试

### 黄金样本 001

必须持续检查：

- 不把慧能解释成反脆弱理论提出者。
- “反脆弱”标题清楚且定义不退化为适应或韧性。
- 混沌工程只作为相关案例，不冒充反脆弱同义词。
- 反自欺实验系统保留四个输入机制及新增能力。
- 21 个 Markdown 文件中的 wikilink 可解析。

### 第二盲测

第二份材料必须：

- 与佛教故事和反脆弱无关；
- 至少包含一个多方立场或价值冲突；
- 存在诱人的表面类比；
- 能测试系统是否拒绝伪机制；
- 能检验模板是否支持另一类组合。

## 10. 发布门

Skill 进入公开使用前必须：

- 两份黄金／盲测全部通过；
- 至少一次“没有合格机制，所以不建卡”的测试通过；
- 至少一次旧卡复用或修正测试通过；
- 至少一次假迁移被拒绝；
- 至少一次跨系统组合被拒绝；
- 至少一次跨系统组合通过并产生新系统；
- 写入失败可以恢复到完整旧状态。
