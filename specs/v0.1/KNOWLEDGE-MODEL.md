# v0.1 知识模型

## 1. 知识单位

知识库包含五种对象：

| 对象 | 作用 |
|---|---|
| 来源还原稿 | 保存材料原意、论证、解释角色和证据风险 |
| 机制卡 | 保存可跨情境调用的最小因果或约束结构 |
| 系统卡 | 保存多个机制组合后产生的新增能力 |
| 专题分析 | 使用机制网络处理一个现实问题 |
| 运行报告 | 记录一次写库改变了什么、为什么改变 |

文章观点、人物、故事、金句和领域名词可以保留在还原稿，但不能仅凭重要性成为机制卡。

## 2. 机制卡资格

候选只有同时回答以下问题才能建卡：

1. 输入或触发条件是什么？
2. 哪些角色、变量或资源参与？
3. 发生了什么作用？
4. 中间通过什么过程传递？
5. 产生什么可观察结果？
6. 在什么条件下成立？
7. 在什么情况下失效？
8. 离开原文后能否检查一个新情境？

最短函数形式：

```text
输入／触发
→ 作用
→ 中介过程
→ 可观察结果
```

以下候选拒绝建卡：

- 只有名词，没有因果或约束；
- 只有建议，没有可观察结果；
- 只是原文案例的换词复述；
- 只能用于当前人物和故事；
- 依靠模糊词掩盖未知过程；
- 无法给出任何失败条件。

## 3. 系统卡资格

系统卡不是“大一点的机制卡”，也不是多张卡的目录。

它必须满足：

```text
多个机制承担不同职责
→ 通过明确接口连接
→ 共同解决单个机制无法解决的矛盾
→ 产生新增的解释、预测、判断或设计能力
```

移除任一被标记为必要的组件后，如果新增能力没有实质下降，这个组件就不是必要组件。

## 4. 标题与命名

### 4.1 三秒测试

目标读者只看标题，应能说出卡片大致讨论什么。需要读完正文才能理解标题，命名失败。

### 4.2 命名优先级

1. 与机制精确一致、读者会检索的已有术语；
2. 已有术语加一句白话限定；
3. 一眼能懂的白话标题；
4. 只有产生新机制且前三层都无法承载时，才允许新造名称。

新造名称必须同时给出一句白话解释，并明确标记“设计提案”。

### 4.3 大词不得冒名

“反脆弱”“贝叶斯更新”“双环学习”等已有系统名，只能在函数、边界和结果匹配时使用。只覆盖其中一个片段时，应建立组成、相关案例或同构候选关系，不能直接改名。

## 5. 卡片元数据

v0.1 使用最小 frontmatter：

```yaml
---
type: mechanism        # mechanism | system
status: draft           # draft | usable | stable | retired
aliases: []
created: 2026-07-14
updated: 2026-07-14
---
```

规则：

- 文件名与 H1 标题完全一致。
- `aliases` 只放真实可检索别名，不复制标题。
- 来源和关系放在正文，避免把复杂证据压进 YAML。
- 不使用读者看不懂的层级编码作为文件名。

## 6. 机制卡正文

固定章节见 [mechanism-card.md](templates/mechanism-card.md)。必需内容包括：

- 一句话；
- 它解决什么问题；
- 怎么起作用；
- 参与角色；
- 什么时候成立；
- 什么时候失效；
- 新场景测试；
- 来源与证据；
- 关系。

## 7. 系统卡正文

固定章节见 [system-card.md](templates/system-card.md)。系统卡额外要求：

- 要解决的矛盾；
- 输入机制及职责；
- 接口和组合链；
- 单个系统为什么做不到；
- 新增能力；
- 具体规则或设计；
- 可推翻条件。

## 8. 证据角色

每项重要内容必须使用以下角色之一：

| 角色 | 含义 | 能否单独支持稳定卡 |
|---|---|---|
| 原文直接表达 | 来源明确陈述或展示 | 否，仍需检查来源性质 |
| 用户解释 | 用户对来源的理解和延伸 | 否 |
| 系统推导 | AI 补出的抽象结构或因果链 | 否 |
| 独立材料支持 | 不同来源实际支持相同结构 | 可以参与支持 |
| 跨域推演 | 把机制角色映射到新领域 | 否 |
| 组合设计 | 多机制组合产生的新系统 | 否，默认是待验证设计 |
| 反例或限定 | 说明适用边界和失败条件 | 用于收窄边界 |

“有引用”不等于“机制已验证”。引用必须支持被引用的具体关系。

## 9. 生命周期

### draft

新提取、证据不足、只有系统推导或组合设计。可以检索，但调用时必须显示不确定性。

### usable

函数和边界清楚，命名通过，至少有独立材料或可靠领域知识支持关键结构，并通过新场景测试。可以日常调用。

### stable

多个独立来源或权威原理与实际应用共同支持；跨域边界经过反例检查；不存在未解决的身份冲突。稳定不等于永远正确，仍可被新证据降级。

### retired

机制被合并、名称错误、边界无法成立或被证据推翻。文件保留去向和原因，防止未来重复创建。

## 10. 关系模型

### 10.1 正式关系

| 类型 | 含义 |
|---|---|
| 组成 | A 是 B 发挥新增能力所需的组件 |
| 前置 | 调用 B 前必须先具备 A |
| 中介 | A 承担 B 因果链中的传递环节 |
| 约束 | A 限制 B 的范围、风险或解释 |
| 对照 | A 与 B 回答相同问题但采用不同策略 |
| 案例验证 | 某领域材料实际承载该机制 |
| 反例限定 | A 说明 B 在某范围内失效 |
| 相关案例 | A 只承载 B 的一部分结构 |
| 组合 | A 与其他机制共同产生新系统 |

### 10.2 候选关系

“同构候选”和“可能相关”放在独立的候选区，不进入正式双链。只有核验角色、因果方向、输出和边界后才升级。

### 10.3 关系记录格式

```text
`关系类型` → `[[目标卡]]`：理由：为什么必须连接；边界：这条连接不代表什么；证据：证据角色。
```

一条正式关系只能指向一张目标卡。缺少类型、理由、边界或证据角色时，不得写入。

### 10.4 双向维护

正式关系写入后，Skill 必须检查目标卡是否需要反向记录。反向关系可以采用不同类型和解释，不能机械复制箭头。

## 11. 复用与去重

复用旧卡需要同时比较：

- 标题和别名；
- 输入与输出；
- 中介过程；
- 参与角色；
- 成立条件；
- 失效边界；
- 未来调用方式。

处理顺序：

1. 完全承载：复用并补证据。
2. 同一机制但边界过窄：扩大旧卡并保留变更记录。
3. 旧卡过宽：拆分并标明去向。
4. 名称相同但机制不同：分开并在标题中增加白话限定。
5. 近义重复：合并，旧卡转为 retired。
6. 无法承载且确有独立价值：新建 draft。

知识库为空不能作为新建理由；必须说明候选本身为什么有复用价值。
