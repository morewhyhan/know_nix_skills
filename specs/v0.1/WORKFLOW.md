# v0.1 工作流

## 总览

```text
意图识别
→ 来源预检
→ 来源还原
→ 全库检索
→ 机制候选
→ 命名与证据核验
→ 单卡建模
→ 关系审查
→ 跨系统组合
→ 整批写入
→ 质量验收
→ 用户报告
```

## 阶段 1：意图识别

先判断用户要沉淀、调用、组合还是审计。

- 没有写库授权时，不修改知识库。
- 用户明确要求维护或沉淀时，允许更新相关卡和关系。
- 组合探索产生新系统时，默认先作为 draft 保存。

## 阶段 2：来源预检

1. 完整读取材料，不先看关键词抽取。
2. 标记原始材料、用户解释、任务和外部材料。
3. 记录来源路径、标题、作者、日期和可访问性。
4. 标出文本残缺、内部矛盾、观点性断言和事实风险。
5. 不静默补全缺失段落。

## 阶段 3：来源还原

按照 [source-reconstruction.md](templates/source-reconstruction.md) 写还原稿。

还原稿必须让没有看过原文的人回答：

- 原材料说了什么？
- 它怎样推出结论？
- 哪些是作者判断？
- 用户增加了什么？
- 哪些部分不能推出？
- 哪些内容进入机制库，哪些只留在还原稿？

先还原，后抽取。不能因为某部分不能建卡就从还原稿删除。

## 阶段 4：全库检索

每个候选至少经过五路检索：

1. 标题和别名；
2. 输入、作用和输出；
3. 中介过程；
4. 成立条件与失效边界；
5. 可能的上游、下游、组件和系统组合。

检索目的不是找到“有点像”的卡，而是决定复用、修正、拆分或新建。

## 阶段 5：机制候选

逐段扫描，但只保留能写成完整函数的候选。

对每个候选记录：

- 来源定位；
- 候选标题；
- 函数链；
- 证据角色；
- 可能复用的旧卡；
- 主要未知点。

候选清单是内部过程，不要求用户逐项审批。

## 阶段 6：命名与证据核验

1. 先查已有术语和用户会搜索的说法。
2. 检查术语是否覆盖完整函数，而不只是结果相似。
3. 使用专业术语时，补一句白话解释。
4. 找不到精确术语时，使用一眼能懂的白话标题。
5. 新造名称必须证明已有名称无法承载。

涉及事实、历史、科学、法律、医疗或金融时，必须查找可靠来源并降低未经验证断言的强度。

## 阶段 7：单卡建模

使用机制卡模板，并执行三个测试：

### 复述测试

只看标题和一句话，读者能否说出卡片大意？

### 迁移测试

把来源人物和领域词全部替换，卡片能否指导检查一个新情境？

### 反例测试

是否存在具体情境让机制失效？如果不存在，卡片可能写得过宽。

## 阶段 8：关系审查

关系建立顺序：

1. 判断是否有明确的结构作用；
2. 选择正式关系类型；
3. 写建立理由；
4. 写“不代表什么”；
5. 标记证据角色；
6. 检查反向关系；
7. 无法完成时只保留为候选关联。

不以链接数量作为成功指标。孤立但真实的卡优于连接丰富的假网络。

## 阶段 9：跨系统组合

每次沉淀都必须扫描组合机会，但不强制生成新系统。

### 9.1 为每个系统建立接口

```text
系统名称：
解决的问题：
输入：
内部过程：
输出：
盲点：
最容易被什么机制破坏：
```

### 9.2 组合机会来源

优先寻找不同机制家族之间的接口：

- 一个系统的输出成为另一个系统的输入；
- 一个系统暴露另一个系统的盲点；
- 一个系统解释失败，另一个系统提供约束；
- 两个系统分别处理效率与权利、短期与长期、局部与整体；
- 一个系统产生变化，另一个系统负责选择、证伪或治理。

### 9.3 组合门

新系统只有同时通过以下问题才能建卡：

1. 原系统分别解决什么问题？
2. 它们的接口在哪里？
3. 哪个机制解释原因，哪个机制提供约束或操作？
4. 组合后解决了什么新问题？
5. 新能力为什么不能由任何单个机制产生？
6. 移除某个组件后，系统会失去什么？
7. 什么现实结果会推翻或修改这个组合？
8. 能否给出一个具体应用设计？

只把多个概念并排展示，不算组合。

### 9.4 参考范式

黄金样本中的组合：

```text
反脆弱：提供低成本试错
古德哈特定律：揭示指标目标化后的失真
委托代理：解释执行者制造失真的动机
可证伪性：提供事前反驳条件
→ 反自欺实验系统：让组织能够诚实地从试错中学习
```

## 阶段 10：整批写入

写入不是逐张即兴修改：

1. 在内存中完成本批卡片、关系和状态决策。
2. 检查标题冲突、重复与双向关系。
3. 保存待修改文件的原始内容。
4. 写入临时文件。
5. 运行机械校验和语义审核。
6. 全部通过后替换正式文件。
7. 任一关键门失败，恢复旧状态并删除本批无效新增。

不得留下只有一半关系已经写入的知识库。

## 阶段 11：质量验收

执行 [QUALITY-GATES.md](QUALITY-GATES.md)：

- 来源忠实门；
- 卡片机制门；
- 命名可读门；
- 迁移门；
- 关系门；
- 组合新增能力门；
- Obsidian 机械门；
- 用户可用门。

## 阶段 12：用户报告

使用 [run-report.md](templates/run-report.md) 输出结果。报告不把内部候选和维护劳动转嫁给用户，只呈现可审核的关键决策。
