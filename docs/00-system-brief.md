# 机制原子知识系统 v0.4

## 一句话目标

建立一个多尺度机制知识网络：把文章、对话、视频等高冗余文本拆解为最小可复用机制单元；每个机制单元保留机制核、参与角色、成立条件和迁移边界；机制原子可以脱离原领域，在共享同一机制的其他领域中重新绑定对象并生成新的解释与判断。

完整目标契约见 `docs/11-goal-contract.md`。这份系统说明只描述最小结构和运行条件。

## 最小结构

这是一个在当前工作区根目录内运行的 skill。核心目录只有一个：

```text
atoms/
```

不创建 `sources/`、`systems/`、`relations/` 等独立目录。原始材料的来源只写进原子的 `source` 指针。系统不是另一类文件——它只是一种可以被折叠或展开的高层级原子。

## 机制原子

机制原子是这个系统的核心。它不是词语，不是句子，不是笔记卡片。

一个合格的机制原子必须能说清四件事：

1. **机制核**：到底发生了什么。因果链或结构约束。
2. **参与角色**：哪些东西参与了这个机制，各自起什么作用。
3. **成立条件**：这个机制在什么条件下运作。
4. **迁移边界**：在什么条件下不适用。不能迁移到哪些领域。不能为什么理由乱迁移。

判断一个东西是不是机制原子，问四句话：

- 它有没有机制？（不只是标签或观点）
- 它能不能解释为什么？（不只是描述现象）
- 它能不能迁移到共享同一机制的其他领域？（不只是换词造句）
- 它有没有边界，防止瞎迁移？

## 知识库的四层结构

```text
L1 原始文本
    文章、对话、视频转录。不存入 atoms/。
    ↓
L2 领域内容
    在原领域中有含义的知识节点。领域绑定。
    例如：研磨度、粉水比、萃取 → 属于手冲咖啡领域。
    ↓
L3 机制原子 ← 核心资产
    脱离领域的具体对象，只保留机制结构。
    例如：变量、影响、反馈、平衡、过度、不足、阈值。
    例如：固定成本摊薄、到达-处理失衡、介质成分转移。
    ↓
L4 跨领域绑定
    同一机制原子在不同领域中的角色映射。
    例如：固定成本摊薄 → 工厂(机器→产量)、婚姻(住房→两人)、SaaS(服务器→用户)
```

## 原子的三种形态

### 基础原子

不可再拆的概念、属性、操作或关系。`components: []`。

例如：`信息`、`反馈`、`平衡`、`阈值`、`不足`、`过度`、`可复用`、`减少`

### 机制原子

携带完整机制的最小可复用单元。正文必须包含机制核、参与角色、成立条件、迁移边界。

例如：`固定成本摊薄`、`到达-处理失衡`

### 复合原子

由多个原子组合而成的更高阶结构。`components` 不为空。可以折叠为黑箱节点，也可以展开为子结构。

例如：`知识系统` = `知识原子` + `关系网络` + `反馈` + `涌现`

## 脱域

**脱域**是指一个原子从原领域中抽离，成为跨领域可复用的构件。

脱域不是类比。不是"A影响B"换词成"C影响D"。脱域要求底层机制真的一致。

例如：
- 咖啡中的"萃取"机制（介质接触材料→成分转移）可以脱域为"介质成分转移"机制原子，迁移到泡茶、煲汤、中药煎煮、香料泡酒——因为它们共享同一物理化学机制。
- 但不能迁移到"学习"——因为学习领域不存在"介质从材料中溶解可溶物"这个机制。

**脱域节点**和**领域节点**的区分：
- 领域节点：`研磨度`、`粉水比`、`萃取`——绑定在咖啡领域
- 脱域节点：`变量`、`影响`、`反馈`、`平衡`、`阈值`——可在多领域复用

## 折叠与展开

任何知识节点都可以在一种尺度上作为原子（折叠），在另一种尺度上作为系统（展开）。

- **折叠**：把复杂系统当成一个可调用的黑箱节点。例如在分析婚姻时，`固定成本摊薄` 作为一个整体被调用。
- **展开**：打开节点，暴露内部的角色、条件、子机制。例如需要理解 `固定成本摊薄` 时，展开为固定成本 + 共享资源 + 使用人数 + 平均成本 + 容量限制 + 协调成本。

这意味着：**原子不是绝对的最小单位，而是相对于当前任务的最小可操作单元。**

## 多尺度机制网络

知识库不是树，不是平表。它是俄罗斯套娃 + 知识图谱 + 复杂网络的结合体。

每个节点可以有四种关系方向：

- **向上**：它属于哪个更大的系统？（归属）
- **向下**：它由哪些更小的机制或概念构成？（组成）
- **横向**：它在哪些其他领域中有相同或相似的机制绑定？（跨领域同构）
- **斜向**：它和另一个不同层级的节点有什么关联？（跨层连接）

## Nix 思想的迁移

本工程的构建规则受 Nix 的可靠构建模型启发，但其知识理论核心不属于 Nix。

保留的 Nix 思想：

1. **稳定身份**：原子必须有稳定 `id`，标题不能当身份。
2. **来源可追溯**：原子必须有 `source` 指针。
3. **正向依赖显式**：原子必须声明 `depends_on`。
4. **组合关系显式**：复合原子通过 `components` 声明组成。
5. **闭包由图计算**：`closure` 从 `depends_on` 递归计算，不手写。
6. **反向引用由图反查**：`used_by`、`part_of` 不手写。
7. **复用优先**：新输入先查旧原子，能复用就不新建。

Nix 没有覆盖的部分（本系统的核心创新）：
- 机制提取与脱域
- 跨领域绑定与迁移边界
- 多尺度折叠与展开
- 领域节点与脱域节点的区分

## 最小 atom frontmatter

```yaml
id:
status:
source:
depends_on:
components:
```

正文必须包含：定义、边界、机制核心、参与角色、成立条件、迁移边界、复用价值、领域绑定（或脱域说明）、区分说明、必要说明、依赖、组件。

## 充分运转条件

1. 有 atom store：所有原子有统一存放处。
2. 有稳定 ID：每个原子可以被稳定引用。
3. 有来源指针：每个原子能追溯来源。
4. 有依赖关系：理解当前原子需要哪些前置原子。
5. 有组成关系：复合原子由哪些下层原子组成。
6. 有机制提取能力：能从领域内容中分离出脱域机制。
7. 有迁移边界检查：跨领域迁移前必须验证机制一致性。
8. 有构建规则：按 `docs/06-build-decision-rules.md` 判断。
9. 有复用检索：按 `docs/08-reuse-search.md` 查旧原子。
10. 有写入安全：按 `docs/07-operation-modes-and-safety.md` 执行。
11. 有生命周期：按 `docs/09-id-lifecycle-iteration.md` 演化。
12. 有使用协议：按 `docs/10-usage-experience-and-validation.md` 执行。
13. 有目标契约：按 `docs/11-goal-contract.md` 判断 v0 是否完成。
14. 有地基补齐：按 `docs/12-foundation-grounding.md` 补齐底层学科原理。
