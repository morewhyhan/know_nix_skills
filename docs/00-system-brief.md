# Autonomous Mechanism Atom Compiler v1.1：系统简报

## 目标

把高冗余材料自动编译为可替代原文的来源还原稿，以及未来 AI 和人都能调用的机制知识函数。系统同时强调全文语义覆盖、CLC 全表检索、现成术语命名、复用、证据边界、假迁移防护和长期维护。

唯一机器契约：[`../skills/nix-knowledge-compiler/references/nix-knowledge-compiler-v1.1.md`](../skills/nix-knowledge-compiler/references/nix-knowledge-compiler-v1.1.md)。

## 用户与编译器的分工

用户提供材料和任务目标。AI 自主完成：

1. 完整读取和逐段扫描。
2. 抽取知识函数候选及互动结构。
3. 搜索独立材料，区分来源支持、外部验证和编译器推导。
4. 多路检索旧 atom，完成复用和假迁移审查。
5. 自主裁决复用、升级、新建、拆分、合并或丢弃。
6. 快照所有待修改旧文件的原始字节，在同目录临时文件生成并校验完整批次，再原子替换目标。
7. 写入 draft，执行硬校验和整批验收；准备 stable 时由未参与成稿的 subagent 独立语义复核。
8. 失败时自动返工；最多两轮仍失败时恢复旧文件原始字节并删除本轮新增。
9. 满足 stable 证据门后自动升级生命周期状态。

只有来源无法访问、任务边界发生实质变化或需要新的外部权限时，AI 才向用户报告阻塞；不得要求用户替编译器判断候选质量。

## 最小存储

核心 store 有两层：`reconstructions/` 保存每份来源的完整核心意思，`atoms/` 保存高复用机制。来源不复制成原文副本树，运行中间状态不写入知识库。

每个 atom 的唯一 frontmatter：

```yaml
id: 机制短名称
status: draft
source:
  - docs/00-system-brief.md
tags:
  - N94/系统科学
aliases:
  - 常见别名一
  - 常见别名二
depends_on: []
components: []
```

## 机制与组合

atom 必须是完整知识函数。基础机制允许 `components: []`；真正复合系统必须列出组成它且共同产生新能力的 components。系统不采用严格层级编号。

所有活动 atom 的 15 个 H2 机器键固定为中文；文件名、ID、H1、别名和章节内容跟随来源主语言。

## 证据与领域绑定

“必要说明”的八个标签必须各且仅出现一次，其中 `命名依据` 记录学科术语、通俗用语、描述性命名或编译器造词及其核验理由；`独立语义复核` 记录未参与成稿的 subagent 的单一结论与具体范围或原因。

领域绑定逐行只标注以下三种验证状态：

- `来源支持`：输入材料直接支持该领域实例，证据定位以精确 frontmatter source locator 或无歧义 basename 开头。
- `独立材料验证`：独立于输入的来源支持同一函数签名和机制，证据定位同样以精确 locator 或无歧义 basename 开头；两行必须解析到路径 identity 与内容哈希都不同的来源。
- `机制推演`：编译器根据结构映射生成，尚不算验证。

推演绑定可以保留在 draft，但不能用于 stable 证据计数。

`stable` 首先要求 `source` 至少包含两个路径 identity 与内容哈希都不同、且本地可解析的来源文件，URL 不计硬门；随后还要满足以下二选一证据路径：两个独立来源支持同一机制且至少一个领域绑定获得合格定位的独立材料验证；或可核验的权威原理来源且至少一份实际领域材料完成验证。两条路径都必须由未参与成稿的 subagent 独立语义复核通过，并通过全部硬校验。

## 完成标准

一次 compile 的完成不是“写出了文件”，而是 AI 已自动完成检索、复核、裁决、事务写入、验证和返工，并在最终结果中简要说明复用、新建、证据与剩余不确定性。
