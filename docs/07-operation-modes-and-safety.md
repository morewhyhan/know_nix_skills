# 操作模式与写入安全规则 v0.4

定义 skill 如何被使用，以及什么时候允许写入知识库。

## 工作区作用域

这个 skill 只在当前 Agent 打开的工作区根目录内工作。

- 所有新原子写入当前根目录下的 `atoms/`。
- 不默认读取或写入其他项目目录。
- 不默认写入用户全局目录。

## 操作模式

| mode | 用途 | 是否默认写入 |
|---|---|---|
| `compile` | 把文章、对话、摘录、想法编译成原子 | 是，默认写入 `draft` |
| `de-domain` | 检查已有领域节点是否可脱域为机制原子 | 否（仅分析） |
| `reuse` | 只检索已有原子，判断能否复用 | 否 |
| `validate` | 检查原子或整个 store 是否合格 | 否 |
| `query` | 用已有原子辅助当前思考或回答问题 | 否 |
| `refactor` | 合并、拆分、升级、废弃旧原子 | 是，但必须留下理由 |

### de-domain 模式

专门用于判断一个领域节点能否脱域：

1. 列出该节点的所有领域专属概念。
2. 替换为抽象角色。
3. 检查机制是否仍然完整。
4. 搜索是否存在共享此机制的其他领域。
5. 如果能通过以上检查，建议升级为机制原子。
6. 如果不能，说明为什么它必须留在原领域。

## 默认模式

- 用户给文章、对话、摘录、想法，并要求沉淀：默认 `compile`。
- 用户问"这个领域知识能不能迁移到别处"：默认 `de-domain`。
- 用户问"已有知识怎么看""用知识库帮我想"：默认 `query`。
- 用户说"检查一下""有没有问题"：默认 `validate`。
- 用户要求整理旧原子、合并、拆分、升级：默认 `refactor`。
- 意图不明确时，先按 `reuse` 或 `query` 处理，不直接写入。

## 写入安全

新生成原子默认只能是：

```yaml
status: draft
```

`stable` 必须满足：
- 通过单原子质量门。
- 通过整批编译验收。
- 没有重复原子。
- `depends_on` 和 `components` 引用都存在。
- 没有依赖循环。
- 来源可追溯。
- 定义、边界、机制四要素、复用价值、区分说明写清楚。
- 领域绑定或脱域说明写清楚。
- 正文依赖和组件使用 wikilink，并与 frontmatter 一致。

## 修改旧原子

修改旧原子时必须说明修改理由。

允许直接修改且不升版本：修正措辞、补充边界、补充例子、补充来源、补充领域绑定、补充迁移边界说明。

必须谨慎处理：改变定义、改变机制核、改变成立条件、改变迁移边界、改变核心依赖、合并或废弃。这些情况必须进入 `refactor` 模式。

## 硬校验

一次写入完成后，必须检查：

- `id` 是否重复。
- 文件名是否等于 `id`。
- `status` 是否只属于 `draft`、`stable`、`deprecated`、`merged`。
- 是否缺少必备字段。
- `depends_on` 和 `components` 是否引用存在的原子。
- 是否出现循环依赖。
- 是否出现禁止字段。
- 是否出现 `status: rejected`。
- 机制原子是否有机制四要素在正文中。
- 声称脱域的原子是否有迁移边界。

## 事务规则

- 开始前先明确输入来源和操作模式。
- 执行中先生成 `draft`。
- 结束时在本次回复中说明处理结果。
- 质量不通过时，保留 `draft`，不强行 stable。
- 失败候选不进入 atom store。

## 完成标准

一次操作完成时，必须能回答：

- 这次运行是什么模式？
- 写入了什么？
- 为什么允许写入？
- 哪些内容没有写入，为什么？
- 是否通过硬校验？
- 是否在本次回复中说明了主要取舍？
