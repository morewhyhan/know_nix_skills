# 操作模式与写入安全规则 v0

这份文档定义 skill 如何被使用，以及什么时候允许写入知识库。

目标是防止 AI 把一次普通对话、一次不确定判断、或一次失败编译直接污染 atom store。

## 工作区作用域

这个 skill 只在当前 Agent 打开的工作区根目录内工作。

规则：

- 所有新 atom 写入当前根目录下的 `atoms/`。
- 所有模板、schema 和规则文件都按当前根目录解析。
- 不默认读取或写入其他项目目录。
- 不默认写入用户全局目录。
- 不把 `skills/nix-knowledge-compiler/` 当成唯一知识库位置；它只是 skill 规则本身。

如果当前根目录还没有 `atoms/`，可以在当前根目录创建。不要去别的目录寻找“全局 atom store”。

## 操作模式

skill 保留 5 个模式。

| mode | 用途 | 是否默认写入 |
|---|---|---|
| `compile` | 把文章、对话、摘录、想法编译成 atom | 是，默认写入 `draft` |
| `reuse` | 只检索已有 atom，判断能否复用 | 否 |
| `validate` | 检查 atom 或整个 store 是否合格 | 否 |
| `query` | 用已有 atom 辅助当前思考或回答问题 | 否 |
| `refactor` | 合并、拆分、升级、废弃旧 atom | 是，但必须留下理由 |

## 默认模式

- 用户给文章、对话、摘录、想法，并要求沉淀：默认 `compile`。
- 用户问“已有知识怎么看”“用知识库帮我想”：默认 `query`。
- 用户说“检查一下”“有没有问题”：默认 `validate`。
- 用户要求整理旧 atom、合并、拆分、升级：默认 `refactor`。
- 意图不明确时，先按 `reuse` 或 `query` 处理，不直接写入。

## 写入安全

新生成 atom 默认只能是：

```yaml
status: draft
```

不能直接标记为 `stable`。

`stable` 必须满足：

- 通过单 atom 质量门。
- 通过整批编译验收。
- 没有重复 atom。
- `depends_on` 和 `components` 引用都存在。
- 没有依赖循环。
- 来源可追溯。
- 定义、边界、底层机制、复用价值、区分说明写清楚。
- 调用方式写清楚。
- 上层系统候选或可组合方向写清楚。
- 正文依赖和组件使用 wikilink，并与 frontmatter 一致。
- 用户或本次验收明确通过。

## 修改旧 atom

修改旧 atom 时必须说明修改理由。

允许直接修改且不升版本的情况：

- 修正措辞。
- 补充边界。
- 补充例子。
- 补充来源定位。
- 补充底层机制、复用价值、区分说明或必要说明。

必须谨慎处理的情况：

- 改变定义。
- 改变底层机制。
- 改变适用边界。
- 改变核心依赖。
- 合并或废弃旧 atom。

这些情况必须进入 `refactor` 模式，并在本次回复中说明修改理由。若 atom 被标记为 `merged` 或 `deprecated`，理由写入正文。

## 硬校验

一次写入完成后，必须检查：

- `id` 是否重复。
- 文件名是否等于 `id`。
- `status` 是否只属于 `draft`、`stable`、`deprecated`、`merged`。
- 是否缺少 `source`、`depends_on`、`components`。
- `depends_on` 是否引用存在的 atom。
- `components` 是否引用存在的 atom。
- `depends_on` 是否出现循环。
- `components` 是否出现循环。
- 是否出现禁止字段。
- 是否出现 `status: rejected`。

硬校验失败时，新内容不能标记为 `stable`。

## 事务规则

一次编译或重构应当被视为一次事务。

- 开始前先明确输入来源和操作模式。
- 执行中先生成 `draft`。
- 结束时在本次回复中说明处理结果。
- 质量不通过时，保留 `draft`，不强行 stable。
- 失败候选不进入 atom store。

构建失败不是知识资产。失败原因可以在本次回复中说明，但失败候选不要变成 atom。

## 完成标准

一次操作完成时，必须能回答：

- 这次运行是什么模式？
- 写入了什么？
- 写入是否都发生在当前工作区根目录内？
- 为什么允许写入？
- 哪些内容没有写入？
- 是否通过硬校验？
- 是否在本次回复中说明了主要取舍？
