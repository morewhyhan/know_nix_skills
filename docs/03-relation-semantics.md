# 关系语义

v0.5 保留能支撑机制复用、分层构建和多尺度网络的关系体系。

## Frontmatter 关系（机器可读）

### source

当前原子的来源指针。可以指向文章、对话、书摘、外部 URL、已有原子。

### depends_on

理解当前原子必须的前置原子。标准：

> 没有对方，当前原子不能被完整理解或使用。

不要把"同主题""相关""类似"写成依赖。

### components

组成当前原子的下层原子。标准：

> 删除某个 component，会让当前原子的结构或能力不完整。

基础原子 `components: []`。复合原子 `components` 不为空。

## 语义关系（正文 wikilink 中标注）

`depends_on` 和 `components` 只说了"有关系"，没说"什么关系"。语义关系标注在正文 wikilink 后面的破折号中，**不新增 frontmatter 字段**，格式为：

```markdown
- [[atom-id]] - 关系类型：理由
```

### 结构语义关系（出现在 components 中）

| 关系类型 | 含义 | 示例 |
|---------|------|------|
| `composed_of` | A 由 B 作为组件构成 | `[[语义单位]] - composed_of：知识原子的构件` |
| `implements` | A 实现了 B 定义的功能 | `[[固定成本摊薄]] - implements：婚姻生活成本分摊的底层机制` |
| `instantiates` | A 是 B 的领域实例 | `[[到达-处理失衡]] - instantiates：咖啡萃取不足/过度是这个机制在咖啡领域的实例` |

### 认知语义关系（出现在 depends_on 中）

| 关系类型 | 含义 | 示例 |
|---------|------|------|
| `prerequisite` | 必须先理解 B 才能理解 A | `[[机会成本]] - prerequisite：不先理解机会成本就无法理解固定成本摊薄的权衡逻辑` |
| `defined_by` | A 的含义由 B 定义 | `[[知识原子]] - defined_by：知识原子的定义依赖这个概念` |
| `grounded_in` | A 的机制根基在 B | `[[反馈回路]] - grounded_in：反馈-调整-再输出的底层是反馈回路` |

### 逻辑语义关系（出现在 depends_on 或正文中）

| 关系类型 | 含义 | 示例 |
|---------|------|------|
| `supports` | B 为 A 提供论证支持 | `[[原文有冗余]] - supports：冗余存在→需要提取可复用语义单位` |
| `constrains` | B 限制了 A 的适用范围 | `[[原子不是越短越好]] - constrains：知识原子不能只按长度判断` |
| `explains` | B 解释了 A 为什么成立 | `[[涌现来自互动]] - explains：解释了为什么知识原子需要关系` |
| `contrasts_with` | A 和 B 形成对比/对立 | `[[词语不是原子]] - contrasts_with：词语≠知识原子，反例澄清边界` |

### 用法

语义关系不新增字段——只写在 wikilink 后面的标注中。AI 编译时按以下优先级选择关系类型：

1. 先判断这是 `components` 还是 `depends_on`
2. 再判断语义类型（supports/constrains/explains/grounded_in……）
3. 写在正文对应章节的 wikilink 中

```markdown
## 组件
- [[语义单位]] - composed_of：知识原子是由语义单位组合而成的
- [[可复用]] - composed_of：知识原子必须具备可复用性
- [[可操作]] - composed_of：知识原子必须可被识别、处理、调用

## 依赖
- [[机会成本]] - prerequisite：不理解机会成本就无法理解固定成本摊薄的权衡
- [[反馈回路]] - grounded_in：固定成本摊薄的迭代优化依赖反馈回路逻辑
```

## 分层构建顺序

知识原子的构建不是平的——要自底向上，每层只能引用同层或下层的原子。

```text
L0 基础概念    （信息、不确定性、反馈、平衡、阈值、变量、影响……）
  │             components: []  —— 不可再拆
  │             其他原子通过 depends_on 引用它们
  ↓
L1 基础关系    （信息→减少→不确定性、系统→包含→元素）
  │             components 指向 L0 的概念
  │             把 L0 的概念连接成最简单的三元组
  ↓
L2 组合概念    （语义单位、可复用语义单位、知识原子、关系网络）
  │             components 指向 L0 或 L1
  │             由基础概念和基础关系组合出更高阶的概念
  ↓
L3 机制原子    （固定成本摊薄、到达-处理失衡、介质成分转移）
  │             components 指向 L0/L1/L2
  │             携带完整的机制四要素
  ↓
L4 命题        （知识系统由知识原子和关系网络构成）
  │             depends_on 指向 L2/L3
  │             表达一个完整判断，由下层概念和机制组合而成
  ↓
L5 复合系统    （知识原子化理论、手冲咖啡调参系统）
               components 指向 L3/L4
               多个机制和命题组合成的完整理论或系统
```

**编译规则：**
- 创建上层原子前，检查它的所有 components 和 depends_on 目标是否已存在
- 缺少下层 → 先补齐下层，再创建上层
- 不能跨层依赖（L4 的命题不能直接把 L0 的基础概念当 component——中间必须有 L2/L3 的组合）
- L0 基础概念和 L1 基础关系只在被上层引用时才建——不预先建库

## 多尺度网络关系

除了分层构建的垂直关系，原子之间还有四种方向的关系。这些不是新字段，而是从已有关系中推出来的视角。

### 向上（归属）

当前原子属于哪个更大的系统？

→ 由其他原子的 `components` 包含当前原子来体现。不需手写。Obsidian 反向链接自动展示。

### 向下（展开）

当前原子由哪些更小的机制或概念构成？

→ 已由 `components` 承载。沿着 components 递归展开就是完整的向下路径。

### 横向（跨领域同构）

同一个机制原子在哪些不同领域中被绑定？

→ 已由正文"领域绑定"表格承载。当两个领域节点引用同一个机制原子时，它们之间的横向关系自动形成。

```text
咖啡萃取 ──instantiates──→ 介质成分转移 ←──instantiates── 中药煎煮
                                        ←──instantiates── 煲汤
```

### 斜向（跨层连接）

两个不同层级、不同领域的原子有直接的机制关联。

→ 不常见但真实存在。在正文"区分说明"或"必要说明"中手动标注 `[[atom-id]] - connects_to：理由`。

## 编译时建关系：操作步骤

每次 compile 完成后，必须在以下时机建立关系：

### 1. 创建每个原子时

- 写入 `depends_on`（前置理解依赖）和 `components`（结构组成）
- 在正文对应章节用 wikilink + 语义关系类型标注

### 2. 本批原子内部

- 检查本批新建原子之间是否存在 depends_on / components / supports / constrains / contrasts_with 关系
- 例如同批创建的"萃取不足→酸"和"萃取过度→苦"之间：`contrasts_with：两者是对称的偏离方向`

### 3. 与已有原子

- 按 `docs/08-reuse-search.md` 检索后，即使不能复用，也可能存在 supports / explains / contrasts_with 关系
- 在正文"区分说明"中标注

### 4. 跨领域绑定

- 如果新建的是脱域机制原子，检查是否存在已有领域节点可以建立 `instantiates` 关系
- 在领域绑定表中增加新行

## 涌现验证

编译结束后，检查：

> 这批原子中的某几个组合起来，能否生成原文中没有直接表述的新判断？

如果能，在编译结果说明中指出涌现路径：

```text
[atom_X] + [atom_Y] + [atom_Z] → 可涌现判断："……"
```

这是知识库产生复利的可观察信号——不是存档原文，而是通过原子组合生成了原文没有的新理解。

如果没有任何涌现，不是编译失败——有些材料本身就是领域经验，能提取的机制少，涌现空间小。但连续多次编译都没有涌现，说明可能只是在做摘要。

## 不保留的关系

- `used_by` / `part_of`：从 Obsidian 反向链接或全库反查得到
- `reuse_count`：统计计算
- `closure`：从 `depends_on` 递归计算
- `related` / `see_also`：太松，会污染图谱

## 关系质量测试

每条关系都问：

> 如果删除这条关系，系统会失去什么具体能力？

答不上来，就不要写。
