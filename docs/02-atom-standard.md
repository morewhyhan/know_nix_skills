# 知识原子标准

## 核心定义

知识原子是一个有稳定身份、来源可追溯、依赖显式、边界清楚、能被多个任务或上层 atom 复用的知识函数。

这里的“原子”不是越短越好，而是：

- 再拆就失去独立复用价值。
- 不拆就混入多个独立函数。
- 能写清它接收什么输入、执行什么操作、产出什么输出、在哪些边界内成立、未来能被什么调用。

## 知识函数

每个 atom 必须能写成：

```text
输入 -> 操作 -> 输出
```

- 输入：它处理什么材料、信号、问题、概念、目标、案例或系统。
- 操作：它对输入执行什么认知处理，如区分、抽取、映射、压缩、归因、预测、边界测试、组合。
- 输出：它产出什么结果，如判断、变量、边界、风险、解释、候选策略、组件关系。

不能写成这个函数签名的内容，不进入 atom store。

## 基础 atom 与系统 atom

基础 atom 是不可再拆的最小知识函数，`components: []`。

不可再拆的意思不是哲学上绝对不能拆，而是：

> 再拆下去，就得不到一个仍然完整的“输入 -> 操作 -> 输出”函数。

系统 atom 是多个知识函数组合出的更大函数，`components` 必须指向组成它的下层 atom。

任何系统 atom 必须能沿 `components` 和 `depends_on` 一层层追溯到底层基础 atom。不能追溯到底层基础 atom 的系统 atom，是黑箱，不合格。

## 底层学科原理

底层基础 atom 可以来自逻辑与概念分析、数学与图、概率统计、信息论、微观经济学、博弈论、决策理论、心理学、认知科学、进化心理学、语言学、传播学、社会学、系统论、控制论、复杂性科学、工程设计等底层学科。

学科不写成 `type` 或其他 frontmatter 字段。它只作为正文里的来源说明和依赖链接出现。

如果一个上层 atom 依赖某个底层学科原理，不能只写学科名，必须通过 `depends_on` 和正文“依赖”逐层链接到具体原理 atom：

```markdown
- [[底层原理atom-id]] - 当前 atom 为什么依赖这个原理
```

这样 Obsidian 能形成反向链接，也符合 Nix 只保存正向 references、反向 referrers 由图反查的思想。

如果底层原理还没有 atom，必须先创建或复用底层原理 atom；否则不要孤立创建上层 atom。

地基地图和补齐规则见 `docs/12-foundation-grounding.md`。这张地图不是 atom 分类，不进入 frontmatter；它只用于寻找当前候选成立所需的前置原理。

## 新建 atom 的唯一理由

新建 atom 的理由不是“这段内容新”，而是：

> 它具备独立复用价值，并且不能被已有 atom 准确承载。

输入材料进入后，先查旧 atom。能复用旧 atom，就不新建。

## 不划分类型

v0 不给 atom 写 `type`、`object_kind`、`level`、`content_shape`。

原因很简单：这些分类不能直接产生复利，反而会诱导 AI 先分类、后思考。

当前先判断一个问题：

```text
这个候选是否是一个可复用知识函数？
```

系统也是 atom。一个系统级 atom 仍然放在 `atoms/`，只是 `components` 不为空。

## 合格条件

一个 atom 必须满足：

1. **稳定身份**：有稳定 `id`。
2. **来源可追溯**：有 `source` 指针。
3. **语言一致**：文件名、ID、H1、字段标题和正文跟随输入材料主语言。
4. **可定义**：能清楚回答“它是什么”。
5. **函数签名清楚**：能写清“输入 -> 操作 -> 输出”。
6. **边界清楚**：说得清它不是什么、不能用于哪里。
7. **底层机制清楚**：能写清因果链、结构约束或生成机制。
8. **底层依赖可追溯**：能通过 `depends_on` 或 `components` 链到底层基础 atom 或底层学科原理 atom。
9. **复用价值明确**：能说明未来可以支持哪些任务或更大 atom，并写清调用方式。
10. **区分说明具体**：说明为什么不是已有 atom 或本批其他候选。
11. **依赖显式**：需要哪些前置 atom 写在 `depends_on`。
12. **组件显式**：如果由其他 atom 组成，写在 `components`；如果没有，`components: []`。
13. **系统可归属**：能说明它可以接入哪个更大 atom 或系统 atom；暂时没有也要写清候选方向。
14. **不能被旧 atom 准确承载**：否则复用旧 atom。

## 正文字段

中文 atom 正文固定使用：

```markdown
# 原子名称

## 定义
它是什么。

## 边界
它不是什么，不能用于哪里，和相邻 atom 的差异是什么。

## 底层机制
它为什么成立。

函数签名：

```text
输入 -> 操作 -> 输出
```

## 复用价值
它为什么值得单独保存，未来能支持哪些任务或更大 atom。

调用方式：当我需要判断/解释/生成什么时，应该引用这个 atom？

上层系统：它可以被哪些更大 atom 或系统 atom 调用？

## 区分说明
为什么不是已有 atom 或本批其他候选。

## 必要说明
别名、例子、限制、来源定位、底层学科来源、迭代入口。

## 依赖
列出 depends_on 对应 atom，空则写“无”。有依赖时使用 `[[atom-id]] - 依赖理由`。底层学科原理也在这里链接。

## 组件
列出 components 对应 atom，空则写“无”。有组件时使用 `[[atom-id]] - 组件职责`。系统 atom 必须能沿组件链追溯到底层基础 atom。
```

frontmatter 是机器可读结构；正文里的“依赖”和“组件”是人可读解释。两者必须一致。

## 可调用性

一个 atom 不能只解释“它是什么”，还必须让未来使用者知道“它什么时候有用”。

合格 atom 的“复用价值”必须回答：

- 我遇到什么问题时应该调用它？
- 它能帮助我判断、解释、生成或避免什么？
- 它能被哪个更大 atom 或系统 atom 组合使用？
- 它不适合用于什么场景？

如果读完仍然不知道怎么使用这个 atom，它就不合格。

## 互动结构与规则点

当材料涉及多人互动、关系、交易、竞争、合作、权力、承诺、信号或奖惩时，先抽互动结构，再决定哪些内容能成为 atom。

互动结构包括：

- 玩家：谁在局里。
- 目标：每个玩家想得到什么、避免什么。
- 信息：谁知道什么，谁不知道什么，谁在隐藏什么。
- 策略空间：每个玩家能做什么、不能做什么、能不能退出。
- 规则点：稳定影响选择、信息暴露、成本收益或关系走向的约束和生成机制。
- 奖惩机制：做某动作会获得什么、失去什么、把成本转嫁给谁。
- 信号机制：哪些行为、语言、身体反应、历史记录会泄露隐藏偏好或策略。
- 可能结果：合作、竞争、僵局、退出、背叛、责任转移或稳定均衡。

只有规则点能通过原子精度门时，才沉淀为 atom。玩家名单、单个案例、一次性策略、文章里的技巧步骤，不直接成 atom。

规则点可以来自心理学、生理学、社会学、经济学、博弈论、语言学或传播学。学科只是来源，不是 atom 分类。

## 原子精度门

新建 atom 前，必须能写出：

```text
触发条件 -> 中介机制 -> 可观察结果 -> 适用边界
```

如果写不出，它通常还不是 atom，而只是：

- 原文观点。
- 方法步骤。
- 案例。
- 话术。
- 标签。
- 金句。
- 普通分类名。

这些内容最多进入某个 atom 的“必要说明”，不要单独建 atom。

## 太大的判断

太大不是字数多，而是：

> 一个 atom 混入了多个可独立复用的机制、边界或依赖结构。

信号：

- 不同任务只需要其中一部分。
- 它可以自然拆成多个独立定义。
- 它的 `depends_on` 指向几组互不相干的前置 atom。
- 它同时承担概念解释、方法建议、案例证明和系统总结。

处理方式：拆成多个 atom。

## 太碎的判断

太碎不是字数少，而是：

> 它单独存在没有复用价值，必须依附另一个 atom 才能用。

信号：

- 只是例子、措辞、局部表达、话术或单条建议。
- 不能支持任何未来任务。
- 没有独立定义。
- 写不出底层机制。
- 不能被上层 atom 引用。

处理方式：并入更大的 atom 的“必要说明”，或只保留为来源说明，不单独建 atom。

## 材料角色不等于 atom

以下内容默认不独立成 atom：

- 方法步骤。
- 案例故事。
- 证据数据。
- 作者观点。
- 金句修辞。
- 普通误区。
- 文章标题和小标题。

它们只有在能提炼出可复用知识函数，并通过原子精度门时，才可能成 atom。

## 合并规则

两个 atom 只有在以下情况才合并：

- 底层机制一致。
- 边界兼容。
- 复用功能一致。
- 依赖结构兼容。
- 分开不能带来额外复用价值。

如果底层相似，但服务的系统、边界或调用方式不同，不要粗暴合并。更好的做法是升级出一个更通用的上层 atom，再让不同 atom 依赖它。

## 一句话验收

一个 atom 只有在“语言跟随输入、定义清楚、函数签名清楚、边界清楚、底层机制清楚、能追溯到底层基础 atom 或底层学科原理 atom、复用价值和调用方式明确、区分说明具体、依赖和组件准确、能接入更大系统、且不能被旧 atom 准确承载”时才新建；否则复用、合并、并入字段或丢弃。
