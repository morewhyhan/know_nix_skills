# 旧 Atom 检索与复用规则 v0

这份文档让“复用优先”变成可执行流程。

知识复利来自复用。如果新建前没有认真查旧 atom，知识库会变成重复文件堆。

## 新建前必须检索

每个候选知识点新建前，必须先查旧 atom。

检索顺序：

1. 查候选名称。
2. 查同义词和别名。
3. 查底层机制词、触发条件、结果词。
4. 查相邻概念。
5. 查可能的上层 atom。
6. 如果是互动结构材料，查玩家关系、奖惩、信号、策略、责任转移、信息不对称等同类规则点。

至少阅读最接近的 3 个旧 atom。若库中候选很少，则阅读所有可能相关 atom。

## 检索充分标准

只有同时满足以下条件，才算检索充分：

- 搜过候选名称。
- 搜过至少 2 个同义表达。
- 搜过底层机制词、触发条件、结果词。
- 搜过可能的上层系统 atom。
- 读过最接近的旧 atom。
- 判断过旧 atom 是否能准确承载。
- 判断过当前候选能否作为某个系统 atom 的组件。

检索不充分时，不允许新建 `stable` atom。

## 复用判定

旧 atom 能准确承载时，必须复用，不新建。

准确承载的判断仍以 `docs/06-build-decision-rules.md` 为准：

- 底层机制一致。
- 边界兼容。
- 复用功能一致。
- 依赖结构兼容。

## 新建理由

如果最终新建 atom，本次回复必须说明：

- 检查过哪些旧 atom。
- 检查过哪些本批候选。
- 这些旧 atom 或本批候选为什么不能准确承载。
- 当前 atom 的独立复用价值是什么。
- 当前 atom 的调用方式是什么。
- 当前 atom 能接入哪个上层系统或更大 atom；暂时没有时，上层系统候选是什么。
- 当前 atom 是否会被未来任务或上层 atom 引用。

不能说清这些，就不要新建 atom。

## 复用说明

如果复用旧 atom，不新建新文件。

本次回复里说明：

- 复用了哪个 atom。
- 为什么它能准确承载。
- 当前输入材料是否提供了新例子或新语境。
- 是否需要升级旧 atom。

复用不是省略工作。复用必须有判断依据，但判断依据不单独落盘成日志。

## 别名规则

不新增 frontmatter 字段。

别名写进正文“必要说明”，必要时也可以在“定义”或“边界”里说明：

```markdown
常见别名：复用优先、先复用再新建、reuse-before-create
```

检索旧 atom 时必须检查别名。

## 来源保存边界

v0 只写 `source` 指针，不创建 `sources/` 目录，不保存完整原文快照。

如果来源需要更精确，补清 `source.locator`，或在正文“必要说明”里写短定位信息。不要把原材料变成另一类知识资产。

## 案例与例子

例子默认不单独建 atom。

例子只进入 atom 正文“必要说明”，除非它满足：

- 案例本身会被未来任务反复引用。
- 案例包含可复用知识函数。
- 案例的结构比单个机制更重要。

如果案例包含可复用知识函数，优先拆出知识函数 atom。案例本身只有满足新建 atom 的全部标准，才单独保留；否则不保留。

## 最终判定句

新建前必须能说：

> 我已经检查了最相关的旧 atom 和本批候选；它们不能准确承载这个候选；因此新建这个 atom 会增加可复用知识，而不是制造重复。
