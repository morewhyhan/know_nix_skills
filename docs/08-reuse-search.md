# 多路检索与复用 v1.1

唯一机器契约见 [`../skills/nix-knowledge-compiler/references/nix-knowledge-compiler-v1.1.md`](../skills/nix-knowledge-compiler/references/nix-knowledge-compiler-v1.1.md)。

## 原则

复用检索不是单一关键词搜索，也不以 `depends_on` 是否重叠作为新建开关。候选零依赖、零名称重叠或来自不同领域时，AI 仍必须完成全部适用检索通道。

候选排序和盲审属于编译器内部状态，不要求用户维护或审批。

## 检索通道

先扫描 CLC 22 个基本大类并展开相关正式类号，再至少检查：

1. **CLC 与术语**：主类、邻接类、正式学科术语及其边界。
2. **名称**：候选名称、旧 ID、标题。
3. **别名**：frontmatter aliases、生活口语、来源原话、同义和反向词。
4. **函数签名**：输入、操作、输出是否一致。
4. **机制核心**：因果方向、结构约束和中介机制。
5. **参与角色**：角色数量、职责、控制权和信息位置。
6. **成立条件**：必要条件、触发阈值和排除条件。
7. **迁移边界**：新候选是否落在旧 atom 的不适用区。
8. **图关系**：depends、components、被引用者和相邻机制。
9. **跨领域结构**：不同术语是否实际共享同一角色—条件—机制—结果映射。

## 检索充分标准

AI 只有在以下条件满足后才能裁决新建：

- 名称和别名没有准确命中。
- 最接近的函数签名已逐项比较。
- 机制、角色、条件和边界均无法由旧 atom 承载。
- 跨领域结构搜索没有发现同构机制。
- 本批候选之间已经去重。
- 新候选确有独立调用价值。

## 复用审查

发现近似旧 atom 后，AI 应采用反方审查：先尝试证明旧 atom 能承载，再尝试寻找一个会导致错误调用的反例。只有不存在关键不兼容时才复用。

不得只读“机制核心”就结束判断；角色、条件、边界、证据和依赖结构同样属于准确承载标准。

## 零重叠处理

`depends_on: []` 可能表示基础机制，也可能表示依赖尚未识别。它从不意味着“库中没有相似 atom”。零重叠时转向名称、签名、机制、角色、条件和跨域检索，而不是直接新建。

## 结果落点

- 旧机制准确承载：复用，并按需补 source、别名或验证绑定。
- 机制相同但证据或边界过窄：非破坏性升级。
- 机制不相同但容易混淆：在区分说明中使用 `contrasts_with` 或边界反例。
- 所有通道均不能承载：才进入新建质量门。
