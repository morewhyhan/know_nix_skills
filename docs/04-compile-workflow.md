# 编译流程

这个流程把输入材料编译成 atom。重点不是摘要，而是把材料压缩成可复用、可追溯、可组合的知识函数。

## 总流程

1. 明确操作模式，通常是 `compile`。
2. 读取输入材料。
3. 把候选先写成“输入 -> 操作 -> 输出”的知识函数。
4. 如果候选是上层函数，先拆 `components`，直到能追溯到底层基础 atom。
5. 如果候选依赖底层学科原理，先检索或创建对应底层原理 atom；缺地基时先补地基，再写上层 atom。
6. 如果材料涉及互动、关系、交易、竞争、合作、权力、信号或奖惩，再抽互动结构。
7. 从互动结构或普通材料中提取候选函数，而不是按文章标题拆。
8. 过滤方法、案例、证据、观点、误区、修辞等材料角色。
9. 按检索规则查已有 atom。
10. 能复用旧 atom 就复用。
11. 旧 atom 机制匹配但边界太窄，优先升级旧 atom。
12. 不能复用时，判断是否值得新建。
13. 太大则拆，太碎则并入字段或丢弃。
14. 写入最小 atom 文件，默认 `draft`。
15. 运行单 atom 质量检查、硬校验和整批编译验收。
16. 在本次回复中说明复用、新建、升级、拆分、合并和丢弃结果。

## Step 1：读取输入材料

输入可以是文章、对话、引用、摘录、想法。

不为每段输入创建 source 文件。只要在生成的 atom 中写清来源指针即可。

## Step 2：知识函数签名

每个候选先写成：

```text
输入 -> 操作 -> 输出
```

不能写出完整函数签名的，不进入 atom store。

判断：

- 输入是否明确？
- 操作是否明确？
- 输出是否明确？
- 未来是否能被其他 atom 或任务调用？

只有完整函数才有资格继续判断。

## Step 3：组件拆解与底层追溯

如果候选是上层函数，不要直接保存成黑箱。先拆出组成它的下层函数。

规则：

- 能继续拆出完整“输入 -> 操作 -> 输出”的，就继续拆。
- 拆到再拆会失去完整函数结构时，才是基础 atom。
- 上层 atom 的 `components` 必须指向下层 atom。
- 任意系统 atom 必须能沿 `components` 和 `depends_on` 追溯到底层基础 atom。

如果候选依赖底层学科原理，比如概率统计、信息论、微观经济学、博弈论、决策理论、心理学、认知科学、进化心理学、语言学、社会学、系统论、复杂性科学，也要通过 `depends_on` 链到底层原理 atom。

如果底层原理 atom 不存在，优先创建或复用它；如果不能创建或复用，就不要孤立创建上层 atom。

地基 atom 的写入顺序必须早于依赖它的上层 atom。它的 `source` 仍然指向当前输入，并在正文“必要说明”里写明“由当前材料触发的底层原理补齐”。这不是给真伪打分，只是保证后续可以观察、链接、更新和替换。

## Step 4：互动结构扫描

不是所有材料都需要这一步。只要材料中存在多人互动、关系判断、交易、竞争、合作、权力、信号、承诺、奖惩或责任转移，就必须先做。

扫描这些位置：

- 玩家：谁在局里。
- 目标：每个玩家想得到什么、避免什么。
- 信息：谁知道什么，谁不知道什么，谁在隐藏什么。
- 策略空间：每个玩家能做什么、不能做什么、能不能退出。
- 规则点：稳定影响选择、信息暴露、成本收益或关系走向的约束和生成机制。
- 奖惩机制：做某动作会获得什么、失去什么、把成本转嫁给谁。
- 信号机制：哪些行为、语言、身体反应、历史记录会泄露隐藏偏好或策略。
- 可能结果：合作、竞争、僵局、退出、背叛、责任转移或稳定均衡。

这一步只产出候选，不直接落盘。真正能进入 atom store 的，是其中可复用的知识函数。

## Step 5：提取候选函数

先问：

- 这段材料真正提供了什么可复用函数？
- 它的输入、操作、输出是什么？
- 它为什么成立？
- 它在哪些边界内成立？
- 它是否只是方法、案例、观点或修辞？
- 如果它来自互动结构，它改变了哪个玩家的选择、信息、成本、收益、信号或责任？

遇到“如何 X”“提升 X”“解决 X”时，不要把步骤直接建成 atom。先提炼其中可复用的底层机制。

## Step 6：原子精度门

候选必须能写清：

```text
输入 -> 操作 -> 输出
```

也必须能写清：

```text
触发条件 -> 中介机制 -> 可观察结果 -> 适用边界
```

写不清就不要新建。可以并入相关 atom 的“必要说明”，或在本次回复中说明丢弃。

## Step 7：复用优先

创建新 atom 前，先查旧 atom。

具体判定以 `docs/06-build-decision-rules.md` 为准。检索流程以 `docs/08-reuse-search.md` 为准。

判断旧 atom 能否承载：

- 底层机制是否一致？
- 边界是否兼容？
- 复用功能是否一致？
- 依赖结构是否兼容？

能准确承载就引用旧 atom，不新建。

## Step 8：新建条件

只有同时满足以下条件，才新建 atom：

- 有独立复用价值。
- 旧 atom 无法准确承载。
- 定义清楚。
- 函数签名清楚。
- 边界清楚。
- 底层机制清楚。
- 能追溯到底层基础 atom 或底层学科原理 atom。
- 缺少底层学科原理时，已先补齐地基 atom。
- 复用价值明确。
- 调用方式明确。
- 区分说明具体。
- 能接入某个更大 atom 或系统 atom；暂时没有时，写清上层系统候选。
- 它能被未来任务或上层 atom 引用。

## Step 9：写入 atom

每个 atom 最小 frontmatter：

```yaml
id:
status:
source:
depends_on:
components:
```

中文正文固定写：

- 定义。
- 边界。
- 底层机制。
- 复用价值。
- 区分说明。
- 必要说明。
- 依赖。
- 组件。

frontmatter 是机器可读结构；正文里的“依赖”和“组件”是人可读解释。两者必须一致。

正文“依赖”和“组件”必须使用 Obsidian wikilink：

```markdown
- [[关系信号识别-v1]] - 没有它就无法判断当前机制中的信号来源
```

这样既保留最小 frontmatter，也让 Obsidian 能产生反向链接。

## Step 10：质量检查

按 `docs/05-quality-gates.md` 检查。

再按 `docs/07-operation-modes-and-safety.md` 做硬校验，按 `docs/10-usage-experience-and-validation.md` 做整批编译验收。

不通过的内容不能标记为 `stable`。如果没有独立复用价值，不要勉强保存为 atom。

## Step 11：结果说明

编译结束时，在本次回复中说明：

- 复用了哪些旧 atom。
- 新建了哪些 draft atom。
- 升级、拆分、合并了哪些 atom。
- 丢弃了哪些候选，以及为什么丢弃。
- 哪些 atom 仍需保持 `draft`。

这只是交互说明，不单独落盘成日志文件。

## 编译失败标准

出现以下情况，必须返工：

- 只是摘要。
- 只是摘录。
- 按文章标题或小标题建 atom。
- 把方法、案例、观点或修辞误建成 atom。
- 没有稳定 ID。
- 没有来源指针。
- 没有必要依赖判断。
- 没有定义。
- 没有函数签名。
- 没有底层机制。
- 系统 atom 不能追溯到底层基础 atom。
- 依赖了底层学科原理，却没有通过 `depends_on` 和正文 wikilink 链接对应原理 atom。
- 依赖了底层学科原理，但库中没有对应地基 atom，也没有先补齐。
- 边界不清。
- 复用价值不清。
- 调用方式不清。
- 看不出能接入哪个更大系统。
- 依赖和组件没有正文 wikilink，导致 Obsidian 无法形成反向链接。
- 区分说明空泛。
- 太大，混入多个独立机制。
- 太碎，不能被复用。
- 硬校验失败。
- 整批验收失败。
