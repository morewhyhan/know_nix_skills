# 编译流程

把输入材料编译成机制原子。重点不是摘要，而是从具体领域中提取可脱域的机制。

## 总流程

1. 明确操作模式，通常是 `compile`。
2. 读取输入材料。
3. 分离领域内容与机制内容：哪些只是领域经验，哪些可能包含跨领域机制。
4. 对可能的机制内容，提取机制核、参与角色、成立条件。
5. 判断是否可脱域：能否脱离原领域的具体对象，仍保留完整机制。
6. 如果能脱域，抽象为机制原子；如果不能，保留为领域内容。
7. **判断原子层级**：这是基础概念（L0）、基础关系（L1）、组合概念（L2）、机制原子（L3）、命题（L4）还是复合系统（L5）？检查它的 components 是否都处于更低层级。
8. 如果候选引用了库中未定义的概念，先补齐概念定义原子（从更下层开始补齐）。
9. 如果材料涉及互动、关系、交易、竞争、合作、权力、信号或奖惩，先抽互动结构。
10. 按检索规则查已有原子。
11. 能复用旧原子就复用。
12. 旧原子机制匹配但边界太窄，优先升级旧原子。
13. 不能复用时，判断是否值得新建。
14. 太大则拆，太碎则并入字段或丢弃。
15. 写入原子文件，默认 `draft`。正文 wikilink 必须标注语义关系类型。
16. **建立本批原子间关系**：检查本批新建原子之间的 depends_on / components / supports / constrains / contrasts_with 关系。
17. **检查跨领域绑定**：新脱域机制原子是否有已有领域节点可以建立 instantiates 关系。
18. 运行质量检查、硬校验和整批编译验收。
19. **涌现验证**：检查这批原子组合后能否生成原文没有的新判断。
20. 在本次回复中说明复用、新建、升级、拆分、合并、丢弃结果，以及涌现路径。

## Step 1：读取输入材料

输入可以是文章、对话、引用、摘录、想法。

不为每段输入创建 source 文件。只要在生成的原子中写清来源指针即可。

## Step 2：分离领域内容与机制内容

先不要急着拆成原子。先区分两样东西：

**领域内容**：在原领域中有意义，但依赖领域专属概念。
- 例如："研磨度影响萃取速度"——研磨度、萃取都是咖啡专属概念
- 处理方式：保留为领域节点，或作为机制原子的领域绑定实例

**机制内容**：可能包含跨领域可复用的机制结构。
- 例如："当介质与材料接触时，材料中的可迁移成分进入介质；接触面积、温度、时间会改变迁移速度"
- 处理方式：提取为机制原子，并标注原领域来源

判断句：

> 把这句话中的领域专属词（研磨度、咖啡豆、风味）全部替换成抽象角色（变量A、材料、结果），它的机制逻辑还成立吗？

如果成立，它有脱域潜力。如果不成立，它是领域内容。

## Step 3：提取机制四要素

对可能的机制内容，必须能写清：

```text
机制核：发生了什么（因果链或结构约束）
参与角色：哪些东西参与，各自起什么作用
成立条件：机制在什么条件下运作
```

如果不能写全，说明它还不是机制原子。

## Step 4：判断脱域可能性

机制提取后，判断它能脱域到什么程度：

1. 用抽象角色替换领域对象后，机制是否仍然完整？
2. 机制不依赖原领域的任何专属物理/化学/社会条件吗？
3. 能找到至少一个共享此机制的其他领域吗？

如果能通过以上检查，创建机制原子。
如果只能通过 1 但不能通过 2 或 3，先保留为领域绑定节点，标注"待验证脱域"。

## Step 5：创建领域绑定

如果创建的是脱域机制原子，必须给出至少一个领域绑定，说明该机制在具体领域中的角色映射。

如果只有一个领域绑定（就是原领域），在"领域绑定"中写清原领域映射，并标注"待验证其他领域"。

## Step 6：组件拆解与底层追溯

如果候选是上层复合函数，不要直接保存成黑箱：

- 能继续拆出完整机制或概念的就继续拆。
- 拆到再拆会失去完整机制时，才是基础原子。
- 上层原子的 `components` 必须指向下层原子。
- 任意复合原子必须能沿 `components` 和 `depends_on` 追溯到底层基础原子。

如果候选引用了库中没有定义的概念，先补齐概念定义原子，再通过 `depends_on` 链接它。

## Step 7：判断原子层级

每个新建原子必须判断它处于分层构建的哪一层。层级决定了它能引用什么、能被什么引用。

```text
L0 基础概念    components: []  —— 不可再拆的概念/属性/操作
L1 基础关系    components 指向 L0 —— 最简单的三元组（A→关系→B）
L2 组合概念    components 指向 L0/L1 —— 由基础概念和关系组合出更高阶概念
L3 机制原子    components 指向 L0/L1/L2 —— 携带完整四要素
L4 命题        depends_on 指向 L2/L3 —— 一个完整判断，由下层组合而成
L5 复合系统    components 指向 L3/L4 —— 多机制多命题构成的完整系统
```

层级检查：
- 当前原子的所有 components 是否都处于**严格低于**当前层？
- 当前原子的 depends_on 目标是否**不高于**当前层？
- 如果 components 中有与当前原子同层或更高层的 → 要么层级判断错误，要么需要拆出中间层

**不能跨层**：L4 的命题不能直接把 L0 的概念当 component。中间必须有 L2/L3 的组合步骤。

## Step 7：互动结构扫描

当材料涉及多人互动、关系、交易、竞争、合作、权力、信号或奖惩时，先找：

- 玩家：谁在局里。
- 目标：每个玩家想得到什么、避免什么。
- 信息：谁知道什么，谁不知道什么，谁在隐藏什么。
- 策略空间：每个玩家能做什么、不能做什么、能不能退出。
- 规则点：稳定影响选择、信息暴露、成本收益或关系走向的约束和生成机制。
- 奖惩机制：做某动作会获得什么、失去什么、把成本转嫁给谁。
- 信号机制：哪些行为、语言、身体反应、历史记录会泄露隐藏偏好或策略。
- 可能结果：合作、竞争、僵局、退出、背叛、责任转移或稳定均衡。

只有规则点能通过机制精度门时，才沉淀为机制原子。

## Step 8：提取候选机制

先问：

- 这段材料真正提供了什么可复用机制？
- 它的机制核、角色、条件、边界是什么？
- 它为什么成立？
- 它是否只是方法、案例、观点或修辞？
- 如果它来自互动结构，它改变了哪个玩家的选择、信息、成本、收益、信号或责任？

遇到"如何 X""提升 X""解决 X"时，不要把步骤直接建成原子。先提炼其中可复用的底层机制。

## Step 9：机制精度门

候选必须能写清：

```text
机制核：发生了什么
参与角色：哪些东西参与
成立条件：在什么条件下运作
迁移边界：什么时候不适用，不能迁移到哪里
```

写不清就不要新建。可以并入相关原子的"必要说明"，或在本次回复中说明丢弃。

## Step 10：复用优先

创建新原子前，先查旧原子。

判断旧原子能否承载：
- 机制核是否一致？
- 角色结构是否兼容？
- 成立条件是否兼容？
- 迁移边界是否兼容？
- 领域绑定是否重叠？

能准确承载就引用旧原子，不新建。

## Step 11：新建条件

只有同时满足以下条件，才新建原子：

- 携带完整机制（四要素齐全）。
- 有独立复用价值。
- 旧原子无法准确承载。
- 定义清楚。
- 边界清楚。
- 每个引用的概念在库中有清晰定义。
- 缺少底层原理时，已先补齐地基。
- 能给出至少一个领域绑定。
- 如果声称脱域，能在至少一个其他领域验证机制迁移。
- 区分说明具体。
- 能被未来任务或上层原子引用。

## Step 12：写入原子

每个原子最小 frontmatter：

```yaml
id:
status:
source:
depends_on:
components:
```

正文必须包含：定义、边界、机制核心、参与角色、成立条件、迁移边界、复用价值、领域绑定（或脱域说明）、区分说明、必要说明、依赖、组件。

正文"依赖"和"组件"使用 Obsidian wikilink：`[[atom-id]] - 理由`

## Step 13：质量检查

按 `docs/05-quality-gates.md` 检查。再按 `docs/07-operation-modes-and-safety.md` 做硬校验，按 `docs/10-usage-experience-and-validation.md` 做整批验收。

## Step 14：写入原子文件

每个原子最小 frontmatter：

```yaml
id:
status:
source:
depends_on:
components:
```

正文必须包含：场景用途、CLC方向、定义、边界、机制核心、参与角色、成立条件、迁移边界、复用价值、领域绑定（或脱域说明）、区分说明、必要说明、依赖、组件。

正文"依赖"和"组件"使用 Obsidian wikilink，**必须标注语义关系类型**：

```markdown
## 组件
- [[语义单位]] - composed_of：知识原子的构件
- [[可复用]] - composed_of：知识原子必须可复用

## 依赖
- [[机会成本]] - prerequisite：不理解机会成本就无法理解此处的权衡
```

## Step 15：建立本批原子间关系

写入所有本批原子后，检查它们**彼此之间**是否存在语义关系：

1. 列出本批所有新建原子的 ID 和层级。
2. 两两检查：
   - 是否存在 depends_on 或 components？（结构关系）
   - 是否存在 supports / constrains / explains / contrasts_with？（逻辑语义关系）
   - 是否存在 instantiates？（一个原子是另一个的领域实例）
3. 发现关系后在相关原子的正文中补充 wikilink 标注。
4. 如果两个原子描述了同一个机制的两个对称方向（如"不足→酸"和"过度→苦"），它们之间至少应有 `contrasts_with`。

**特别检查**：同批创建的领域节点是否共享同一个脱域机制。如果是，在脱域机制原子的领域绑定表中增加对应的新行。

## Step 16：检查跨领域绑定

新创建的脱域机制原子 → 检查已有原子库中是否存在可以建立 `instantiates` 关系的领域节点：

```text
新建 介质成分转移（机制原子，O6）
  → 已有 咖啡萃取（领域节点）
  → 已有 泡茶萃取（领域节点）
  → 已有 煲汤（领域节点）
  → 在介质成分转移的领域绑定表中增加这些行
```

## Step 17：质量检查

按 `docs/05-quality-gates.md` 检查。再按 `docs/07-operation-modes-and-safety.md` 做硬校验，按 `docs/10-usage-experience-and-validation.md` 做整批验收。

## Step 18：涌现验证

编译结束后，检查：

> 这批原子中的某几个组合起来，能否生成**原文中没有直接表述**的新判断？

操作：
1. 列出本批原子的核心机制。
2. 尝试组合 2-3 个不同层级的原子。
3. 看能否推理出原文没有直接写、但底层机制支撑的新结论。

例如，如果本批创建了：
- `atom_A`：介质接触面积影响成分转移速度
- `atom_B`：温度影响成分转移速度
- `atom_C`：过度转移会带出不需要的成分

可以涌现：
> "如果希望通过降低温度来控制苦涩成分的转移，同时又不想牺牲香气成分的转移速度，可能需要利用不同成分对温度的敏感度差异——而不是同时调整温度和时间。"

这个新判断原文没有直接说，但它由 atom_A + atom_B + atom_C 组合推理而来。

**记录涌现**：在编译结果说明中写出涌现路径：
```text
[atom_A] + [atom_B] + [atom_C] → 可涌现判断："……"
```

没有涌现不一定是失败。但如果连续多次编译都零涌现，说明编译产物可能只是原文的重新排列，而不是可独立组合的机制构件。

## Step 19：结果说明

编译结束时，在本次回复中说明：
- 复用了哪些旧原子。
- 新建了哪些 draft 原子（区分机制原子与领域节点）。
- 哪些候选脱域了，脱域验证结果如何。
- 升级、拆分、合并了哪些原子。
- 丢弃了哪些候选，以及为什么丢弃。
- 哪些原子仍需保持 `draft`。

## 编译失败标准

出现以下情况，必须返工：

- 只是摘要或摘录。
- 按文章标题建原子。
- 把方法、案例、观点或修辞误建成机制原子。
- 把换词类比当成跨领域迁移。
- 声称脱域但没有验证机制一致性。
- 没有稳定 ID 或来源指针。
- 机制四要素不完整。
- 复合原子不能追溯到底层基础原子。
- 引用了库中未定义的概念。
- 边界不清，迁移边界缺失。
- 区分说明空泛。
- 太大，混入多个独立机制。
- 太碎，不能被复用。
- 硬校验或整批验收失败。
