# 目标契约 v0

这份文档定义本工程到底要完成什么。

它不是新的执行规则，而是目标真源。其他文档负责“怎么做”，这份文档负责说明“为什么做、做到什么算完成、什么算失败、v0 不做什么”。

## 最终目标一句话

构建一个 Codex skill，使我可以在当前 Agent 打开的项目根目录内，把文章、对话、摘录和想法编译成可复用、可追溯、可组合、可被 AI 调用的 Markdown knowledge atoms；这些 atoms 本质上是知识函数，通过显式依赖和组件关系形成知识图谱，并在未来任务中被持续复用，从而产生知识复利。

## 目标不是什么

这个项目不是：

- 文章摘要工具。
- 普通 Obsidian 双链笔记库。
- 按主题分类的资料仓库。
- 把所有概念都拆进去的百科系统。
- 只服务单篇文章的阅读笔记。
- 一开始就自动化、可视化、工程化到完整产品。

文章、对话、摘录和想法只是输入材料，不是最终目标。

最终目标是沉淀能被未来任务复用的知识函数。

## v0 交付物

v0 只交付必要内容：

1. 一个可触发的 Codex skill：`skills/nix-knowledge-compiler/`。
2. 一个当前工作区根目录内的 atom store：`atoms/`。
3. 一个最小 atom 模板：`templates/knowledge-atom.md`。
4. 一组核心规则文档：`docs/00` 到 `docs/11`。
5. 一个最小 frontmatter schema：`schemas/atom-frontmatter.yaml`。

这些东西存在，并且彼此指向一致，才算 v0 具备可运行基础。

## v0 完成态

v0 完成不是指知识库已经产生“超级涌现”。

v0 完成是指：

> 给 skill 一段真实输入，它能按规则产出 draft atoms，并能在本次回复中说明每个候选为什么被复用、新建、升级、拆分、合并或丢弃；产出的 atom 能通过质量门、硬校验和整批验收，必要时再升级为 stable。

换句话说，v0 的完成态是“编译闭环成立”，不是“知识复利已经大规模发生”。

## v0 成功标准

一次真实样本测试通过时，v0 算成功。

成功标准：

- 能明确选择操作模式：`compile`、`reuse`、`validate`、`query` 或 `refactor`。
- 能从输入材料提取候选概念，而不是只做摘要。
- 新建 atom 前能检索旧 atom。
- 能说明复用了哪些旧 atom。
- 能说明为什么某些旧 atom 不能准确承载新候选。
- 新建 atom 默认是 `draft`。
- 每个新 atom 都有 `id`、`status`、`source`、`depends_on`、`components`。
- 每个新 atom 都能写清“输入 -> 操作 -> 输出”。
- 每个新 atom 都写清定义、边界、底层机制、复用价值、区分说明、必要说明、依赖和组件。
- 每个新 atom 都写清调用方式：未来遇到什么问题时应该引用它。
- 每个新 atom 都能说明它能接入哪个更大 atom 或系统 atom；暂时没有时，写清上层系统候选。
- 基础 atom 的 `components: []`，且再拆就失去完整函数。
- 系统 atom 的 `components` 不为空，且能递归追溯到底层基础 atom。
- 依赖底层学科原理时，必须链接到底层原理 atom。
- 缺少底层原理 atom 时，必须先补齐地基 atom，再创建上层 atom。
- 互动、关系、商业、策略材料能先抽互动结构，再沉淀可复用规则点。
- 正文依赖和组件使用 `[[atom-id]]`，让 Obsidian 能产生反向链接。
- 失败候选不进入 atom store。
- `depends_on` 和 `components` 没有被普通相关关系污染。
- 没有明显重复 atom。
- 没有把文章摘要误当 atom。
- 整批结果能解释为什么增强了知识图谱。

## v0 失败标准

出现以下情况，说明 v0 还没成功：

- skill 只总结输入材料，没有生成可复用 atom。
- AI 看到新概念就新建，没有先查旧 atom。
- 新建 atom 不能说明为什么旧 atom 不能承载。
- 输出大量细碎概念，但不能被未来任务复用。
- 输出节点看不出怎么调用。
- 输出节点看不出属于或可接入哪个更大系统。
- 输出节点不能写清输入、操作、输出。
- 大节点没有 components，或者 components 不能追溯到底层基础 atom。
- 依赖底层学科原理，却没有对应底层原理 atom 链接。
- 依赖底层学科原理，但没有先补齐缺失的地基 atom。
- 输出节点没有正文 wikilink，无法形成 Obsidian 反向链接。
- 对互动、关系、商业、策略材料只做摘要，没有抽出玩家、规则点、奖惩和信号机制。
- 输出一个大而全的总结型 atom，里面混入多个独立机制。
- 把例子、句子、修辞、情绪当成 atom。
- `depends_on` 写成“相关概念”。
- `components` 写成“相关阅读列表”。
- 失败候选进入了 atom store。
- 新 atom 直接标记为 `stable`，但没有经过质量门、硬校验和整批验收。
- ID 不稳定，文件名和 ID 不一致。
- 文档之间互相矛盾，导致 skill 不知道按哪份规则执行。

## 核心能力边界

v0 必须支持：

- `compile`：把输入材料编译成 draft atoms。
- `reuse`：只查旧 atom，不写入。
- `validate`：检查 atom 或 store 是否合格。
- `query`：用已有 atom 辅助思考。
- `refactor`：升级、拆分、合并、废弃旧 atom。

v0 不支持：

- 跨项目全局知识库。
- 自动写入用户全局目录。
- 单独的运行日志目录。
- 单独的原文快照目录。
- 手写索引目录。
- 手写图谱地图。
- 示例 atom 目录。
- 自动生成完整图谱可视化。
- 自动计算所有反向引用并写回文件。
- 自动批量迁移旧 atom。
- 自动同步到远程数据库。
- 完整 UI。
- 大规模自动化脚本。

这些功能即使以后要做，也不能进入 v0 核心。

## 可否证

这个目标允许失败。

如果第一次真实样本测试后发现：

- 产物只是摘要；
- 没有复用判断；
- atom 无法被未来任务引用；
- 依赖和组件关系无法形成可用图谱；

那就说明 v0 没有通过。

## 下一步行动

完成目标契约后，下一步不是继续补理论。

下一步是做第一次真实样本测试：

```text
选择一段真实文章或对话
-> 用 nix-knowledge-compiler 进入 compile 模式
-> 生成 draft atoms
-> 做质量门、硬校验、整批验收
-> 根据失败点微调规则
```

## 一句话

这个项目的 v0 目标不是“建成一个完美知识库”，而是“证明一段真实输入可以被稳定编译成可复用、可被 AI 调用的知识 atom”。
