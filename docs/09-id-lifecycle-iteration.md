# ID、生命周期、可观察与迭代规则 v0.4

定义原子如何长期演化，以及如何保证它可以被观察、更新、替换和合并。

## ID 规则

ID 使用输入材料的主语言。

```text
概念短名称
```

规则：
- 文件名等于 ID。
- 中文输入生成中文 ID、中文文件名、中文 H1、中文正文标题。
- 英文输入生成英文。
- 混合输入按主要语言生成；必要术语可以保留原词。
- ID 一旦被引用，必须保持稳定。

## 版本规则

不升版本：普通改写、修正错别字、补充例子、补充边界、补充来源、补充领域绑定、补充迁移边界说明。

需要新版本或新原子：定义实质变化、机制核实质变化、成立条件变化、迁移边界变化、依赖结构变化。

优先新建语义更准确的原子。只有确实是同一原子的新版，才使用 `v2`。

## 状态规则

| status | 含义 |
|---|---|
| `draft` | 初次生成，尚未通过完整检查 |
| `stable` | 结构和关系通过质量门，可以被其他原子引用 |
| `deprecated` | 不建议新引用，但保留追溯 |
| `merged` | 已合并到另一个原子 |

`stable` 不表示"内容为真"。它只表示这个原子的身份、来源指针、边界、机制四要素、依赖和组成关系已经稳定到可以被复用。

## 生命周期

```text
draft -> stable
draft -> 丢弃
stable -> deprecated
stable -> merged
stable -> 新版本或替代原子
```

## 可观察规则

原子必须保留未来能被回看和更新的接口：

- `source` 能追溯它从哪里来。
- `depends_on` 能看到它依赖哪些前置原子。
- `components` 能看到它由哪些下层原子组成。
- 正文能看到它的机制四要素、领域绑定、脱域说明。

## 迭代规则

- 发现表达不清：直接改正文，不升版本。
- 发现边界不清：补边界或迁移边界，不升版本。
- 发现领域绑定不完整：补绑定。
- 发现旧原子无法承载新含义：新建替代原子。
- 发现两个原子重复：合并，标记 `merged`。
- 发现领域节点可以脱域：进入 `de-domain` 模式评估，通过后升级为机制原子。

## 领域绑定演化

一个机制原子的领域绑定不是固定不变的：

- 发现新的共享该机制的领域 → 新增领域绑定行。
- 发现某个绑定实际上是换词类比 → 移除该绑定。
- 领域绑定的演化不升版本，只在正文"领域绑定"表中增删行。

## 脱域升级

领域节点在满足脱域条件后，可以升级为机制原子：

1. 在 `de-domain` 模式下验证机制可脱域。
2. 重写正文：将领域专属词替换为抽象角色。
3. 原领域作为第一个领域绑定。
4. 验证至少一个其他领域。
5. 补充迁移边界。
6. 旧领域节点标记为 `merged` 或 `deprecated`，指向新的机制原子。

## 冲突规则

冲突原子可以并存。处理方式：
- 写清各自适用边界和迁移边界。
- 写清冲突点。
- 判断是否来自不同前提、不同系统、不同层级。
- 必要时抽象出一个更高阶原子，用来解释冲突为什么存在。

## 依赖循环规则

`depends_on` 和 `components` 原则上不能形成循环。循环必须修复。

## depends_on 与 components 边界

- 没有它就理解不了当前原子：写 `depends_on`。
- 当前原子是由它组成的：写 `components`。
- 只是相关、相似、启发：不写进核心关系。
