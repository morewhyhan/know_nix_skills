# ID、生命周期、可观察与迭代规则 v0

这份文档定义 atom 如何长期演化，以及如何保证它可以被观察、更新、替换和合并。

本工程不管理 atom 的“真伪分数”或“可信度评分”。知识库的复利依赖稳定身份、来源可追溯、依赖显式、变更路径清楚。

一个 atom 进入系统，不代表它永远正确；只代表它已经成为一个可观察、可复用、可迭代的知识对象。

## ID 规则

ID 使用输入材料的主语言，而不是统一翻译成英文。

```text
概念短名称-v1
```

例子：

```text
关系行为惯性-v1
失败归因暴露可靠性-v1
言外之意的意图分类-v1
```

规则：

- 文件名等于 ID：`关系行为惯性-v1.md`。
- 中文输入生成中文 ID、中文文件名、中文 H1、中文正文标题。
- 英文输入生成英文 ID、英文文件名、英文 H1、英文正文标题。
- 混合输入按主要语言生成；必要术语可以保留原词。
- 不把中文概念翻译成英文 slug，也不优先使用拼音。
- ID 一旦被引用，必须保持稳定；只有语义实质变化时才新建替代 atom。

## 版本规则

不升版本的情况：

- 普通改写。
- 修正错别字。
- 补充例子。
- 补充边界。
- 补充来源。
- 补充底层机制、复用价值、区分说明或必要说明的表述。

需要新版本或新 atom 的情况：

- 定义发生实质变化。
- 底层机制发生实质变化。
- 适用边界发生变化。
- 依赖结构发生变化。
- 原 atom 已经不能准确承载新含义。

优先新建语义更准确的 atom。只有确实是同一 atom 的新版，才使用 `v2`。

## 状态规则

核心状态只保留：

| status | 含义 |
|---|---|
| `draft` | 初次生成，尚未通过完整检查 |
| `stable` | 结构和关系通过质量门，可以被其他 atom 引用 |
| `deprecated` | 不建议新引用，但保留追溯 |
| `merged` | 已合并到另一个 atom |

`rejected` 不是 atom 状态。失败候选不进入 atom store。

`stable` 不表示“内容为真”。它只表示这个 atom 的身份、来源指针、边界、依赖和组成关系已经稳定到可以被复用。

## 生命周期

生命周期流转：

```text
draft -> stable
draft -> 丢弃
stable -> deprecated
stable -> merged
stable -> 新版本或替代 atom
```

`deprecated` atom 可以保留旧引用，但不建议新引用。

`merged` atom 必须在正文中指向承载它的新 atom，并写清合并原因。不新增 `replaced_by`、`merge_note` 等 frontmatter 字段。

## 可观察规则

atom 必须保留未来能被回看和更新的接口：

- `source` 能追溯它从哪里来。
- `depends_on` 能看到它依赖哪些前置 atom。
- `components` 能看到它由哪些下层 atom 组成。
- 正文能看到它的定义、边界、底层机制、复用价值和区分说明。
- 正文能看到它为什么被合并、废弃或替换。

这些接口保证未来可以观察它，而不是保证它绝对为真。

## 迭代规则

不需要给 atom 打可信度分数。

需要保证它可以被迭代：

- 发现表达不清：直接改正文，不升版本。
- 发现边界不清：补边界，不升版本。
- 发现依赖不清：修正 `depends_on`。
- 发现组件不清：修正 `components`。
- 发现旧 atom 无法承载新含义：新建替代 atom。
- 发现两个 atom 重复：合并，并标记 `merged`。
- 发现旧 atom 不再建议使用：标记 `deprecated`。

真伪不是核心字段。可观察、可替换、可合并、可废弃，才是核心接口。

## 断言边界

如果一个 atom 的适用边界不清，不要用“可信度低”处理。

正确处理方式是：

- 补清边界。
- 补清来源指针。
- 补清它依赖什么。
- 补清它适用于什么任务。
- 必要时保持 `draft`，直到边界足够清楚。

`draft` 不是“不可信”。`draft` 表示它还没准备好被稳定复用。

## 冲突规则

冲突 atom 可以并存。

不要因为冲突就强行合并，也不要因为相似就删除。

处理方式：

- 写清各自适用边界。
- 写清冲突点。
- 判断是否来自不同前提、不同系统、不同层级。
- 必要时抽象出一个更高阶 atom，用来解释冲突为什么存在。

冲突管理的目标不是裁判谁真谁假，而是保证未来调用时能看见边界差异，并能继续迭代。

## 依赖循环规则

`depends_on` 和 `components` 原则上不能形成循环。

如果出现循环，通常说明：

- 依赖方向写反。
- 把普通相关误写成依赖。
- 两个 atom 实际上需要抽象出第三个更底层 atom。
- 某个“大 atom”被误当成它自己的组件。

循环必须修复。不能 stable。

## depends_on 与 components 边界

`depends_on` 是理解或使用当前 atom 的前置条件。

`components` 是当前 atom 的结构组成部分。

不是所有前置条件都是组件。不是所有组件都是理解前置。

判断句：

- 没有它就理解不了当前 atom：写 `depends_on`。
- 当前 atom 是由它组成的：写 `components`。
- 只是相关、相似、启发：不写进核心关系。
