# 使用体验、AI 调用与整批验收规则 v0

这份文档定义知识库如何被人和 AI 使用，以及一次编译如何验收。

目标是让知识库不只是 Obsidian 笔记，而是未来 AI 思考时可以调用的底座。

## AI 调用知识库协议

AI 使用知识库处理任务时，按以下顺序：

1. 明确当前任务。
2. 检索相关 atom。
3. 读取直接命中的 atom。
4. 必要时展开 `depends_on`。
5. 如果任务需要系统结构，再展开 `components`。
6. 默认最多展开 2 层。
7. 不读取无关 atom。
8. 任务结束后判断是否有新 atom 值得沉淀。

`depends_on` 用于补前置理解。`components` 用于理解系统结构。

不要为了“多读一点”而展开整个图谱。

注意：这是任务查询时的读取限制，不是 `compile` 时的地基追溯限制。编译新 atom 时，必须追溯到足以支撑当前 atom 的底层基础 atom 或底层学科原理 atom。

## 反向沉淀

AI 完成任务后，只有满足以下条件才建议沉淀新 atom：

- 这次产生了可复用知识函数。
- 不是一次性回答。
- 未来任务可能再次使用。
- 旧 atom 不能准确承载。
- 能写清来源和边界。

否则只作为本次任务输出，不进入 atom store。

## 整批编译验收

一次 `compile` 结束后，不能只检查单个 atom。

必须检查整批结果：

- 是否漏掉输入材料中的核心概念。
- 是否把摘要当成 atom。
- 是否拆得太碎。
- 是否拆得太粗。
- 是否重复新建已有 atom。
- 是否充分复用旧 atom。
- 新增 atom 是否增强了依赖图或组件图。
- 新增 atom 是否写清“输入 -> 操作 -> 输出”。
- 基础 atom 是否真的不可再拆。
- 系统 atom 是否能沿 `components` 和 `depends_on` 追溯到底层基础 atom 或底层学科原理 atom。
- 缺少底层学科原理时，是否先补齐了地基 atom。
- 新增 atom 是否写清调用方式。
- 新增 atom 是否能接入某个更大系统或上层 atom。
- 依赖和组件是否用正文 wikilink 形成 Obsidian 反向链接。
- 是否有不确定项。
- 本次回复是否说明主要取舍。

整批不合格时，新 atom 保持 `draft`。

## 系统 atom 创建规则

系统也是 atom。不要单独创建 `systems/` 文件类型。

只有满足以下条件，才创建系统 atom：

- `components` 不为空。
- 它不是简单列表。
- 组件组合后产生了单个 atom 没有的新能力。
- 它能服务未来任务。
- 正文能解释这些组件如何共同运转。
- 它能沿 `components` 和 `depends_on` 递归追溯到底层基础 atom。
- 如果依赖底层学科原理，正文“依赖”必须链接到对应底层原理 atom。

系统 atom 的正文至少写：

- 系统定义。
- 系统边界。
- 系统底层机制。
- 系统复用价值。
- 系统区分说明。
- 系统必要说明。
- 系统依赖。
- 系统组件。

系统 atom 不能是黑箱。凡是它声称拥有的能力，都必须能在“组件”里找到来源；凡是它依赖的底层学科原理，都必须能在“依赖”里找到链接。

## Obsidian 展示规则

Obsidian 是阅读入口，不是核心数据模型。

规则：

- 文件名等于 ID。
- H1 标题必须跟随输入材料主语言。
- frontmatter 保持最小。
- `depends_on` 和 `components` 在 frontmatter 中维护机器可读 ID。
- 正文“依赖”和“组件”必须同步写 `[[atom-id]] - 理由`，用于阅读和 Obsidian 反向链接。
- Obsidian 双链只是阅读呈现，不能替代 frontmatter 的 `depends_on` 和 `components`。
- 不手写 index 页面。

正文链接是阅读辅助。真正的结构关系仍以前置 frontmatter 为准。

不要新增 `part_of`、`used_by`、`related`、`see_also` 等字段。反向关系通过 Obsidian backlinks、全库搜索、以及系统 atom 的 `components` 反查得到。

每一层 atom 都必须向下链接：上层系统 atom 链到下层组件 atom；下层组件 atom 继续链到自己的组件或依赖；直到抵达 `components: []` 的基础 atom，或承载底层学科原理的基础 atom。

## 语言规则

- atom 的可读部分必须跟随输入材料的主语言。
- 中文输入时，文件名、`id`、H1、正文标题和正文都使用中文。
- 英文输入时，文件名、`id`、H1、正文标题和正文都使用英文。
- 混合输入按主要语言生成，必要术语可以保留原词。
- 同一概念的中英文叫法写入“常见别名”。
- 不因为中英文表达不同就新建 atom。

中文 atom 的正文标题固定使用：

- 定义
- 边界
- 底层机制
- 复用价值
- 区分说明
- 必要说明
- 依赖
- 组件

不要把中文输入生成 `Definition`、`Boundary`、`Underlying mechanism` 等英文标题。

## 真实样本测试

补齐规则后，必须用真实材料跑一次完整测试。

测试目标不是产出完美知识库，而是暴露规则漏洞。

测试流程：

1. 选择一篇文章或一段真实对话。
2. 按 `compile` 模式处理。
3. 生成候选概念。
4. 执行旧 atom 检索。
5. 判断复用、新建、升级、拆分、合并、丢弃。
6. 写入 draft atom。
7. 运行单 atom 质量门。
8. 运行整批编译验收。
9. 根据失败点修改规则。

第一次测试成功标准：

- 没有明显重复 atom。
- 没有把摘要当 atom。
- `depends_on` 和 `components` 没有乱写。
- 每个新 atom 都能写清“输入 -> 操作 -> 输出”。
- 基础 atom 不能再拆出完整函数。
- 系统 atom 可以递归追溯到底层基础 atom 或底层学科原理 atom。
- 上层 atom 不悬空；缺少底层学科原理时，已先补齐地基 atom。
- 每个新 atom 都能说清定义、边界、底层机制、复用价值和区分说明。
- 每个新 atom 都能说清调用方式和上层系统候选。
- 每个新 atom 的依赖和组件正文链接可被 Obsidian 反查。
- 本次回复能解释主要决策。
