# 构建裁决规则 v1.1

唯一机器契约见 [`../skills/nix-knowledge-compiler/references/nix-knowledge-compiler-v1.1.md`](../skills/nix-knowledge-compiler/references/nix-knowledge-compiler-v1.1.md)。

## 准确承载

旧 atom 准确承载新候选，必须同时满足：

- 输入、操作和输出兼容。
- 中介机制相同。
- 参与角色及职责兼容。
- 成立条件与迁移边界兼容。
- 未来调用方式一致。
- 依赖和组件结构兼容。

只是名称、案例、来源或领域不同，不能成为新建理由；只是结论相似，也不能成为复用理由。

## 裁决顺序

1. **复用**：旧 atom 能准确承载；补来源或绑定即可。
2. **升级**：机制相同，但旧 atom 的证据、边界、别名或绑定过窄；非破坏性扩展保持原 ID。
3. **拆分**：候选含多个可独立触发和调用的机制。
4. **合并**：多个 atom 实质是同一函数；`merged` 去向链必须无环并最终终止于 `draft` 或 `stable` 活动 atom。
5. **新建**：多路检索后仍无准确承载者，且候选通过函数、证据和假迁移门。
6. **丢弃**：没有完整函数或独立复用价值。

裁决由 AI 完成。内部可使用候选表、反方审查或盲审，但不要求用户批准每个候选。

## 基础机制与复合系统

- 基础机制允许 `components: []`。
- 不因缺少 components 而制造词语、关系或层级占位 atom。
- 只有多个下层函数协作后产生新能力时才创建复合系统。
- 复合系统必须列出真实 components，并解释组合逻辑。

## 新建理由

新建说明至少回答：

- 检查了哪些最相近旧 atom。
- 为什么它们在签名、机制、条件或边界上不能承载。
- 来源直接支持什么。
- 编译器推导了什么。
- 哪些迁移只是待验证推演。
- 当前 atom 未来如何被调用。

## 假迁移裁决

亲密关系、职场、商业、政治等领域可能共享角色名称，却不一定共享机制。以下情况必须判为不能复用：

- 同样出现“投入”，但一个由沉没成本锁定，另一个由外部评价基线锁定。
- 同样出现“不行动”，但一个是有能力后主动克制，另一个是缺乏权力或能力。
- 同样出现“给予”，但一个保留自由选择，另一个以回报承诺换取劳动。

迁移裁决看中介机制，不看情绪词、人物关系或最终结果。
