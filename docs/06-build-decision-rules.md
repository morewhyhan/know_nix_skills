# Atom 构建判定规则 v0

这份文档决定一个候选知识点应该被复用、新建、升级、拆分、合并，还是丢弃。

它是本工程的构建规则，作用类似 Nix 里的构建约束：不要靠感觉分类，而要靠稳定规则决定什么能进入 store。

## 核心原则

atom store 只保存已经值得长期复用的知识函数。

输入材料不是知识资产。输入材料只是 source。只有被编译成稳定、可引用、可组合的 atom 后，它才进入知识资产层。

候选内容默认不进入 atom store。它必须通过构建判定。

## 判定顺序

处理每个候选知识点时，按以下顺序判断：

1. **函数签名判断**：它能否写成“输入 -> 操作 -> 输出”。
2. **基础/系统判断**：它是不可再拆的基础函数，还是由多个函数组合成的系统函数。
3. **底层追溯判断**：它能否沿 `components` 和 `depends_on` 追溯到底层基础 atom 或底层学科原理 atom；缺地基时必须先补齐地基 atom。
4. **互动结构判断**：如果材料涉及互动、关系、交易、竞争、合作、权力、信号或奖惩，先抽玩家、目标、信息、策略空间、规则点、奖惩、信号和可能结果。
5. **复用价值判断**：它是否离开原文仍能被未来任务调用。
6. **机制精度判断**：它能否写清“触发条件 -> 中介机制 -> 可观察结果 -> 适用边界”。
7. **丢弃或归位**：如果只是方法、案例、证据、观点、误区、修辞，不建 atom，并入相关 atom 的“必要说明”或丢弃。
8. **检索**：按 `docs/08-reuse-search.md` 查旧 atom。
9. **复用**：如果已有 atom 能准确承载，引用旧 atom。
10. **升级**：如果旧 atom 底层机制一致但边界、复用价值或依赖结构太窄，优先升级旧 atom。
11. **新建**：只有旧 atom 无法准确承载，且候选具备独立复用价值并通过机制精度门时，才新建 atom。
12. **拆分**：如果候选混入多个独立函数，先拆成多个候选。
13. **合并**：如果多个 atom 实质重复，合并或抽象出更通用 atom。

这个顺序不能反过来。尤其不能先新建，再事后考虑复用。

## 什么叫准确承载

一个旧 atom 能准确承载新候选，必须同时满足：

- **函数签名兼容**：输入类型、操作方式、输出类型相同或兼容。
- **底层机制一致**：两者解释的是同一种因果链、结构约束或生成机制。
- **边界兼容**：旧 atom 的适用范围不会把新候选说歪，也不会过度扩大。
- **复用功能兼容**：未来任务引用它时，得到的是同一种知识能力。
- **依赖结构兼容**：理解或使用它需要的前置 atom 没有本质差异。

只要满足以上条件，就复用旧 atom，不新建。

如果只是词不一样、例子不一样、来源不一样，但底层机制、边界和依赖结构一致，也复用旧 atom。

## 什么叫独立复用价值

候选内容有独立复用价值，必须满足：

- 离开原文仍能被理解。
- 未来任务可以直接引用它。
- 它能支持更大的 atom 或系统 atom。
- 它能说清“调用方式”：遇到什么问题时该引用它。
- 它不是单纯例子、句子、标签、情绪、修辞或一次性表达。
- 它比原文更稳定、更根本，而不是把原文换一种说法。

没有独立复用价值，就不进入 atom store。最多把它写进某个 atom 的“必要说明”。

## 什么叫基础 atom

基础 atom 是不可再拆的最小知识函数。

必须满足：

- 能写成“输入 -> 操作 -> 输出”。
- `components: []`。
- 再拆下去，输入、操作、输出至少一个会缺失。
- 能被上层 atom 作为组件或依赖调用。

不是基础 atom：

- 只有输入，如“表达内容”。
- 只有操作，如“抽取”。
- 只有输出，如“隐藏意图”。
- 还能继续拆出多个完整函数的候选。

## 什么叫系统 atom

系统 atom 是多个知识函数组合出的更大函数。

必须满足：

- `components` 不为空。
- 组件组合后产生单个组件没有的新能力。
- 每个组件都用 `[[atom-id]] - 组件职责` 链接。
- 能递归追溯到 `components: []` 的基础 atom。
- 如果依赖底层学科原理，必须在 `depends_on` 和正文“依赖”里链接到底层原理 atom。

不能追溯到底层基础 atom 的系统 atom，是黑箱，不新建。

## 什么叫地基 atom

地基 atom 是承载底层学科原理的基础 atom。

它不是“学科名”，而是一个可复用知识函数。

例如：

- 机会成本：输入一个选择和可替代选项，操作是识别被放弃的最高价值替代用途，输出是真实成本判断。
- 贝叶斯更新：输入先验判断和新证据，操作是按证据强度调整概率，输出后验判断。
- 反馈回路：输入系统行为及其结果，操作是识别结果如何反过来改变后续行为，输出强化或平衡路径。

地基 atom 必须满足：

- 能写成“输入 -> 操作 -> 输出”。
- `components: []`，除非它本身仍需要更底层的组件。
- 可以被多个上层 atom 依赖。
- 在正文“必要说明”里说明底层学科来源。
- 如果由当前材料触发补齐，在正文“必要说明”里写明。

## 什么叫机制精度

新建 atom 必须能写清：

```text
触发条件 -> 中介机制 -> 可观察结果 -> 适用边界
```

四个位置的意思：

- 触发条件：什么情况下这个机制开始起作用。
- 中介机制：中间发生了什么，为什么会从条件走向结果。
- 可观察结果：它会造成什么可识别的变化。
- 适用边界：在哪些情况下不能这么用。

如果只能写成“这个方法能帮你做某事”，不新建。

## 什么叫规则点

规则点不是文章里的“规则清单”，而是稳定影响系统运转的机制。

一个规则点必须影响至少一项：

- 玩家选择。
- 信息暴露或信息隐藏。
- 成本、收益、奖惩或责任分配。
- 策略空间。
- 信号生成和信号识别。
- 合作、竞争、退出、背叛、僵局或均衡结果。
- 关系走向或系统演化。

规则点可以来自心理学、生理学、社会学、经济学、博弈论、语言学、传播学等学科。学科只是来源，不是 atom 分类。

不能成为规则点的内容：

- 只有名字的标签。
- 单个玩家或人物。
- 单个案例。
- 一次性建议。
- 只有态度没有机制的价值判断。
- 无法观察、无法迭代、无法复用的解释。

## 什么时候复用旧 atom

满足以下条件时，复用旧 atom：

- 新候选只是旧 atom 的新例子。
- 新候选只是旧 atom 在另一个来源里的表达。
- 新候选只是旧 atom 在另一个系统中的应用，但没有改变底层机制、边界、依赖结构和复用功能。
- 旧 atom 已经能让未来任务完成同样的推理。

复用时，可以在新产出的上层 atom 里引用旧 atom；不要为了“这篇文章里也提到了”而新建重复 atom。

## 什么时候升级旧 atom

如果旧 atom 的底层机制一致，但写得太窄，优先升级旧 atom，而不是新建平行 atom。

升级旧 atom 的条件：

- 新候选和旧 atom 的底层机制一致。
- 新候选暴露了旧 atom 的边界、复用价值、必要说明或依赖结构过窄。
- 扩写后不会让旧 atom 变得含糊。
- 扩写后旧 atom 能服务更多系统或任务。

升级时，保留稳定 ID。普通表达优化、边界补充、例子补充，不需要新版本。

只有当 atom 的含义发生实质变化，才考虑新建替代 atom，并把旧 atom 标为 `deprecated` 或 `merged`。

## 什么时候新建 atom

只有同时满足以下条件，才新建 atom：

- 有独立复用价值。
- 旧 atom 无法准确承载。
- 定义清楚。
- 函数签名清楚。
- 边界清楚。
- 底层机制清楚。
- 能追溯到底层基础 atom 或底层学科原理 atom。
- 缺少底层学科原理时，已经先创建或复用地基 atom。
- 复用价值明确。
- 调用方式明确。
- 区分说明具体。
- 能接入某个更大 atom 或系统 atom；暂时没有时，写清上层系统候选。
- 未来可以被任务、AI 推理或更大 atom 引用。
- 不是只服务当前文章的一次性表达。

## 什么时候拆分

候选太大时，先拆分，不要直接写成一个大 atom。

拆分信号：

- 同时包含多个可独立复用函数。
- 不同未来任务只需要其中一部分。
- 它的依赖来自几组互不相干的前置 atom。
- 它同时承担概念解释、方法建议、案例证明和系统总结。

拆分后的每个 atom 仍然必须有独立复用价值并通过机制精度门。不能为了拆而制造碎片。

## 什么时候合并

发现两个 atom 实质重复时，合并。

合并条件：

- 底层机制一致。
- 边界一致或可以被一个更清楚的边界统一。
- 复用功能一致。
- 依赖结构兼容。
- 分开保存不能带来额外复用价值。

合并后，被替代的 atom 标为 `merged`，并在正文中写清合并原因和承载它的新 atom。

## 材料角色处理

方法、案例、证据、观点、误区、修辞不是默认 atom。

- 方法步骤：提炼其中的底层机制；提不出来就不建。
- 案例故事：作为说明材料，除非案例本身会被长期复用。
- 证据数据：支撑定义、边界或必要说明。
- 作者观点：转成可复用知识函数，否则丢弃。
- 误区提醒：写入边界；形成稳定失败机制时才考虑独立 atom。
- 修辞夸张：标准化，不直接成卡。

例子不能默认丢弃。先判断它是否承载了独立机制。

## 批量候选之间也要去重

一次 compile 里，即使原来的 `atoms/` 为空，也必须先比较本批候选。

不能写“拆解前 atoms 为空”作为新建理由。必须具体检查旧 atom 或本批候选是否能承载。

## 丢弃不是状态

`rejected` 不是 atom 的核心状态。

如果候选内容不合格，它就不进入 atom store。可以在本次回复中说明“为什么丢弃”，但不要创建一个 `status: rejected` 的 atom。

核心状态只保留：

- `draft`
- `stable`
- `deprecated`
- `merged`

## 最终判定句

新建 atom 前，必须能说清楚：

> 这个候选是一个可复用知识函数；能写清输入、操作、输出；如果是基础 atom，再拆就失去完整函数；如果是系统 atom，能沿 components 和 depends_on 追溯到底层基础 atom 或底层学科原理 atom；旧 atom 不能准确承载它；未来可以被其他任务或更大 atom 引用。

说不清，就不要新建。
