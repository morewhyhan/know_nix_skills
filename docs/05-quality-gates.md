# 质量检查规则

用于检查一个候选内容能否进入 atom store。

## Gate 1：身份

- [ ] 有稳定 `id`。
- [ ] 有 `status`。
- [ ] ID 带版本后缀。
- [ ] 文件名等于 ID。
- [ ] 文件名、ID、H1、字段标题跟随输入材料主语言。

不通过：不能进入 atom store。

## Gate 2：来源

- [ ] 有 `source` 指针。
- [ ] 能说明来源类型、标题或定位方式。

不通过：补来源；无法补来源则不要标记为 stable。

## Gate 3：候选资格

- [ ] 它有独立复用价值。
- [ ] 不是文章标题或小标题。
- [ ] 不是方法步骤。
- [ ] 不是案例、人物、金句、修辞。
- [ ] 不是只服务当前材料的局部表达。
- [ ] 不是换词类比（句式相同但机制不同）。

不通过：并入已有原子的"必要说明"或丢弃。

## Gate 4：定义

- [ ] 能回答"它是什么"。
- [ ] 不循环定义。
- [ ] 不用案例代替定义。
- [ ] 不用方法代替定义。

不通过：不能 stable。

## Gate 5：机制核

- [ ] 能写清因果链或结构约束。
- [ ] 不只是"这个方法有用"。
- [ ] 不把技巧命名、建议或标签当机制。
- [ ] 能写成"触发条件 → 过程机制 → 可观察结果"。

不通过：不能进入 atom store。

## Gate 6：参与角色

- [ ] 能列出所有参与角色。
- [ ] 每个角色有明确的功能说明。
- [ ] 角色不依赖原领域的专属对象（如果是脱域原子）。

不通过：补角色说明，或降级为领域节点。

## Gate 7：成立条件

- [ ] 能说清机制在什么条件下运作。
- [ ] 条件不是列"所有情况"。
- [ ] 条件可以用来判断"什么时候这个机制不适用"。

不通过：不能 stable。

## Gate 8：迁移边界

- [ ] 能写清什么情况下不适用。
- [ ] 能写清不能迁移到哪类领域。
- [ ] 能说清为什么不能迁移（不只是"领域不同"）。
- [ ] 边界不是百科式扩展，只保留关键约束。

不通过：不能 stable。对于声称可脱域的机制原子，迁移边界缺失是致命缺陷。

## Gate 9：脱域验证

仅适用于声称可脱域的机制原子：

- [ ] 机制核不依赖原领域的专属概念。
- [ ] 用抽象角色替换领域对象后，机制仍然完整。
- [ ] 能找到至少一个共享此机制的其他领域（不只是句式相同）。
- [ ] 迁移到新领域后能生成有效判断（不是换词造句）。
- [ ] 如果当前只有一个领域绑定，已标注"待验证其他领域"。

不通过：降级为领域节点，或标注"待验证脱域"。

## Gate 10：不可拆分性

- [ ] 如果 `components: []`，再拆就失去完整机制。
- [ ] 如果还能拆出独立机制，必须拆成下层原子。
- [ ] 如果当前原子是复合原子，`components` 不为空。
- [ ] 复合原子的组件组合后产生单个组件没有的新能力。

不通过：继续拆分，或降级为复合原子。

## Gate 11：底层追溯

- [ ] 当前原子能沿 `components` 和 `depends_on` 追溯到底层基础原子。
- [ ] 依赖底层学科原理时，已链接到底层原理原子。
- [ ] 如果底层原理原子原本不存在，已先创建或复用。
- [ ] 地基原子不是学科标签，而是完整知识函数。

不通过：补底层原子、补依赖链，或不要新建。

## Gate 12：边界

- [ ] 写清它不是什么。
- [ ] 写清容易和什么混淆。
- [ ] 写清什么情况下会被误用。
- [ ] 写清适用边界。

不通过：不能 stable。

## Gate 13：复用价值

- [ ] 能说明它未来可以支持哪些任务。
- [ ] 能说明它可能被哪些更大原子引用。
- [ ] 能写清调用方式。
- [ ] 它不是一次性回答。

不通过：不要新建原子。

## Gate 14：区分说明

- [ ] 已检查是否存在可承载的旧原子。
- [ ] 已比较本批候选之间是否重复。
- [ ] 如果旧原子能准确承载，复用旧原子。
- [ ] 如果新建，能说明为什么旧原子或本批候选不够精准。

不通过：先查旧原子，或先做本批去重。

## Gate 15：粒度

- [ ] 不混入多个独立机制。
- [ ] 不是一个无法独立复用的碎片。
- [ ] 再拆会失去独立用途。
- [ ] 不拆不会混入多个用途。

不通过：拆分、合并或丢弃。

## Gate 16：依赖与组件

- [ ] `depends_on` 和 `components` 字段存在。
- [ ] 依赖为空时明确写 `[]`。
- [ ] 依赖是真正前置条件，不是普通相关。
- [ ] 正文"依赖""组件"与 frontmatter 一致。
- [ ] 正文使用 `[[atom-id]]` 形成 Obsidian 反向链接。

不通过：重建依赖。

## Gate 17：互动结构

- [ ] 如果材料涉及互动、关系、交易、竞争、合作、权力、信号或奖惩，已先抽互动结构。
- [ ] 已识别玩家、目标、信息、策略空间、规则点、奖惩、信号和可能结果。
- [ ] 只把可复用规则点沉淀为原子。

不通过：回到互动结构扫描。

## Gate 18：分层构建

- [ ] 当前原子已判断层级（L0 基础概念 / L1 基础关系 / L2 组合概念 / L3 机制原子 / L4 命题 / L5 复合系统）。
- [ ] 所有 components 处于严格低于当前层。
- [ ] 所有 depends_on 目标不高于当前层。
- [ ] 没有跨层依赖（L4 不能以 L0 为 component）。
- [ ] 如果 components 中有同层原子 → 已拆出中间层或确认层级判断无误。

不通过：重建依赖链或拆分中间层。

## Gate 19：本批关系

- [ ] 本批新建原子之间的语义关系已检查（depends_on / components / supports / constrains / explains / contrasts_with / instantiates）。
- [ ] 对称机制（如"不足→X"和"过度→Y"）之间至少建立了 contrasts_with。
- [ ] 同批领域节点与脱域机制之间的 instantiates 关系已建立。

不通过：补充本批内部关系。

## Gate 20：涌现验证

- [ ] 已检查本批原子能否组合出原文未直接表述的新判断。
- [ ] 如果能涌现 → 编译结果说明中写清了涌现路径。
- [ ] 如果不能涌现 → 确认不是只做了原文重排列。

不通过：不是硬阻塞（有些材料确实涌现空间小），但连续零涌现需要反思编译质量。

## Gate 21：可观察与迭代

- [ ] `source` 能回查。
- [ ] 未来发现问题时，知道应该修改、合并、废弃还是替换。
- [ ] 边界不清时保持 `draft`。
- [ ] `stable` 只表示结构和关系可复用，不表示内容绝对为真。

不通过：不能 stable。

## 状态规则

| status | 含义 |
|---|---|
| `draft` | 初次生成，尚未通过全部检查 |
| `stable` | 通过质量门，可以被其他原子引用 |
| `deprecated` | 不建议新引用，但保留追溯 |
| `merged` | 已合并到另一个原子 |

`rejected` 不是原子状态。失败候选不进入 atom store。

## 最终验收句

一个原子通过检查时，应该能说：

> 我有稳定身份和来源；能写清机制四要素；能判断自己在分层构建中的层级（L0-L5）；我的 components 都处于更低层级，没有跨层依赖；我和本批其他原子之间的语义关系已建立（supports/constrains/explains/contrasts_with/instantiates）；如果我是脱域机制原子，已检查跨领域绑定；我参与的组合能否涌现新判断已记录。
