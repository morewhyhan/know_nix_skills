# 质量门 v1.1

唯一机器契约见 [`../skills/nix-knowledge-compiler/references/nix-knowledge-compiler-v1.1.md`](../skills/nix-knowledge-compiler/references/nix-knowledge-compiler-v1.1.md)。

## A. 结构硬门

写入 store 的 draft 也必须全部通过：

- 文件名等于 `id + ".md"`，ID 唯一。
- frontmatter 只含七字段且字段类型正确。
- `source` 为至少一个可解析路径或 URL 的列表。
- status 只使用 `draft`、`stable`、`deprecated`、`merged`。
- 固定 15 个中文 H2 机器键存在且顺序正确；文件名、ID、H1、别名和章节内容跟随来源主语言。
- 机器解析忽略 fenced code 与 HTML comment。
- “必要说明”八个强制标签各且仅出现一次，包括 `命名依据` 与 `独立语义复核`；复核项只有一个具体状态与具体范围或原因，无冲突状态或省略号。
- CLC 表覆盖全部 22 个基本大类，atom tags 只使用正式 `CODE/类名`，`CLC方向` 记录全表扫描和主类。
- `reconstructions/` 的来源路径、SHA-256、六节结构、替代验收状态和 atom 映射通过硬校验。
- 依赖、组件和正文 wikilink 一致，目标存在。
- `depends_on` 与 `components` 无循环。
- 领域绑定每行有验证状态；`来源支持` 与 `独立材料验证` 的定位以精确 frontmatter source locator 或无歧义 basename 开头，并解析到路径 identity 与内容哈希都不同的来源。
- `merged` 去向链无环，并最终终止于 `draft` 或 `stable` 活动 atom。

结构硬门失败时，AI 必须先自动修复，不能把残缺文件留给用户审批。

## B. 函数语义门

- 能写成“输入 -> 操作 -> 输出”。
- 能写清触发条件、中介机制、可观察结果和边界。
- 不是标题、标签、建议、案例或换词摘要。
- 不能成为 atom 的核心观点、建议、范围和文本问题已经留在来源还原稿。
- ID 已按学科术语、通俗用语、描述性命名、编译器造词的顺序核验。
- 角色和条件足以区分相邻机制。
- 基础机制 `components: []` 时已经是完整最小函数。
- 复合系统 components 非空，且组合产生新能力。

失败时自动拆分、合并、降级为材料说明或丢弃。

## C. 复用门

新建前必须完成 CLC 22 大类全表扫描、专业术语、名称、别名、签名、机制、角色、条件、边界、图关系和跨域检索。旧 atom 只有在函数签名、机制、边界、复用功能和依赖结构均兼容时才能复用。

零依赖、零名称重叠和领域不同都不能作为直接新建理由。

## D. 假迁移门

每个迁移必须逐项通过：

1. 新领域存在全部必要角色。
2. 成立条件真实满足。
3. 中介机制保持不变。
4. 能生成可观察解释或预测。
5. 不是只替换了名词或复用了结论句式。

机制推演可以写入 draft，但必须显式标注，且不算验证。

## E. 证据门

draft 必须诚实区分：来源支持、编译器推导、独立验证和待验证。学术线索可为空，不能伪造。

stable 需要全部满足：

- `source` 至少包含两个路径 identity 与内容哈希都不同、且本地可解析的来源文件，URL 不计硬门；
- 两个相互独立的来源支持同一机制，且至少一个实际领域绑定获得带证据定位的独立材料验证；或一份可核验的权威原理来源，且至少一份实际领域材料完成验证；
- `待验证：无`，机制身份、函数签名和边界不存在未解决冲突；
- 关键因果链没有未解决的反证或来源冲突；
- 未参与成稿的 subagent 已完成并记录独立语义复核的单一结论与具体范围；主 agent 自查不能标记“通过”，无可用独立审查者时只能保持 draft；
- 复用审查、假迁移审查和全部硬校验通过。

## F. 整批门

- 没有漏掉材料的核心机制。
- 没有把同一机制拆成同义文件。
- 没有把多个独立机制塞进一个 atom。
- 没有为了补图而创建词语级伪原子。
- 没有削弱已有依赖图或制造循环。
- 用户最终报告与实际写入、证据状态一致。

## 自动返工

任一门失败，AI 自动定位原因、修改并重新运行全部受影响门，最多两轮。两轮后仍失败时，恢复所有被改旧文件的原始字节并删除本轮新增，保留无关既有内容，并报告阻塞；不要求用户替编译器裁决候选。

## Validator 运行

Windows PowerShell 从当前 `SKILL.md` 的绝对目录定位并运行 wrapper，不写死 workspace 相对 Python 路径：

```powershell
& "<当前 skill 绝对目录>\scripts\validate_store.ps1" -Root "<workspace-root 绝对路径>"
```

其他环境也必须使用当前 skill 绝对目录中的 validator；runtime 由 AI 选择，不交给用户安装或决定。
