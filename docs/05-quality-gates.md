# 质量检查规则

这个文件用于检查一个候选内容能否进入 atom store。

## Gate 1：身份

- [ ] 有稳定 `id`。
- [ ] 有 `status`。
- [ ] ID 带版本后缀，如 `关系行为惯性-v1`。
- [ ] 文件名等于 ID。
- [ ] 文件名、ID、H1、字段标题跟随输入材料主语言。

不通过：不能进入 atom store。

## Gate 2：来源

- [ ] 有 `source` 指针。
- [ ] 能说明来源类型、标题或定位方式。
- [ ] 不要求保存全文，但未来能追溯。

不通过：补来源；无法补来源则不要标记为 stable。

## Gate 3：候选资格

- [ ] 它有独立复用价值。
- [ ] 不是文章标题。
- [ ] 不是小标题。
- [ ] 不是方法步骤。
- [ ] 不是案例、人物、金句、修辞。
- [ ] 不是只服务当前材料的局部表达。

不通过：并入已有 atom 的“必要说明”或丢弃。

## Gate 4：定义

- [ ] 能回答“它是什么”。
- [ ] 不循环定义。
- [ ] 不用案例代替定义。
- [ ] 不用方法代替定义。
- [ ] 不用口号代替定义。

不通过：不能 stable。

## Gate 5：函数签名

- [ ] 能写成“输入 -> 操作 -> 输出”。
- [ ] 输入明确。
- [ ] 操作明确。
- [ ] 输出明确。
- [ ] 这个函数未来能被其他 atom 或任务调用。

不通过：不能进入 atom store。

## Gate 6：不可拆分性

- [ ] 如果 `components: []`，再拆就得不到完整的“输入 -> 操作 -> 输出”。
- [ ] 如果还能拆出完整函数，必须拆成下层 atom，并把当前 atom 作为系统 atom。
- [ ] 如果当前 atom 是系统 atom，`components` 不为空。
- [ ] 系统 atom 的组件组合后产生单个组件没有的新能力。

不通过：继续拆分，或降级为上层系统 atom。

## Gate 7：底层追溯

- [ ] 当前 atom 能沿 `components` 和 `depends_on` 追溯到底层基础 atom。
- [ ] 依赖底层学科原理时，已链接到底层原理 atom。
- [ ] 正文“依赖”中使用 `[[底层原理atom-id]] - 依赖理由`。
- [ ] 如果底层原理 atom 原本不存在，已先创建或复用地基 atom。
- [ ] 地基 atom 不是学科标签，而是完整的“输入 -> 操作 -> 输出”知识函数。
- [ ] 没有把上层函数孤立保存成黑箱。

不通过：补底层 atom、补依赖链，或不要新建。

## Gate 8：边界

- [ ] 写清它不是什么。
- [ ] 写清容易和什么混淆。
- [ ] 写清什么情况下会被误用。
- [ ] 写清适用边界。
- [ ] 边界不是百科式扩展，只保留关键误区。

不通过：不能 stable。

## Gate 9：底层机制

- [ ] 写出因果链、结构约束或生成机制。
- [ ] 能说明它为什么成立。
- [ ] 能写出“触发条件 -> 中介机制 -> 可观察结果 -> 适用边界”。
- [ ] 不只写“这个方法有用”。
- [ ] 不把技巧命名、建议或标签当机制。

不通过：补机制、拆分、合并或丢弃。

## Gate 10：复用价值

- [ ] 能说明它未来可以支持哪些任务。
- [ ] 能说明它可能被哪些更大 atom 引用。
- [ ] 能写清调用方式：遇到什么问题时应该引用它。
- [ ] 能写清它能接入哪个更大系统或上层 atom；暂时没有时，写清上层系统候选。
- [ ] 它不是一次性回答。
- [ ] 它不是只服务当前文章。

不通过：不要新建 atom。

## Gate 11：区分说明

- [ ] 已检查是否存在可承载的旧 atom。
- [ ] 已按 `docs/08-reuse-search.md` 完成检索。
- [ ] 已比较本批候选之间是否重复。
- [ ] 如果旧 atom 能准确承载，复用旧 atom。
- [ ] 如果新建，能说明为什么旧 atom 或本批其他候选不够精准。

不通过：先查旧 atom，或先做本批去重。

## Gate 12：粒度

- [ ] 不混入多个独立机制。
- [ ] 不是一个无法独立复用的碎片。
- [ ] 再拆会失去独立用途。
- [ ] 不拆不会混入多个用途。

不通过：拆分、合并或丢弃。

## Gate 13：依赖

- [ ] 有 `depends_on` 字段。
- [ ] 依赖为空时明确写 `[]`。
- [ ] 依赖是真正前置条件，不是普通相关。
- [ ] 正文“依赖”与 frontmatter 一致。
- [ ] 正文“依赖”里的 atom 使用 `[[atom-id]]` 形成 Obsidian 反向链接。

不通过：重建依赖。

## Gate 14：组成

- [ ] 有 `components` 字段。
- [ ] 基础 atom 写 `[]`。
- [ ] 大 atom 写出组成它的下层 atom。
- [ ] 正文“组件”与 frontmatter 一致。
- [ ] 正文“组件”里的 atom 使用 `[[atom-id]]` 形成 Obsidian 反向链接。
- [ ] 不单独创建 system 文件类型。

不通过：补组件关系或降级为基础 atom。

## Gate 15：互动结构

- [ ] 如果材料涉及互动、关系、交易、竞争、合作、权力、信号或奖惩，已先抽互动结构。
- [ ] 已识别玩家、目标、信息、策略空间、规则点、奖惩、信号和可能结果中与材料相关的部分。
- [ ] 只把可复用规则点沉淀为 atom。
- [ ] 没有把玩家名单、单个案例、一次性策略或技巧步骤直接建成 atom。

不通过：回到互动结构扫描，重新提取规则点。

## Gate 16：可观察与迭代

- [ ] `source` 能回查。
- [ ] 未来发现问题时，知道应该修改、合并、废弃还是替换。
- [ ] 边界不清时保持 `draft`。
- [ ] `stable` 只表示结构和关系可复用，不表示内容绝对为真。

不通过：不能 stable。

## 状态规则

| status | 含义 |
|---|---|
| `draft` | 初次生成，尚未通过全部检查 |
| `stable` | 通过质量门，可以被其他 atom 引用 |
| `deprecated` | 不建议新引用，但保留追溯 |
| `merged` | 已合并到另一个 atom |

`rejected` 不是 atom 状态。候选内容如果不合格，就不进入 atom store；只在本次回复中说明处理结果，不创建一个 `status: rejected` 的 atom。

## 最终验收句

一个 atom 通过检查时，应该能说：

> 我有稳定身份，知道自己从哪里来，能写清输入、操作和输出；如果我是基础 atom，再拆就失去完整函数；如果我是系统 atom，我能沿 components 和 depends_on 追溯到底层基础 atom 或底层学科原理 atom；我也有清楚边界、底层机制、复用价值和调用方式。
