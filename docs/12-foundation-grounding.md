# 底层学科地基规则

## 核心结论

新 atom 不能凭空成立。

一篇文章里提取出的上层概念，只有在它能沿 `components` 和 `depends_on` 追溯到底层基础 atom 或底层学科原理 atom 时，才允许进入 atom store。

地基不是分类表。地基是一条可追溯的依赖链：

```text
上层知识函数 -> 中间组件/前置原理 -> 底层学科原理 atom -> 不可再拆的基础 atom
```

底层学科不写成 `type`、`level`、`discipline` 等 frontmatter 字段。它只在正文“必要说明”里说明来源，并在正文“依赖”里用 `[[atom-id]] - 依赖理由` 链接到具体原理 atom。

## 地基补齐算法

编译文章时，按这个顺序处理：

1. 先提取候选知识函数，写成 `输入 -> 操作 -> 输出`。
2. 对候选的每个核心机制追问：它为什么能成立？它依赖哪个更底层的原理？
3. 在已有 atom 中检索这些前置原理。
4. 如果已有 atom 能准确承载，写入当前 atom 的 `depends_on` 或 `components`。
5. 如果没有对应地基 atom，先创建最小地基 atom，再创建上层 atom。
6. 如果无法说清地基，当前候选不能作为独立 atom；最多并入其他 atom 的“必要说明”。

最终写入顺序应当是：

```text
底层原理 atom -> 中间组件 atom -> 上层系统 atom
```

这样新卡片不是漂浮的观点，而是被已有图谱托住的可复用函数。

## 追溯到多深

没有固定层数。

停止下钻的标准是：

- 当前 atom 已经是完整的 `输入 -> 操作 -> 输出`。
- 再拆下去会丢失输入、操作或输出中的至少一项。
- 它已经是地基地图中的稳定原理，未来可以被多个上层 atom 调用。
- 继续下钻只会进入当前任务不需要的物理、生物或神经细节。

不要为了显得“更底层”而一路拆到物理学或神经元。只拆到当前知识函数成立所必需、且未来可复用的地基原理。

## depends_on 与 components

`components` 表示当前 atom 是由哪些下层函数组成的。

`depends_on` 表示理解或使用当前 atom 必须先理解哪些前置原理。

判断方式：

- 没有它，当前函数就少一个结构组成部分：写 `components`。
- 没有它，当前函数仍是当前函数，但读者无法理解它为什么成立：写 `depends_on`。
- 底层学科原理通常写 `depends_on`。
- 一个系统 atom 的模块、步骤、子函数通常写 `components`。

不要把“相关概念”写进 `depends_on`。依赖必须是前置条件，不是联想。

## 缺少地基时怎么办

如果当前文章触发了一个上层 atom，而库中缺少它的前置原理：

1. 先创建或复用必要的地基 atom。
2. 地基 atom 的 `source` 写当前输入的来源指针，并在正文“必要说明”里标注“由当前材料触发的底层原理补齐”。
3. 地基 atom 仍然必须满足普通 atom 标准：定义、边界、函数签名、底层机制、复用价值、区分说明、依赖、组件。
4. 然后再创建上层 atom，让上层 atom 链到这些地基 atom。

`source` 不是“真理证明”，只是说明这张卡为什么在这次编译中被创建。这个系统不管理真伪字段，只保证以后可以观察、复用、链接、更新和替换。

## 跨学科同构

同一个底层机制可能同时出现在多个学科里。

例如“代价信号”可以出现在博弈论、进化心理学、传播和关系判断中。

处理方式：

- 如果底层函数一致，优先创建一个更通用的 atom。
- 不要为每个学科复制一张重复 atom。
- 在正文“必要说明”里写它常见于哪些学科。
- 让不同上层 atom 通过 `depends_on` 链到这个通用 atom。

这比按学科分类更符合复利：同一个底层函数被多个系统复用。

## 验收句

新建上层 atom 前，必须能说清：

> 这个 atom 是一个可复用知识函数；它的输入、操作、输出清楚；它依赖的前置原理已经通过 `depends_on` 或 `components` 链接；这些链条能继续追溯到底层学科原理 atom 或不可再拆的基础 atom；如果缺少地基，我已经先补齐了地基 atom。
