# 前置概念补齐 v0.4

## 核心结论

一个机制原子被上层原子引用时，它的每个参与角色和关键概念必须在库中有清晰的定义。如果某个概念使用者无法独立理解，就需要先补齐。

补齐的动机不是"找到学术归属"，而是：**没有这个前置定义，上层机制就不能被独立读懂。**

## 为什么不做底层学科追溯

对话记录里定义的机制原子验证，是横向的：

> 机制核清楚吗？能脱域吗？能在新领域生成有效判断吗？

不是纵向的：

> 它属于哪个学科？它的底层原理是什么？那个原理的更底层原理又是什么？

纵向追溯没有自然终点——每一个"为什么"都可以继续追问，最终掉进物理定律、数学公理、哲学本体论。这不是知识库要解决的问题。

**知识库要保证的是可复用、可迁移、可独立理解，而不是可学术论证。**

## 什么时候需要补齐

只在一个情况下需要：**上层机制原子引用了一个概念，而这个概念在库中没有定义。**

例如：

- 上层原子说"这遵循贝叶斯更新的逻辑"——但库中没有 `贝叶斯更新` 的定义 → 需要补齐
- 上层原子说"这体现了机会成本"——但库中没有 `机会成本` 的定义 → 需要补齐
- 上层原子说"变量A通过反馈回路影响变量B"——库中已有 `反馈回路` 的定义 → 不需要补齐

判断句：

> 把这个机制原子给一个没读过原文的人看，他需要额外查什么才能理解？

需要额外查的东西如果在库中没有，就补齐。

## 补齐到什么程度

补齐到**能独立理解当前上层原子**即可，不需要继续下钻。

停止标准：

- 补齐的概念已经能写清"它是什么"（定义 + 边界 + 一句话机制）
- 它能被其他上层原子复用
- 再往下拆不会增加当前上层原子的可理解性

例如：上层原子引用了 `机会成本`，补齐 `机会成本` 的定义即可。不要继续追问"机会成本依赖什么微观经济学原理"、"微观经济学依赖什么数学基础"——除非另一个上层原子真的需要这些东西才能被理解。

## 两类需要补齐的概念

### 来自公认概念的概念

这些概念有相对稳定的通行定义：`机会成本`、`贝叶斯更新`、`反馈回路`、`瓶颈`、`代价信号`、`损失厌恶`、`路径依赖`

补齐时写清它在当前系统中的含义和边界即可。不需要标注"来自XX学科"——学科标签不能替代机制理解。

### 来自跨领域经验的机制构件

这些是从多领域经验中反复出现的结构，不属于任何单一学科：

| 机制构件 | 一句话 | 常见出现在 |
| --- | --- | --- |
| 反馈回路 | 输出反过来影响输入，形成增强或平衡循环 | 几乎任何系统 |
| 不足/过度/平衡 | 变量偏离最优区间两边都会出问题 | 调参、设计、管理、关系 |
| 瓶颈 | 系统中最慢的环节限制整体吞吐量 | 生产、软件、组织、交通 |
| 阈值 | 变量越过某个临界点后行为发生质变 | 物理、生理、心理、社会 |
| 敏感度 | 不同变量对结果的单位影响不同 | 工程、经济、产品、关系 |
| 固定成本摊薄 | 共享不可分成本导致平均成本随规模下降 | 生产、SaaS、同居、公共交通 |
| 到达-处理失衡 | 输入速度长期大于处理速度导致积压增长 | 餐厅、服务器、医院、客服 |
| 介质成分转移 | 介质接触材料导致成分从材料进入介质 | 咖啡、泡茶、煲汤、中药、香水 |

这些机制构件本身就可以作为基础原子。它们不需要再往下追溯到"哪个学科"——它们已经是最小可复用机制单元了。

## 补齐流程

编译时按这个顺序处理：

1. 提取上层机制原子的四要素。
2. 列出它引用的所有概念和参与角色。
3. 检查每个概念在已有原子库中是否有可独立理解的定义。
4. 有 → 写入 `depends_on`，链接到已有原子。
5. 没有 → 创建最小定义原子（定义+边界+一句话机制，`components: []`），再让上层原子链接它。
6. 如果某个概念无法被明确定义，说明当前上层原子可能依赖了模糊概念——先澄清它，再决定是否新建。

## 创建概念定义原子

概念定义原子是最简单的基础原子。它只需要：

```markdown
# 概念名称

## 定义
一句话说清它是什么。

## 边界
它不是什么，容易和什么混淆。

## 一句话机制
它背后的因果结构或约束（如果有的话）。

## 必要说明
常见别名、来源。如果由当前材料触发补齐，标注"由当前材料触发的概念补齐"。
```

不需要写参与角色、成立条件、迁移边界——那些是机制原子的要求。概念定义原子的作用是**被引用、被理解**。

## 和机制原子的关系

- 概念定义原子通常出现在 `depends_on` 中（理解上层机制需要先理解这个概念）。
- 机制构件原子（如 `反馈回路`、`瓶颈`）本身已经是机制原子——它们携带完整的四要素，可以在多领域复用。
- 一个原子是"概念定义"还是"机制构件"，不由它的来源决定，而由它的内容决定——能写出完整四要素的就是机制原子，只能写出定义的就是概念定义原子。

## 不要做的事

- 不要按学科给概念分类。
- 不要追溯概念到更底层的概念（除非不追溯就无法理解当前上层原子）。
- 不要预先补齐一整套"学科基础知识库"——只补齐当前材料真正需要的概念。
- 不要因为"这个概念属于XX学科"就创建它——只有在上层原子真的引用它时才需要。

## 验收句

新建上层机制原子前，必须能说清：

> 这个原子引用的每个概念和参与角色，在库中都有清晰的定义——要么已有原子承载，要么我已经补齐了它的定义。补齐到能让一个没读过原文的人独立理解这个机制即可，不继续下钻。
