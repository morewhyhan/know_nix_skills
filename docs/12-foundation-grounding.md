# 前置概念补齐 v0.4.1

## 核心结论

机制原子要能跨领域复用，它依赖的概念本身也得是可复用的。如果一个上层机制引用了只在原领域里有含义的概念，它就不可能脱域。

所以要补齐前置概念。补齐的目标是：**上层机制引用的每个概念，都有独立于任何单一领域的清晰定义。**

## 为什么从学科里找

不是"学科权威所以正确"，而是：**学科里那些被反复验证过的中层概念（机会成本、反馈回路、贝叶斯更新、瓶颈、代价信号……），本身就经过了跨领域抽象。它们是从无数具体案例中提炼出来的——这恰好就是"脱域"。**

学科概念的可复用性不是来自它们的学术血统，而是来自它们已经被证明了可以在截然不同的系统中成立：

- `机会成本`：选择A的代价是放弃B——适用于生产决策、职业选择、时间分配、关系取舍
- `反馈回路`：输出反过来影响输入——适用于工程控制、人际关系、组织管理、气候系统
- `瓶颈`：最慢的环节拖慢全局——适用于工厂产线、软件架构、交通路网、行政审批
- `代价信号`：成本越高信号越可信——适用于求偶展示、品牌定价、学历筛选、承诺可信度

这些概念天生脱域。一个知识库如果能积累一批这样的概念定义，上层机制原子的复用能力会强得多——因为它依赖的东西本身就是可迁移的。

## 但不要无限追溯

学科概念也有层级。`机会成本` 可以继续拆成 `稀缺性 + 选择 + 权衡`，`稀缺性` 可以被继续追问"为什么资源稀缺"……一路拆到物理学和哲学。

知识库不需要做这件事。停止标准：

- 当前概念已经是一个**完整的、能被独立使用的脱域概念**——定义清晰、边界清楚
- 再往下拆，拆出来的东西不能独立使用，只能作为这个词的释义零件
- 它已经可以作为多个上层机制原子的 `depends_on` 目标

例如：`机会成本` 停在这里就够了。不需要拆成 `稀缺性`、`权衡`、`替代选项`——那些是解释"机会成本"时可以提及的东西，但它们单独拿出来不能被上层机制直接复用。

### 跨领域机制构件

这些不属于单一学科方向，但在大量领域中反复出现。它们本身已经是完整的机制原子，可以直接作为上层原子的前置依赖：

| 机制构件 | 本质 |
| --- | --- |
| 不足 / 过度 / 平衡 | 变量偏离最优区间两边都会出问题 |
| 到达-处理失衡 | 输入速率 > 处理速率 → 积压增长 → 延迟上升 |
| 固定成本摊薄 | 不可分成本被更多使用者分摊，受容量和协调成本限制 |
| 介质成分转移 | 介质接触材料导致选择性迁移，受时间/温度/表面积/介质性质影响 |
| 敏感度差异 | 不同变量对同一输出的单位影响力不同 |
| 反馈-调整-再输出 | 结果反馈回来后调整输入，形成迭代闭环 |
| 稀缺-竞争-分配 | 有限资源引发竞争，分配机制决定结果 |
| 局部最优 vs 全局最优 | 短视优化可能导致系统整体更差 |

这些机制构件和学科概念没有本质区别——都是脱域的。区别只在于：学科概念有约定俗成的名字（"机会成本"），这些机制构件可能需要自己命名。

每个概念可以写成机制原子（如果能写全四要素）或概念定义原子（如果只需要定义+边界+一句话机制）。不管哪种形式，它都必须是可独立理解的——正如原子是化学里不可再分却保留化学性质的最小单元，概念原子也是"再拆就失去独立复用价值"的最小可操作定义。

### 从多领域经验中涌现的机制构件

这些不属于单一学科，但在大量领域中反复出现：

| 机制构件 | 本质 |
| --- | --- |
| 不足 / 过度 / 平衡 | 任何变量在两个方向偏离最优区间都会出问题 |
| 到达-处理失衡 | 输入速率超过处理速率导致积压增长 |
| 固定成本摊薄 | 不可分成本被更多使用者分摊，受容量和协调成本限制 |
| 介质成分转移 | 介质接触材料导致选择性成分迁移，受时间/温度/表面积影响 |
| 敏感度差异 | 不同变量对同一结果的单位影响力不同 |

它们和学科概念没有本质区别——都是脱域的。区别只在于：学科概念有名字（"机会成本"），这些机制构件可能需要自己命名。

## 补齐到什么程度

补齐到概念能**被独立理解、被其他原子引用**即可。不需要解释它为什么成立——那是它自己的底层机制，属于另一个问题。

例如：
- 补齐 `机会成本` → 写清"它是什么：被放弃的最高价值替代选项"即可
- 不需要在 `机会成本` 里解释为什么资源稀缺、为什么选择必然有代价
- 除非上层原子真的需要理解"稀缺性"才能成立——那就再补齐 `稀缺性`

链条的终点是：**这个概念已经短到可以直接被理解，再拆不会增加任何上层原子的可理解性。**

## 补齐流程

1. 提取上层机制原子，列出它引用的所有概念和参与角色。
2. 对每个概念，在已有原子中检索是否有可独立理解的定义。
3. 有 → 写入 `depends_on`，链接过去。
4. 没有 → 判断：这个概念是领域专属的（无法脱域）还是可以脱域的？
   - 领域专属 → 说明当前机制原子可能无法完全脱域，或者这个角色需要被进一步抽象
   - 可以脱域 → 创建概念原子（定义+边界+一句话机制），让上层原子链接它
5. 不继续追问"这个概念又依赖什么"——除非另一个上层原子确实需要。

## 概念原子的写法

比机制原子简单。它只需要：

```markdown
# 概念名称

## 定义
一句话说清它是什么。

## 边界
它不是什么，容易和什么混淆。

## 一句话机制
它背后的因果结构（如果有的话）。不是所有概念都有机制——"稀缺性"是一个前提条件而不是机制。

## 必要说明
常见别名、为什么由当前材料触发补齐。
```

不需要参与角色、成立条件、迁移边界——这些是机制原子的要求。概念原子的角色是被引用，本身不一定要是一个运作中的机制。

## 不要做的事

- 不要追溯概念到更底层的概念（除非不追溯上层原子就无法理解）。
- 不要预先建一整套"学科知识库"——只补齐当前材料触发的。
- 不要因为"这个概念属于XX学科所以必须建"——建的动力是上层原子引用了它。
- 不要给概念标注学科分类字段。

## 验收句

新建上层机制原子前：

> 它引用的每个概念在库中都有清晰定义。如果某个概念来自学科（机会成本、反馈回路、贝叶斯更新……），它本身就是脱域的——这正是它能被多个上层原子复用的原因。如果某个概念是领域专属的，它不能被上层机制原子直接引用——先把它抽象到可脱域的层级。补齐到每个概念都能被独立理解为止，不继续下钻。
