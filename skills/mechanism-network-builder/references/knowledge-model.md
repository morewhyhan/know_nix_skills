# 知识模型

## 对象

- **来源还原稿**：保存材料原意、用户解释、不能推出的内容和证据风险。
- **机制卡**：保存离开原文仍能调用的最小因果或约束结构。
- **系统卡**：保存多个机制通过接口组合后产生的新增能力。
- **专题分析**：使用知识网络处理现实问题。
- **运行报告**：记录一次写库的决策、变更与验收。

## 机制卡资格

必须说明输入、作用、中介过程、可观察结果、参与角色、成立条件、失效边界和新场景测试。以下内容不单独建卡：标题、人物、案例、金句、普通概念、一次性建议、无结果的观点、只替换领域词的类比。

## 系统卡资格

多个组件承担不同职责，通过明确接口解决单卡无法解决的矛盾，并产生新增解释、判断、预测或设计能力。移除被标记为必要的组件后，如果能力不变，该组件不是必要组件。

## 元数据

```yaml
---
type: mechanism        # mechanism | system
status: draft           # draft | usable | stable | retired
aliases: []
created: YYYY-MM-DD
updated: YYYY-MM-DD
---
```

文件名与 H1 完全一致。来源和关系写在正文。retired 卡必须说明去向和原因。

## 证据角色

| 角色 | 含义 |
|---|---|
| 原文直接表达 | 来源明确陈述或展示 |
| 用户解释 | 用户的理解、假设或延伸 |
| 系统推导 | AI 抽象出的结构或因果链 |
| 独立材料支持 | 不同来源支持相同关键结构 |
| 跨域推演 | 角色映射，尚未独立验证 |
| 组合设计 | 多机制组合产生的新系统 |
| 反例限定 | 收窄边界或否定一般化 |

## 正式关系

| 类型 | 含义 |
|---|---|
| 组成 | A 在指定系统实现中承担组件职责 |
| 前置 | 调用 B 前必须具备 A |
| 中介 | A 承担 B 因果链中的传递环节 |
| 约束 | A 限制 B 的范围、风险或解释 |
| 对照 | A 与 B 回答相同问题但策略不同 |
| 案例验证 | 某领域材料实际承载该机制 |
| 反例限定 | A 说明 B 在某范围内失效 |
| 相关案例 | A 只承载 B 的部分结构 |
| 组合 | A 与其他机制共同产生新系统 |

候选关系不得包含 wikilink，防止 Obsidian 把未经核验的相似性当正式关系。

每条正式关系只能指向一张卡，并写成：

```text
`关系类型` → [[目标卡]]：理由：为什么必须连接；边界：这条连接不代表什么；证据：证据角色。
```

## 复用顺序

1. 完全承载：复用并补来源。
2. 同一机制但旧边界过窄：更新旧卡。
3. 旧卡混入多个函数：拆分。
4. 名称相同但机制不同：分开并加白话限定。
5. 近义重复：合并，旧卡转 retired。
6. 无法承载且有独立复用价值：新建 draft。

比较时必须检查函数、过程、角色、条件、边界和调用方式，不能只比标题。
