# 机制网络构建器

把材料转成可读、可追溯、可迁移、可组合的机制知识网络。优先保证来源忠实和知识可用，不追求卡片数量、链接数量或新概念数量。

## 选择操作

- **沉淀材料**：用户要求整理、沉淀、写入或维护知识库。读取全部规则并写库。
- **调用知识**：用户要求用已有机制回答问题或生成内容。检索并回答，默认不写库。
- **组合探索**：用户要求让机制碰撞、组合或产生新系统。执行组合协议；通过组合门才写系统卡。
- **审计维护**：用户要求检查卡片、证据、命名、重复或关系。修复授权范围内的问题并报告。

意图不明确时先只读调用；不要擅自大规模改库。

## 开始前

1. 定位 workspace 根目录与 `.mechanism-network.yml`。没有配置时，只有在用户明确要求写库后才复制 [assets/mechanism-network.yml](assets/mechanism-network.yml) 为根目录配置，并创建其中的默认目录。
2. 沉淀、组合或审计前，完整读取：
   - [references/workflow.md](references/workflow.md)
   - [references/knowledge-model.md](references/knowledge-model.md)
   - [references/quality-gates.md](references/quality-gates.md)
3. 调用知识时至少读取知识模型，并扫描全部 `knowledge/mechanisms/` 与 `knowledge/systems/`；不能只搜标题。
4. 写入前保存本批将修改文件的原始内容；失败时恢复，不留下半批状态。

默认知识库目录：

```text
knowledge/
  reconstructions/
  mechanisms/
  systems/
  analyses/
  runs/
```

## 核心工作流

### 1. 分离输入角色

明确区分：原始材料、用户解释、用户任务、外部材料。不得把用户延伸写成作者原意，也不得把现代概念倒灌给古代材料。

### 2. 先做来源还原

使用 [assets/source-reconstruction.md](assets/source-reconstruction.md)。保留核心结论、论证顺序、作者判断、用户增加内容、不能推出的内容和证据风险。不能建卡的内容仍留在还原稿。

### 3. 全库复用检索

对每个候选同时检索：标题与别名、输入与输出、中介过程、角色、条件、边界、上游下游和系统组件。零标题重叠不等于没有旧卡可复用。

### 4. 只提取完整机制

有效机制必须能写成：

```text
输入或触发 → 作用 → 中介过程 → 可观察结果
```

并说明成立条件、失效边界和新场景迁移。只有名词、观点、建议、案例、金句或换领域词的类比不建卡。

### 5. 让标题先被看懂

命名优先级：精确的已有术语 → 已有术语加白话限定 → 一眼能懂的白话标题 → 必要的新造名称。标题必须通过三秒测试；使用大词时必须覆盖完整机制。

### 6. 分层证据

正文明确标记：原文直接表达、用户解释、系统推导、独立材料支持、跨域推演、组合设计、反例限定。引用存在不等于机制已经验证。

### 7. 建立有用关系

正式关系只允许：`组成`、`前置`、`中介`、`约束`、`对照`、`案例验证`、`反例限定`、`相关案例`、`组合`。

每条写成：

```text
`关系类型` → [[目标卡]]：理由：为什么必须连接；边界：这条连接不代表什么；证据：证据角色。
```

每条正式关系只能指向一张目标卡，便于逐边审核充分性与必要性。“有点相关”只进候选关联，不建立 wikilink。正式关系必须检查目标卡中的反向记录。

### 8. 扫描跨系统组合

每次沉淀都检查组合机会，但不强制产出。为候选系统分别写清问题、输入、过程、输出和盲点；寻找不同机制家族之间的接口。

只有同时满足以下条件才建系统卡：

- 每个组件职责不同且必要；
- 说明一个机制怎样暴露盲点、解释原因、提供约束或提供操作；
- 组合解决单个机制无法解决的新问题；
- 产生新的解释、判断、预测或设计能力；
- 有具体应用和可推翻条件。

使用 [assets/system-card.md](assets/system-card.md)。只并列概念或制造新名字不算组合。

### 9. 事务写入并验证

在内存中完成整批卡片、关系和状态。先把完整批次按 `knowledge/` 相对路径写进 workspace 内的临时 staging 目录，再用事务脚本提交；脚本会在校验失败或中途异常时恢复全部旧文件。卡片使用 [assets/mechanism-card.md](assets/mechanism-card.md)，运行报告使用 [assets/run-report.md](assets/run-report.md)。

Windows：

```powershell
& <skill-dir>/scripts/commit_batch.ps1 -Root <workspace-root> -Staging <staging-dir>
```

其他环境：

```bash
python <skill-dir>/scripts/commit_batch.py --root <workspace-root> --staging <staging-dir>
```

只做审计、不写入时，直接运行 `validate_store.ps1` 或 `validate_store.py`。机械校验通过后，再执行来源忠实、假迁移、命名和组合新增能力的语义检查。语义检查失败时按运行报告列出的本批文件恢复；失败最多修复两轮。

## 生命周期

- `draft`：新推导、单一来源、组合设计或仍有证据缺口。
- `usable`：函数、边界、命名和迁移通过，关键结构获得独立支持。
- `stable`：多源或权威原理加应用支持，反例与身份冲突已处理。
- `retired`：被合并、边界错误、名称错误或被证据推翻；保留去向和原因。

不要因语言流畅或链接很多自动升级状态。

## 用户可见输出

只报告：读取了什么；生成或更新了哪些还原稿；复用、新建、更新、合并、退休或拒绝了哪些卡；关键关系如何改变；组合产生了什么或为何拒绝；验证是否通过；仍有哪些证据缺口。

不要要求用户筛候选、补字段、选关系或维护状态。用户负责最终价值判断，Skill 负责把知识结构维护完整。

## 安全边界

- 只修改当前 workspace 的配置目录和知识库目录。
- 不修改无法解析的来源文件。
- 不把法律、医疗、金融、历史或科学断言仅凭单一评论性来源写成事实。
- 不静默覆盖、合并或退休已有卡；运行报告必须说明理由。
- 不为了让校验通过而制造关系、提高证据强度或删除真实矛盾。
