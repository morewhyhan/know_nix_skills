# Mechanism Atom Compiler

把输入材料编译成可跨领域复用的机制原子。不要做摘要。不要做文章笔记。只提取能脱域、能迁移、能在共享机制的其他领域重新绑定并生成新判断的机制原子。

## 模式

行动前先选一个模式：

- `compile`：把文章、对话、摘录、想法编译成 draft 原子。
- `de-domain`：检查已有领域节点是否可脱域为机制原子。不写入。
- `reuse`：只检索已有原子，解释哪些能复用，不写入。
- `validate`：检查原子或 store，不新建原子。
- `query`：用已有原子辅助当前思考，不写入。
- `refactor`：合并、拆分、升级、废弃旧原子。

意图不明确时，不写入；先用 `reuse` 或 `query`。

## 工作区

只在当前 workspace root 内工作。核心目录只有：

```text
atoms/
```

不要创建 `sources/`、`systems/`、`relations/`、`views/` 等目录。

## 语言规则

原子的可读部分必须跟随输入材料的主语言。中文输入 → 中文文件名、ID、H1、正文标题、正文。不把中文概念翻译成英文 slug。文件名必须等于 `id + ".md"`。

中文原子正文标题固定使用以下顺序，不可改变：

```text
场景用途 → CLC方向 → 定义 → 边界 → 机制核心 → 参与角色 → 成立条件 → 迁移边界 → 复用价值 → 领域绑定 → 脱域说明 → 区分说明 → 必要说明 → 依赖 → 组件
```

**注意：** 机制核心、参与角色、成立条件、迁移边界 这四个 section 仅供机制原子使用。概念原子（L0-L2）和领域节点跳过这四个 section，不要写"（概念原子，不适用）"等占位文字。其他 section 所有原子类型都必须写。

**领域绑定 和 脱域说明 是两个独立 section，不合并。**

## 最小 Frontmatter

```yaml
id:
status:
source:
tags:
aliases:
depends_on:
components:
```

`source` 写入来源材料的路径或引用标识，如 `01-文章/爱戏剧的柏拉图/001_看完你将认清婚姻4大真相.md`。
`tags` 用于 Obsidian 标签面板筛选。每个原子固定包含以下标签，缺一不可：
  - `clc/码`：CLC 分类码，如 `clc/F0`。跨学科方向（如社会心理学）写 `clc/社会心理学`
  - 方向名：与 CLC方向 正文中写明的方向名一致，如 `经济学`、`社会心理学`
  - `type/原子类型`：`type/机制原子`、`type/概念原子`、`type/领域节点` 三选一
  效果：Obsidian 标签面板中以层级树展示。点击 `clc/F0` 可筛选全部经济学原子，点击 `type/机制原子` 可筛选全部机制原子。三层标签可叠加筛选。

`aliases` 取自必要说明中的"常见别名"段落，与正文同步更新。每个原子至少列出 2 个别名。
  效果：Obsidian 搜索框直接输入别名可定位到该原子。撰写 wikilink 时输入 `[[别名]]`，Obsidian 自动补全为 `[[id|别名]]` 并正确链接。别名作为 H1 之外的第二套索引入口，覆盖同一概念的不同称呼方式（如"张力制造"→"克制生成吸引"）。

不要写能从图里推出来的字段：`used_by`、`reuse_count`、`part_of`、`closure`、`type`、`related` 等。

## 核心本体

**机制原子**是携带完整机制的最小可复用单元。它不是词语，不是句子，不是笔记卡片。

每个机制原子必须能说清四件事：

1. **机制核**：到底发生了什么（因果链或结构约束）
2. **参与角色**：哪些东西参与了这个机制，各自起什么作用
3. **成立条件**：机制在什么条件下运作
4. **迁移边界**：什么情况下不适用，不能迁移到哪些领域，为什么

判断一个东西是不是机制原子，问四句话：
- 它有没有机制？（不只是标签）
- 它能不能解释为什么？
- 它能不能迁移到共享同一机制的其他领域？
- 它有没有边界，防止瞎迁移？

### 基础原子、机制原子、复合原子

- **基础原子**：不可再拆的概念、属性、操作或关系。`components: []`。例如：`反馈`、`平衡`、`阈值`
- **机制原子**：携带完整四要素的最小可复用单元。核心资产。
- **复合原子**：由多个原子组合而成。`components` 不为空。可折叠/展开。

### 领域节点 vs 脱域节点

- **领域节点**：绑定在原领域，依赖领域专属概念。如 `研磨度`、`粉水比`
- **脱域节点**：已抽离领域对象，可在多领域复用的机制原子

## 编译流程

## Compile 执行清单

按以下顺序执行，每一步完成后再进入下一步。完成后在回复中报告每步结果。

### 1. 通读全文
完整阅读材料一遍。目的：了解文章讨论的领域、核心观点和整体结构。不做提取，不做判断。

### 2. 逐段逐句扫描
从文章第一行开始，按顺序逐段逐句向下推进。每读完一个语义单元（一个自然段或一句完整论述），停下来回答：

- 这句话/这一段是否包含一个可脱域的机制？
- 如果有 —— 记录候选。写出：原文哪句话触发、初步判断为什么是一个机制、暂定名称。
- 如果没有 —— 简述原因（如：过渡句 / 纯例子 / 重复前文 / 论证材料无独立机制 / 领域专属不可脱域）。

扫描完最后一个段落之前，不进入步骤3。确保每一段都被处理过，不跳读、不遗漏。

### 3. 分离领域与机制
对步骤2产出的每个候选，判断：去掉领域专属词→替换为抽象角色→机制还完整吗？
完整→进入步骤4。不完整→标记为领域节点（不脱域），写入 atoms/。

### 4. 提取机制四要素
机制核（因果链）、参与角色（列表+功能）、成立条件（何时运作）、迁移边界（何时不适用、不能迁移到哪、为什么）。

### 5. 判断层级
按 L0-L5 分层标准判断每个候选原子的层级。检查：components 是否都低于当前层？depends_on 目标是否不高于当前层？跨层依赖→禁止。

### 6. 分类
按 `docs/12-foundation-grounding.md` 分类框架判断每个概念/机制的 CLC 方向。**看场景用途，不看表面词汇。** 遇到易混淆边界参照分类决策规则。

### 7. 检索已有原子

**depends_on 就是索引。** 新候选的 depends_on 列了哪些概念，就对每个概念 grep 全库：`grep -rl "\[\[概念名\]\]" atoms/`。不需要读正文，毫秒级。

**排序：** 筛掉通用地基概念（反馈回路、系统、涌现……全库大量引用的那些）上的重叠。按专有概念（被引用数极少的那些）的重叠数排序。重叠数相同的，层级低的排前面。

**读取：** 从排名最高的开始，只读正文"机制核心"段落。因果链一样→复用。不一样→读下一个。读到能确认"没有已有原子能承载"为止。不对读取数量设上限——AI 自己判断什么时候停。

**零重叠：** 所有 depends_on 概念在全库都没有被引用过→不存在复用候选，直接新建。

### 8. 复用/新建/升级/拆分/合并
按 `docs/06-build-decision-rules.md` 判断。旧原子能承载→复用。机制一致但边界窄→升级。混入多机制→拆分。重复→合并。没有独立复用价值→丢弃（不建 rejected）。

### 9. 补齐前置概念
上层原子引用的概念在库中没有定义？→先建概念原子（场景用途+CLC方向+定义+边界+一句话机制），再建上层原子。

### 10. 写入原子
默认 `draft`。正文必须包含以下章节，按原子类型区分：

**所有原子类型都必须写：**
- **场景用途**：它在什么场景中使用、回答什么问题
- **CLC方向**：按步骤6的判断结果写明 CLC 码和方向名（如"社会心理学""F0 经济学"）。领域节点写"领域节点（不脱域）"。格式要求：CLC码+方向名+一句话子领域锚定（如"社会心理学 — 人际吸引与信号传递"），不复述机制核心内容
- **定义**（维特根斯坦式：场景锚定。结构：一句定义正文 → "与XX的区别"列举 → "排除："收尾）
- **边界**：它不是什么，容易和什么混淆
- **复用价值**：两段式。第一段说机制价值，第二段以"调用方式："开头给出 checklist
- **领域绑定**：表格列出跨领域角色映射。如果暂时只有一个领域，写清"当前仅在X领域中验证"
- **脱域说明**：原领域是什么、提取中去掉了什么、保留了哪些跨领域机制、为什么可以迁移。领域节点写"当前未脱域"
- **区分说明**：先声明"atoms/ 中暂无旧原子可复用"或列出检查过的旧原子，再用 wiki link + 语义关系类型列出与已有原子的关系
- **必要说明**：必须包含"学科来源"（学术脉络+引用）和"常见别名"（至少2个）。按需补充：必要例子、限制条件、本批新建术语声明
- **依赖**：正文中必须用 `[[atom-id]] - 语义关系类型：理由` 列出所有 depends_on 目标。即使为空写"无"
- **组件**：正文中必须用 `[[atom-id]] - 语义关系类型：理由` 列出所有 components 目标。即使为空写"无"

**仅机制原子额外必写（概念原子和领域节点跳过以下四个 section）：**
- **机制核心**：包裹在 ` ```text ``` ` 代码块中。纵向叙事流——`→` 箭头串联因果，缩进表示层级，因果关系通过缩进和箭头自然展开。多信号用"信号A/B"分列，合成公式用引号包裹，并行效应编号列出。结构形态按机制实际需要自由选择，详见 `templates/knowledge-atom.md`
- **参与角色**：表格格式。列出所有角色及其功能
- **成立条件**：编号列表。机制在什么条件下运作
- **迁移边界**：**必须包含三层，缺一不可：** (1) **可迁移到**（正面迁移示范，至少2个领域）；(2) **不能迁移到**（负面边界）；(3) **假迁移警惕**（具体错误示例+判断标准）

正文依赖/组件必须与 frontmatter 一致。Obsidian 通过正文 wikilink 产生反向链接。

### 11. 命名查重
检查本批新建原子的名称是否与已有原子重复或高度相似。已有概念→复用旧原子。本批内同义不同名→合并为一个。自创术语（原文没有、通用词典没有）→在必要说明中标注"本批新建术语"并给出定义依据。

### 12. 建立本批原子间关系
所有本批原子写入后，两两检查语义关系。对称机制必须建立 contrasts_with。同批领域节点与脱域机制间必须建立 instantiates。

### 13. 检查跨领域绑定
新脱域机制原子→已有原子库中是否存在可建立 instantiates 关系的领域节点？有→补入领域绑定表。

### 14. 质量检查
单原子质量门（22道）+ 硬校验 + 整批验收。特别检查：(1) 每个原子正文是否有 CLC方向 章节；(2) 正文 依赖/组件 章节是否与 frontmatter 一致且有 wikilink；(3) 是否有无依据的自创术语；(4) 机制原子的迁移边界是否包含三层（可迁移到/不能迁移到/假迁移警惕）；(5) 复用价值是否包含"调用方式："checklist；(6) 必要说明是否包含学科来源和常见别名。

### 15. 涌现验证
这批原子组合→能否生成原文没有的新判断？能→写涌现路径。连续零涌现→检查是否只在做摘要。

### 16. 结果报告
复用了哪些、新建了哪些（区分机制原子/领域节点/概念原子，标注层级 L0-L5 和 CLC 方向）、哪些脱域了、补齐了哪些前置概念、升级/拆分/合并/丢弃了哪些及理由、涌现路径。

## 分层构建

原子不按主题分类，按**构造层级**分层。上层只能引用下层，不能跨层。

```text
L0 基础概念    components: []  不可再拆的概念/属性/操作
L1 基础关系    components→L0   最简单的三元组（A→关系→B）
L2 组合概念    components→L0/L1 由基础概念和关系组合
L3 机制原子    components→L0/L1/L2 携带完整四要素
L4 命题        depends_on→L2/L3 一个完整判断
L5 复合系统    components→L3/L4 多机制多命题的系统
```

创建每个原子时必须判断层级。检查：components 是否都处于严格低于当前层？depends_on 目标是否不高于当前层？

## 语义关系

`depends_on` 和 `components` 只说"有关系"，不说"什么关系"。语义类型标注在 wikilink 中，不新增字段：

```markdown
- [[atom-id]] - 关系类型：理由
```

结构语义（components 中）：`composed_of` | `implements` | `instantiates`
认知语义（depends_on 中）：`prerequisite` | `defined_by` | `grounded_in`
逻辑语义（任意）：`supports` | `constrains` | `explains` | `contrasts_with`

选择规则：先判断是 components 还是 depends_on，再选语义子类型。

## 本批原子关系

所有本批原子写入后，两两检查是否存在 depends_on / components / supports / constrains / explains / contrasts_with / instantiates。同批创建的领域节点与脱域机制之间必须建立 instantiates。对称机制（"不足→X"和"过度→Y"）必须建立 contrasts_with。

## 涌现验证

编译结束后，检查这批原子能否组合出原文没有的新判断。能 → 在结果说明中写出涌现路径 `[atom_X] + [atom_Y] → "新判断"`。连续多次零涌现 → 可能只在做摘要。

## 跨领域迁移验证

判断一个机制原子能否迁移到新领域：

1. 新领域存在该机制需要的所有参与角色吗？
2. 新领域满足该机制的成立条件吗？
3. 迁移后能在新领域生成有效的解释或预测吗？
4. 这不是只在句式上相同吧？

四个都是"是"→ 可以迁移。任何一个"否"→ 不能迁移。

**真迁移**（共享机制）：咖啡萃取 → 泡茶/煲汤/中药煎煮（共享：介质接触材料→成分转移）

**假迁移**（换词造句）：咖啡萃取 → 学习掌握（机制完全不同）

## 互动结构扫描

当材料涉及多人互动、关系、交易、竞争、合作、权力、信号或奖惩时，先找：玩家、目标、信息、策略空间、规则点、奖惩、信号、可能结果。只有规则点能通过机制精度门时，才沉淀为原子。

## 正文要求

每个原子按类型写对应的 section（详见步骤9）。机制原子写全所有15个 section；概念原子和领域节点跳过机制核心/参与角色/成立条件/迁移边界。

Section 标题名称和顺序固定不可变：场景用途 / CLC方向 / 定义 / 边界 / 机制核心 / 参与角色 / 成立条件 / 迁移边界 / 复用价值 / 领域绑定 / 脱域说明 / 区分说明 / 必要说明 / 依赖 / 组件。**领域绑定和脱域说明为两个独立 section，不合并。**

## 状态

只使用：`draft`、`stable`、`deprecated`、`merged`。不要使用 `rejected`。

## 安全

新原子默认 `draft`。只有通过质量门、硬校验和整批验收后，才能 `stable`。

写入或重构后，在最终回复里说明：复用了哪些、新建了哪些（区分机制原子与领域节点）、哪些脱域了、补齐了哪些地基、升级/拆分/合并了哪些、丢弃了哪些。
