# Mechanism Atom Compiler

把输入材料编译成可跨领域复用的机制原子。不要做摘要。不要做文章笔记。只提取能脱域、能迁移、能在共享机制的其他领域重新绑定并生成新判断的机制原子。

## 模式

行动前先选一个模式：

- `compile`：把文章、对话、摘录、想法编译成 draft 原子。
- `de-domain`：检查已有领域节点是否可脱域为机制原子。不写入。
- `reuse`：只检索已有原子，解释哪些能复用，不写入。
- `validate`：检查原子或 store，不新建原子。
- `query`：用已有原子辅助当前思考，不写入。
- `refactor`：合并、拆分、升级、废弃旧原子。

意图不明确时，不写入；先用 `reuse` 或 `query`。

## 工作区

只在当前 workspace root 内工作。核心目录只有：

```text
atoms/
```

不要创建 `sources/`、`systems/`、`relations/`、`views/` 等目录。

## 语言规则

原子的可读部分必须跟随输入材料的主语言。中文输入 → 中文文件名、ID、H1、正文标题、正文。不把中文概念翻译成英文 slug。文件名必须等于 `id + ".md"`。

中文原子正文标题固定使用：

- 定义 / 边界 / 机制核心 / 参与角色 / 成立条件 / 迁移边界
- 复用价值 / 领域绑定 / 脱域说明 / 区分说明 / 必要说明 / 依赖 / 组件

## 最小 Frontmatter

```yaml
id:
status:
source:
depends_on:
components:
```

不要写能从图里推出来的字段：`used_by`、`reuse_count`、`part_of`、`closure`、`type`、`related` 等。

## 核心本体

**机制原子**是携带完整机制的最小可复用单元。它不是词语，不是句子，不是笔记卡片。

每个机制原子必须能说清四件事：

1. **机制核**：到底发生了什么（因果链或结构约束）
2. **参与角色**：哪些东西参与了这个机制，各自起什么作用
3. **成立条件**：机制在什么条件下运作
4. **迁移边界**：什么情况下不适用，不能迁移到哪些领域，为什么

判断一个东西是不是机制原子，问四句话：
- 它有没有机制？（不只是标签）
- 它能不能解释为什么？
- 它能不能迁移到共享同一机制的其他领域？
- 它有没有边界，防止瞎迁移？

### 基础原子、机制原子、复合原子

- **基础原子**：不可再拆的概念、属性、操作或关系。`components: []`。例如：`反馈`、`平衡`、`阈值`
- **机制原子**：携带完整四要素的最小可复用单元。核心资产。
- **复合原子**：由多个原子组合而成。`components` 不为空。可折叠/展开。

### 领域节点 vs 脱域节点

- **领域节点**：绑定在原领域，依赖领域专属概念。如 `研磨度`、`粉水比`
- **脱域节点**：已抽离领域对象，可在多领域复用的机制原子

## 编译流程

## Compile 执行清单

按以下顺序执行，每一步完成后再进入下一步。完成后在回复中报告每步结果。

### 1. 读取输入
读取材料全文。不预判，不跳读。

### 2. 分离领域与机制
对每一段核心内容判断：去掉领域专属词→替换为抽象角色→机制还完整吗？
完整→进入步骤3。不完整→标记为领域节点（不脱域），写入 atoms/。

### 3. 提取机制四要素
机制核（因果链）、参与角色（列表+功能）、成立条件（何时运作）、迁移边界（何时不适用、不能迁移到哪、为什么）。

### 4. 判断层级
按 L0-L5 分层标准判断每个候选原子的层级。检查：components 是否都低于当前层？depends_on 目标是否不高于当前层？跨层依赖→禁止。

### 5. 分类
按 `docs/12-foundation-grounding.md` 分类框架判断每个概念/机制的 CLC 方向。**看场景用途，不看表面词汇。** 遇到易混淆边界参照分类决策规则。

### 6. 检索已有原子
名称、别名、机制核关键词、角色关键词、条件关键词、跨领域同构。读最接近的 3 个已有原子。

### 7. 复用/新建/升级/拆分/合并
按 `docs/06-build-decision-rules.md` 判断。旧原子能承载→复用。机制一致但边界窄→升级。混入多机制→拆分。重复→合并。没有独立复用价值→丢弃（不建 rejected）。

### 8. 补齐前置概念
上层原子引用的概念在库中没有定义？→先建概念原子（场景用途+CLC方向+定义+边界+一句话机制），再建上层原子。

### 9. 写入原子
默认 `draft`。正文 wikilink 必须标注语义关系类型（composed_of/prerequisite/supports/constrains/explains/contrasts_with/instantiates...）。领域绑定表至少有一行。

### 10. 建立本批原子间关系
所有本批原子写入后，两两检查语义关系。对称机制（"不足→X"和"过度→Y"）必须建立 contrasts_with。同批领域节点与脱域机制间必须建立 instantiates。

### 11. 检查跨领域绑定
新脱域机制原子→已有原子库中是否存在可建立 instantiates 关系的领域节点？有→补入领域绑定表。

### 12. 质量检查
单原子质量门（21道）+ 硬校验（ID唯一/无循环/必填字段/禁用字段）+ 整批验收（未漏核心机制/未拆太碎/未拆太粗/未重复新建/未把摘要当原子/未把假迁移当真迁移）。

### 13. 涌现验证
这批原子中的某几个组合→能否生成原文没有直接表述的新判断？能→在结果说明中写涌现路径。连续多次零涌现→检查是否只在做摘要。

### 14. 结果报告
本次编译：复用了哪些、新建了哪些（区分机制原子/领域节点/概念原子）、哪些脱域了验证结果如何、补齐了哪些前置概念、升级/拆分/合并/丢弃了哪些及理由、涌现路径。

## 分层构建

原子不按主题分类，按**构造层级**分层。上层只能引用下层，不能跨层。

```text
L0 基础概念    components: []  不可再拆的概念/属性/操作
L1 基础关系    components→L0   最简单的三元组（A→关系→B）
L2 组合概念    components→L0/L1 由基础概念和关系组合
L3 机制原子    components→L0/L1/L2 携带完整四要素
L4 命题        depends_on→L2/L3 一个完整判断
L5 复合系统    components→L3/L4 多机制多命题的系统
```

创建每个原子时必须判断层级。检查：components 是否都处于严格低于当前层？depends_on 目标是否不高于当前层？

## 语义关系

`depends_on` 和 `components` 只说"有关系"，不说"什么关系"。语义类型标注在 wikilink 中，不新增字段：

```markdown
- [[atom-id]] - 关系类型：理由
```

结构语义（components 中）：`composed_of` | `implements` | `instantiates`
认知语义（depends_on 中）：`prerequisite` | `defined_by` | `grounded_in`
逻辑语义（任意）：`supports` | `constrains` | `explains` | `contrasts_with`

选择规则：先判断是 components 还是 depends_on，再选语义子类型。

## 本批原子关系

所有本批原子写入后，两两检查是否存在 depends_on / components / supports / constrains / explains / contrasts_with / instantiates。同批创建的领域节点与脱域机制之间必须建立 instantiates。对称机制（"不足→X"和"过度→Y"）必须建立 contrasts_with。

## 涌现验证

编译结束后，检查这批原子能否组合出原文没有的新判断。能 → 在结果说明中写出涌现路径 `[atom_X] + [atom_Y] → "新判断"`。连续多次零涌现 → 可能只在做摘要。

## 跨领域迁移验证

判断一个机制原子能否迁移到新领域：

1. 新领域存在该机制需要的所有参与角色吗？
2. 新领域满足该机制的成立条件吗？
3. 迁移后能在新领域生成有效的解释或预测吗？
4. 这不是只在句式上相同吧？

四个都是"是"→ 可以迁移。任何一个"否"→ 不能迁移。

**真迁移**（共享机制）：咖啡萃取 → 泡茶/煲汤/中药煎煮（共享：介质接触材料→成分转移）

**假迁移**（换词造句）：咖啡萃取 → 学习掌握（机制完全不同）

## 互动结构扫描

当材料涉及多人互动、关系、交易、竞争、合作、权力、信号或奖惩时，先找：玩家、目标、信息、策略空间、规则点、奖惩、信号、可能结果。只有规则点能通过机制精度门时，才沉淀为原子。

## 正文要求

每个原子至少写：定义、边界、机制核心、参与角色、成立条件、迁移边界、复用价值、领域绑定（或脱域说明）、区分说明、必要说明、依赖、组件。

## 状态

只使用：`draft`、`stable`、`deprecated`、`merged`。不要使用 `rejected`。

## 安全

新原子默认 `draft`。只有通过质量门、硬校验和整批验收后，才能 `stable`。

写入或重构后，在最终回复里说明：复用了哪些、新建了哪些（区分机制原子与领域节点）、哪些脱域了、补齐了哪些地基、升级/拆分/合并了哪些、丢弃了哪些。
