# Nix Knowledge Compiler v0.4 Reference

## 目标

把文章、对话、摘录、想法编译成可复用、可追溯、可组合的 Markdown atoms。目标不是摘要，而是建立未来 AI 和人都能调用的知识底座。

atom 是可复用知识函数，必须能写成：

```text
输入 -> 操作 -> 输出
```

基础 atom 是不可再拆的最小知识函数，`components: []`。系统 atom 是多个知识函数组合出的更大函数，`components` 不为空。

任何系统 atom 必须能沿 `components` 和 `depends_on` 递归追溯到底层基础 atom。依赖底层学科原理时，必须通过 `depends_on` 和正文 wikilink 逐层链接到底层原理 atom，不能只写学科名。

## 最小 store

只在当前 workspace root 内工作。核心目录只有：

```text
atoms/
```

所有可复用知识都是 atom。系统不是单独类型，而是 `components` 不为空的高层级 atom。

如果当前 workspace root 没有 `atoms/`，只创建 `atoms/`。不要创建全局库，不要创建额外目录。

## 语言与命名

atom 的可读部分跟随输入材料主语言：

- 中文输入：中文文件名、中文 `id`、中文 H1、中文正文标题、中文正文。
- 英文输入：英文文件名、英文 `id`、英文 H1、英文正文标题、英文正文。
- 混合输入：按主要语言生成，必要术语保留原词。

中文输入示例：

```text
atoms/关系行为惯性-v1.md
```

```yaml
id: 关系行为惯性-v1
```

禁止把中文概念翻译成英文 slug。禁止用英文正文标题处理中文输入。

中文正文标题固定为：

- 定义
- 边界
- 底层机制
- 复用价值
- 区分说明
- 必要说明
- 依赖
- 组件

## 最小 frontmatter

```yaml
id:
status:
source:
depends_on:
components:
```

不添加 type、object kind、level、content shape、system scope、related、reverse links、reuse count、closure、replaces、replaced_by、merge_note。

## 决策顺序

每个候选按顺序判断：

1. 能否写成“输入 -> 操作 -> 输出”。
2. 如果是上层函数，先拆 `components`，直到能追溯到底层基础 atom。
3. 如果依赖底层学科原理，先检索或创建对应底层原理 atom；缺地基时先补地基，再写上层 atom。
4. 如果材料涉及互动、关系、交易、竞争、合作、权力、信号或奖惩，抽互动结构。
5. 没有独立复用价值，丢弃。
6. 搜索已有 atom：候选名称、别名、函数签名、机制词、相邻概念、可能的上层 atom。
7. 旧 atom 能准确承载，复用旧 atom。
8. 旧 atom 机制正确但边界太窄，升级旧 atom。
9. 旧 atom 无法承载且候选有独立复用价值，才新建。
10. 候选混入多个独立函数，拆分。
11. atom 实质重复，合并。

不要创建 `status: rejected` atom。失败候选不进入 `atoms/`。

## 准确承载

旧 atom 能准确承载候选，必须满足：

- 函数签名兼容。
- 底层机制相同。
- 边界兼容。
- 复用功能相同。
- 依赖结构兼容。

如果满足，就复用，不新建重复 atom。

## 原子精度

合格 atom 必须是可复用知识函数，不是文章观点、技巧标题或局部话术。

新建前必须能写出：

```text
输入 -> 操作 -> 输出
```

新建前必须能写出：

```text
触发条件 -> 中介机制 -> 可观察结果 -> 适用边界
```

判断：

- 只能服务当前文章：不建。
- 只是例子、句子、标签、修辞、话术：不建。
- 只是把原文换个名字：不建。
- 混入多个独立机制：拆分。
- 再拆就失去独立用途：可以保留。
- 能被多个任务或更大 atom 引用：可以保留。

系统 atom 只有在组件组合后产生新能力时才建。文章目录、章节结构、技巧清单默认不等于系统 atom。

系统 atom 必须能沿 `components` 和 `depends_on` 递归追溯到底层基础 atom。不能追溯的系统 atom 是黑箱，不新建。

## 底层学科原理

底层基础 atom 可以来自逻辑与概念分析、数学与图、概率统计、信息论、微观经济学、博弈论、决策理论、心理学、认知科学、进化心理学、语言学、传播学、社会学、系统论、控制论、复杂性科学、工程设计等底层学科。

学科不写成 frontmatter 字段。正文“必要说明”里写学科来源；正文“依赖”里用 `[[底层原理atom-id]] - 依赖理由` 逐层链接到底层原理 atom。

如果上层候选依赖某个底层原理，但没有对应 atom 承载它，必须先创建或复用底层原理 atom；否则不要孤立创建上层 atom。

## 地基补齐

地基 atom 是承载底层学科原理的基础 atom，不是学科标签。

初始地基检索地图：

- 逻辑与概念分析：定义边界、必要条件、充分条件、因果链、反例检验、分类标准。
- 数学、关系与图：函数、集合、关系、映射、层级、组合、网络节点、路径、闭包。
- 概率统计：基率、条件概率、贝叶斯更新、样本偏差、方差、噪声、相关不等于因果。
- 信息论与信号：信息增益、信号噪声分离、压缩损失、信道、代价信号、可伪造信号。
- 微观经济学：稀缺性、机会成本、边际收益、边际成本、激励、交易成本、外部性、委托代理。
- 博弈论与机制设计：玩家、策略空间、收益函数、均衡、重复博弈、承诺、协调、信息不对称、信号博弈。
- 决策理论与行为经济学：期望值、风险、损失厌恶、时间偏好、权衡、有限理性、启发式判断。
- 心理学与认知科学：注意力、记忆、预测误差、强化、习惯、认知偏差、情绪调节、动机。
- 进化论与进化心理学：选择压力、适应、互惠、亲缘选择、地位竞争、性选择、代价信号。
- 语言学、语用学与传播：语义、语用、言语行为、框架、叙事、共同知识、歧义、受众解码。
- 社会学、制度与网络：角色、规范、身份、地位、信任、社会资本、制度约束、扩散。
- 系统论、控制论与复杂性：反馈回路、延迟、库存流量、瓶颈、涌现、路径依赖、吸引子、非线性。
- 工程、设计与组织：模块化、接口、约束、鲁棒性、冗余、摩擦、可维护性、可组合性。

这张地图不是分类体系，不进入 frontmatter。只在当前材料真的依赖某个原理时，才创建或复用对应 atom。

缺少地基时：

1. 先创建或复用必要的地基 atom。
2. 地基 atom 的 `source` 指向当前输入，并在正文“必要说明”里写“由当前材料触发的底层原理补齐”。
3. 地基 atom 仍要写清定义、边界、函数签名、底层机制、复用价值、区分说明、依赖、组件。
4. 再创建上层 atom，让上层 atom 链到这些地基 atom。

停止下钻的标准：当前 atom 已经是完整的“输入 -> 操作 -> 输出”，再拆会丢失输入、操作或输出，且它已经能作为多个上层 atom 的底层原理使用。

同一个底层机制跨学科出现时，优先创建一个更通用的 atom，在正文“必要说明”里写常见学科来源，不要按学科复制重复 atom。

## 互动结构

互动结构不是新字段，也不是 atom 类型。它是 compile 前的提取步骤。

当材料涉及互动、关系、交易、竞争、合作、权力、信号或奖惩时，先找：

- 玩家：谁在局里。
- 目标：每个玩家想得到什么、避免什么。
- 信息：谁知道什么，谁不知道什么，谁在隐藏什么。
- 策略空间：每个玩家能做什么、不能做什么、能不能退出。
- 规则点：稳定影响选择、信息暴露、成本收益或关系走向的约束和生成机制。
- 奖惩机制：做某动作会获得什么、失去什么、把成本转嫁给谁。
- 信号机制：哪些行为、语言、身体反应、历史记录会泄露隐藏偏好或策略。
- 可能结果：合作、竞争、僵局、退出、背叛、责任转移或稳定均衡。

只有规则点能通过原子精度门时，才沉淀为 atom。玩家名单、单个案例、一次性策略、文章里的技巧步骤，不直接成 atom。

规则点可以来自心理学、生理学、社会学、经济学、博弈论、语言学、传播学等学科。学科只是来源，不是 atom 分类。

## 批量去重

一次 compile 里，即使原 `atoms/` 为空，也要先比较本批候选。

区分说明不能写“拆解前 atoms 为空”。必须写具体检查过哪些旧 atom 或本批候选，以及为什么不能承载。

## 正文

中文 atom 至少写：

```markdown
# 概念名称

## 定义

## 边界

## 底层机制

## 复用价值

## 区分说明

## 必要说明

## 依赖

## 组件
```

“底层机制”必须写因果链、结构约束或生成机制，不只写“这个方法有用”。

“底层机制”必须包含函数签名：

```text
输入 -> 操作 -> 输出
```

“复用价值”必须写清调用方式：未来遇到什么问题时应该引用这个 atom。

“依赖”和“组件”正文必须同步使用 Obsidian wikilink：

```markdown
- [[atom-id]] - 为什么理解当前 atom 必须先理解它
```

frontmatter 保留纯 ID 列表；正文链接用于阅读和反向链接。不要新增 `part_of`、`used_by`、`related`、`see_also` 等字段。

## 状态与生命周期

只使用：

- `draft`
- `stable`
- `deprecated`
- `merged`

新 atom 默认 `draft`。

`stable` 不表示“内容为真”，只表示身份、来源指针、边界、依赖和组成关系稳定到可以复用。

普通措辞、例子、边界、来源补充不需要新版本。定义、机制、边界或依赖结构发生实质变化时，才新建版本或替代 atom。

`merged` atom 的承载目标和合并原因写在正文“必要说明”，不新增 frontmatter 字段。

## 校验

写入后检查：

- ID 唯一。
- 文件名等于 ID。
- ID 带版本后缀。
- 文件名、ID、H1、正文标题符合输入语言。
- frontmatter 必填字段存在。
- status 合法。
- 依赖和组件指向存在的 atom。
- `depends_on` 和 `components` 无循环。
- 没有禁用字段。
- 没有 `status: rejected`。
- source 可追溯。
- 区分说明具体。
- 函数签名具体。
- 底层机制通过精度门。
- 系统 atom 能追溯到底层基础 atom。
- 依赖底层学科原理时已链接到底层原理 atom。
- 缺少底层学科原理时，已先补齐地基 atom。
- 复用价值写清调用方式。
- 能说明上层系统候选或可组合方向。
- 正文依赖和组件使用 wikilink。

整批验收还要检查是否漏掉核心概念、拆太碎、拆太粗、重复新建、把摘要当 atom、未充分复用旧 atom，或削弱依赖图。

## AI 调用协议

使用 store 辅助思考时：

1. 明确任务。
2. 检索相关 atom。
3. 读取直接命中的 atom。
4. 需要前置理解时展开 `depends_on`。
5. 需要系统结构时展开 `components`。
6. 默认最多展开 2 层。
7. 不读取无关 atom。
8. 任务结束后判断是否有新 atom 值得沉淀。

## Obsidian

Obsidian 只是阅读入口，不是核心数据模型。

- 文件名等于 ID。
- H1 使用输入语言。
- frontmatter 保持最小。
- 依赖和组件章节使用 `[[atom-id]] - 理由`。
- Obsidian 双链只是阅读呈现，不能替代 frontmatter。
- 结构关系以前置 frontmatter 的 `depends_on` 和 `components` 为准。
- 反向关系不手写字段，通过 Obsidian backlinks、全库搜索和系统 atom 的 `components` 反查获得。
- 每一层 atom 都必须向下链接，直到抵达 `components: []` 的基础 atom 或承载底层学科原理的基础 atom。
