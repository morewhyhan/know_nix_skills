# Mechanism Atom Compiler v0.5 Reference

## 目标

把文章、对话、摘录、想法编译成可脱域、可跨领域复用的机制原子。目标不是摘要，而是提取可迁移的机制单元。

机制原子是携带完整机制的最小可复用单元，必须能说清：

```text
机制核：发生了什么（因果链或结构约束）
参与角色：哪些东西参与
成立条件：在什么条件下运作
迁移边界：什么情况下不适用，不能迁移到哪里
```

基础原子是不可再拆的概念、属性、操作或关系。复合原子是多个原子组合出的更高阶结构，可折叠/展开。

## 核心分类原则

**按场景用途分类，不按表面词汇分类。** "成本"在企业→经济学（F0），在注意力→心理学，在候鸟迁徙→生态学（Q4）。同一个词在不同场景属于不同学科。判断依据不是"词是什么"，而是"这个场景在回答哪个学科的核心问题"。

**定义遵循维特根斯坦原则：意义在于用法。** 不给概念写终极词典定义——写它在当前场景中具体怎么用、和什么对立、排除什么。同一个词在不同场景中是不同的知识对象。

完整分类框架和边界定义见 `docs/12-foundation-grounding.md`。

## 最小 store

只在当前 workspace root 内工作。核心目录只有 `atoms/`。

所有可复用知识都是原子。系统不是单独类型，而是可展开的复合原子。

## 语言与命名

原子的可读部分跟随输入材料主语言。中文输入 → 中文文件名、中文 ID、中文 H1、中文正文标题、中文正文。不把中文概念翻译成英文 slug。

中文正文标题固定为，按此顺序不可变：场景用途 / CLC方向 / 定义 / 边界 / 机制核心 / 参与角色 / 成立条件 / 迁移边界 / 复用价值 / 领域绑定 / 脱域说明 / 区分说明 / 必要说明 / 依赖 / 组件。其中"场景用途"和"CLC方向"是 v0.5 新增——先锚定场景再分类，不按表面词汇归类。

**领域绑定和脱域说明是两个独立 section，不合并为"领域绑定或脱域说明"。**

**Section 按原子类型区分：** 机制原子写全上述所有15个 section。概念原子（L0-L2）和领域节点跳过 机制核心/参与角色/成立条件/迁移边界 这四个 section，不要写"（概念原子，不适用）"占位文字。

## 最小 frontmatter

```yaml
id:
status:
source:
depends_on:
components:
```

`source` 写入来源材料的路径或引用标识，如 `01-文章/爱戏剧的柏拉图/001_看完你将认清婚姻4大真相.md`。

不添加 type、object_kind、level、content_shape、system_scope、related、reverse links、reuse count、closure 等字段。

## 决策顺序

每个候选按顺序判断：

1. 分离领域内容与机制内容（替换领域词为抽象角色，机制还完整吗？）
2. 提取机制四要素：机制核、参与角色、成立条件、迁移边界。
3. 判断脱域可能性：能在至少一个其他领域验证吗？
4. 如果引用了库中未定义的概念，先补齐概念定义原子。
5. 如果材料涉及互动、关系、交易等，抽互动结构。
6. 没有独立复用价值，丢弃。
7. 搜索已有原子：名称、别名、机制核关键词、角色关键词、条件关键词、跨领域同构检索。
8. 旧原子能准确承载，复用。
9. 旧原子机制一致但边界太窄，升级。
10. 旧原子无法承载且候选有独立复用价值且通过机制精度门，新建。
11. 候选混入多个独立机制，拆分。
12. 原子实质重复，合并。

不要创建 `status: rejected` 原子。

## 准确承载

旧原子能准确承载候选，必须满足：
- 机制核一致。
- 角色结构兼容。
- 成立条件兼容。
- 迁移边界兼容。
- 领域绑定兼容。

如果满足，就复用。

## 机制精度门

合格机制原子必须能写全四要素：

```text
机制核 + 参与角色 + 成立条件 + 迁移边界
```

判断：
- 只能服务当前文章：不建。
- 只是例子、句子、标签、修辞、话术：不建。
- 只是换词类比（句式相同、机制不同）：不建，这是假迁移。
- 混入多个独立机制：拆分。
- 再拆就失去完整机制：可以保留。
- 能被多个任务或更大原子引用：可以保留。
- 能在至少一个其他领域验证迁移：可以标注为脱域机制原子。

复合原子只有在组件组合后产生新能力时才建。

## 脱域

脱域是从领域内容中提取跨领域机制的过程：

1. 识别领域专属概念。
2. 替换为抽象角色。
3. 验证机制完整性。
4. 找至少一个其他领域验证迁移。
5. 创建领域绑定表。
6. 写清楚迁移边界。

脱域不是换词。脱域要求底层机制真的一致。

真迁移：介质接触材料 → 成分转移（咖啡、泡茶、煲汤、中药）
假迁移：萃取时间影响萃取程度 → 练习时间影响掌握程度（机制不同）

## 跨领域迁移验证

判断机制原子能否迁移到新领域：
1. 新领域存在所有参与角色吗？
2. 新领域满足成立条件吗？
3. 迁移后能生成有效判断吗？
4. 不是只在句式上相同吧？

四个"是"→ 可迁移。任何"否"→ 不能迁移。

## 多尺度折叠展开

任何节点都可以在一种尺度上作为原子（折叠），在另一种尺度上作为系统（展开）。

- 折叠：把复杂系统当成可调用的黑箱节点。
- 展开：打开节点，暴露内部的角色、条件、子机制。

原子不是绝对最小单位。原子是相对于当前任务的最小可操作机制节点。

## 前置概念补齐

上层机制原子引用的每个概念，在库中必须有清晰的脱域定义。学科概念（机会成本、反馈回路、瓶颈、贝叶斯更新、代价信号……）天然适合作为前置依赖——它们本身就是被多领域验证过的脱域概念。

优先检索方向（参照 DDC/CLC/LCC 图书馆分类体系对知识领域的覆盖）：哲学逻辑与认识论、数学关系与图、概率与统计、物理学、化学、生物学与生态学、信息论与信号、计算机科学、经济学、金融、博弈论与策略、决策理论与行为经济学、心理学与认知科学、进化论与进化心理学、语言学与语用学、传播学与媒体、社会学与制度、法律与制度设计、教育学、地理学与空间分析、系统论控制论与复杂性、工程设计组织、军事与战略、美学与感知。

以及跨领域机制构件：不足/过度/平衡、到达-处理失衡、固定成本摊薄、介质成分转移、敏感度差异、反馈-调整-再输出、稀缺-竞争-分配、局部最优vs全局最优。

补齐到概念能被独立理解即可，不继续下钻。学科来源只说明它天然可复用，不需要写成分类字段。

## 互动结构

当材料涉及互动、关系、交易、竞争、合作、权力、信号或奖惩时，先找：玩家、目标、信息、策略空间、规则点、奖惩、信号、可能结果。

只有规则点能通过机制精度门时，才沉淀为原子。

## 批量去重

一次 compile 里，即使原 `atoms/` 为空，也要先比较本批候选。区分说明不能写"拆解前 atoms 为空"。

## 正文

每个原子按类型写入对应的 section。机制原子写全所有15个 section；概念原子和领域节点跳过 机制核心/参与角色/成立条件/迁移边界。

**各 section 格式规则：**

**机制核心**（仅机制原子）：包裹在 ` ```text ``` ` 代码块中。纵向叙事流——`→` 箭头串联因果，缩进表示层级，因果关系通过缩进和箭头自然展开。

结构形态按机制实际需要自由选择——线性因果、信号合成、条件分叉等模式可自由组合嵌套。核心原则：纵向叙事流、`→` 缩进层级串联因果、正文自然展开不分段加粗标签。完整规范和参考示例见 `templates/knowledge-atom.md`。

**参与角色**（仅机制原子）：表格格式（`| 角色 | 说明 |`），列出所有角色及其功能。

**成立条件**（仅机制原子）：编号列表。

**迁移边界**（仅机制原子）：**必须包含三层，缺一不可：**
1. **可迁移到**：正面迁移示范，每个领域给出角色映射+迁移后效果。至少2个领域。
2. **不能迁移到**：负面边界，说明什么领域/场景不适用以及为什么。
3. **假迁移警惕**：具体错误示例+一句话判断标准。

**复用价值**：两段式。第一段说机制价值，第二段以"调用方式："开头给出可操作 checklist（检查 a/b/c 条件）。

**定义**：结构为"一句定义正文 → '与XX的区别'列举 → '排除：'收尾"。与相近概念的区分写在定义中，概念的精细辨析放在区分说明。

**领域绑定**：表格列出跨领域角色映射。如果只有一个领域验证，写清"当前仅在X领域中验证"。

**脱域说明**：原领域、去掉了什么、保留了什么、为什么可以迁移。领域节点写"当前未脱域"。**此 section 与领域绑定为独立 section，不合并。**

**必要说明**：必须包含"学科来源"（学术脉络+引用）和"常见别名"（至少2个逗号分隔）。按需补充必要例子、限制条件、本批新建术语声明。

**区分说明**：先声明与已有原子的关系，再用 `[[atom-id]] - 语义关系类型：理由` 格式列出。同时说明与本批其他原子的关系。

**依赖**和**组件**：使用 Obsidian wikilink `[[atom-id]] - 语义关系类型：理由`。即使为空写"无"。

## 状态与生命周期

只使用：`draft`、`stable`、`deprecated`、`merged`。新原子默认 `draft`。

`stable` 不表示"内容为真"，只表示结构稳定到可以复用。

领域绑定的增删不升版本。机制四要素变化才升版本或新建替代原子。

## 校验

写入后检查：ID 唯一、文件名等于 ID、必填字段存在、status 合法、依赖和组件指向存在的原子、无循环、无禁用字段、机制四要素在正文中完整、声称脱域的原子有迁移边界和领域绑定、复合原子能追溯到底层基础原子。

整批验收还要检查是否漏掉核心机制、拆太碎、拆太粗、重复新建、把摘要当原子、把假迁移当真迁移、未充分复用旧原子。

## AI 调用协议

使用 store 辅助思考时：
1. 明确任务。
2. 按机制检索（不只是关键词检索）。
3. 读取直接命中的原子。
4. 需要前置理解时展开 `depends_on`。
5. 需要系统结构时展开 `components`。
6. 默认最多展开 2 层。
7. 跨领域查询时查找同构机制原子的领域绑定。
8. 不读取无关原子。

## Obsidian

- 文件名等于 ID。
- H1 使用输入语言。
- frontmatter 保持最小。
- 依赖和组件章节使用 `[[atom-id]] - 理由`。
- Obsidian 双链只是阅读呈现。
- 结构关系以前置 frontmatter 为准。
- 反向关系通过 Obsidian backlinks 和全库搜索获得。
