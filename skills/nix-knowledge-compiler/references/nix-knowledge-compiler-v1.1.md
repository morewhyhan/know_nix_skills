## 目标与责任边界

把文章、对话、摘录和想法编译成两类互补产物：足以替代重新阅读原文的来源还原稿，以及可复用、可追溯、可组合的 Markdown 机制 atom。还原稿保存来源完整意思；atom 建立未来 AI 和人都能调用的知识函数库。

维护责任全部属于 AI：全文还原、CLC 全表路由、术语核验、候选提取、独立复核、复用检索、证据分层、关系维护、写入、验证、修复、合并和状态升级都不能转嫁给用户。用户只提供材料或查询任务。

## 双层编译对象

### 来源还原稿

每个可解析来源在 `reconstructions/` 中最多对应一个活动还原稿。它不是原文副本，也不是为了简短而省略重要信息的摘要，而是保留理解来源所必需的语义模型。

frontmatter 只允许：

```yaml
source: C:/可解析的来源路径
source_sha256: 原文字节的完整 SHA-256
```

正文固定为六节：

1. 核心结论
2. 论证链
3. 关键概念
4. 建议与适用范围
5. 证据与文本问题
6. 机制 atom 映射

`证据与文本问题` 必须含 `原文替代验收：通过；范围：... / 未通过；原因：... / 未执行；原因：...`。只有未读取 atom 成稿、且由隔离答题与裁判流程实际完成的审查者才能写“通过”。来源字节变化后，`source_sha256` 失配，还原稿立即视为过期并由 AI 自动重建。

### 机制 atom

atom 只保存跨来源可复用的知识函数。不能成为 atom 的作者立场、文章结构、关键但来源专属的概念、建议、话术、案例、修辞、事实断言和文本矛盾必须保留在还原稿，不能直接丢弃。

## atom 本体

每个有效 atom 都是知识函数，必须同时能写成：

```text
输入 -> 操作 -> 输出
触发条件 -> 中介机制 -> 可观察结果 -> 适用边界
```

以下内容不能单独成为 atom：

- 文章标题、章节、目录或摘要。
- 概念标签、人物、案例、金句或修辞。
- 一次性建议、话术或方法步骤。
- 只有相关性、没有中介机制的观点。
- 只把领域名词替换掉的类比。

### 基础 atom

基础 atom 是不可再拆的完整知识函数。再拆会失去输入、操作或输出。`components: []`。

基础不等于“词语最小”或“学科概念”。一个机制即使包含多个普通词，只要已经是最小完整函数，也可以是基础 atom。

### 系统 atom

系统 atom 由多个下层 atom 组合，并产生任何单个组件都没有的新能力。`components` 非空，且每个组件必须说明职责。

不要使用固定 L0-L5 字段。层级、闭包和反向引用由图计算。

### 依赖

`depends_on` 只包含理解或调用当前 atom 真正必需的前置知识函数。正文已自足定义的术语、学科名和普通概念不自动成为依赖。

缺少真正必要的地基时，AI 自动搜索、复用或编译地基；如果来源不足以支持地基，不得凭空创建，上层候选保持未写入或 draft 并记录缺口。

## 唯一 frontmatter

只允许以下七个字段：

```yaml
id: 机制名称
status: draft
source:
  - C:/absolute/or/workspace-relative/source.md
tags:
  - C93/管理学
aliases:
  - 常见别名一
  - 常见别名二
depends_on: []
components: []
```

### 字段规则

- `id`：store 内唯一；文件名必须等于 `id + ".md"`。
- `status`：只允许 `draft`、`stable`、`deprecated`、`merged`。
- `source`：非空列表；每项必须是可解析的绝对路径、workspace 相对路径或 `http(s)` URL。同一 atom 获得新来源时追加，不覆盖旧来源。URL 可供 `draft` 追溯；升级 `stable` 时至少两项必须解析为两个不同的本地来源文件，不能用未抓取 URL 或同一文件的不同写法凑数。
- `tags`：1–3 个导航标签，只能取自 `classification.md` 的 `CODE/CLC正式类名`。分类不承担地基或机制证明功能；裸学科词和自定义跨学科词不是合法 CLC tag。
- `aliases`：2–6 个可实际用于搜索的别名，不能只是重复标题或全部是技术改写；至少包含普通人实际会说或输入的一种检索表达。
- `depends_on`：前置 atom ID 列表。
- `components`：组成当前系统 atom 的下层 atom ID 列表。

禁止 `type`、`level`、`part_of`、`used_by`、`related`、`evidence_status`、`domain` 等额外字段。可推导信息由图或正文维护。

## 命名与语言

- 文件名、ID、H1、别名和章节内容跟随来源主语言；15 个 H2 机器章节键固定为中文，确保 validator 与跨语言 store 使用同一解析接口。
- 命名前必须按 `classification.md` 扫描全部 22 个 CLC 基本大类，再展开可能相关的正式方向，在入选方向与邻接方向中核验现成术语。
- ID 的严格优先级是：与函数签名、中介机制和边界精确一致的既有学科术语；已有通俗口语；清楚的描述性短语；编译器自造词。
- 专业术语只有部分相似时不得冒名。应拆分 atom、把该术语写为近邻，或降到下一命名优先级。不得为了显得专业而制造似是而非的新术语。
- `必要说明/命名依据` 必须写明 `学科术语`、`通俗用语`、`描述性命名` 或 `编译器造词`，并给出核验方向和理由。使用编译器造词时必须说明前三层为什么都不可用。
- aliases 优先覆盖：标准术语的其他译名或上位词、来源原话、生活口语。口语别名只是检索入口，不自动等于机制定义或证据。
- 初始版本使用干净 ID，不强制 `-v1`。
- 普通措辞、来源和绑定补充不升版本。
- 机制身份、核心因果方向或关键边界发生破坏性变化时，创建 `-v2` 等新 ID；旧 atom 标记 `deprecated` 或 `merged`。

## 正文契约

所有活动 atom 使用以下固定中文机器键与顺序：

1. 场景用途
2. CLC方向
3. 定义
4. 边界
5. 机制核心
6. 参与角色
7. 成立条件
8. 迁移边界
9. 复用价值
10. 领域绑定
11. 脱域说明
12. 区分说明
13. 必要说明
14. 依赖
15. 组件

`deprecated` 和 `merged` 历史 atom 可以省略机制四节，但必须保留身份、来源、去向和迁移理由。

### 机制核心

必须明确写出函数签名和机制链：

```text
函数签名：输入 -> 操作 -> 输出

触发条件
  -> 中介机制
    -> 可观察结果
```

不能用一串解释性名词代替因果或结构约束。

### 参与角色

使用表格说明每个角色在机制中的功能。脱域 atom 使用抽象角色，不把来源人物当角色名称。

### 成立条件与迁移边界

成立条件必须可用于判断“不适用”。迁移边界至少包含：

- 可迁移结构：哪些角色和条件必须保持。
- 不能迁移到：具体失效场景与原因。
- 假迁移警惕：最容易混淆的错误迁移及判定标准。

### 复用价值

必须以“调用方式：”给出未来 AI 可执行的检查清单。

### 区分说明

列出实际检查过的旧 atom 或本批候选，并说明为何复用、升级、相关但不复用或新建。不能写“库为空所以新建”。

### 必要说明

必须使用以下明确标签：

```text
证据状态：单一来源 / 多源互证 / 权威原理+应用验证 / 证据不足
来源支持：来源直接支持的现象、条件或关系及定位
编译器推导：AI 补出的抽象角色、中介机制或一般化
待验证：尚未核实的事实、因果、边界或迁移；没有则写“无”
学术线索：可核验引用；没有可靠依据就写“无”，不能补造
命名依据：命名层级、CLC 核验方向与选择理由
常见别名：与 frontmatter aliases 一致
独立语义复核：通过；范围：... / 未通过；原因：... / 未执行；原因：...
```

“独立语义复核”由未参与成稿的 AI 审查者填写。主 agent 的第二遍自查不能写“通过”；没有可用独立审查者时保持 `draft`。

以上八个标签必须各出现且只出现一次。复核结论必须写成 `通过；范围：具体内容`、`未通过；原因：具体内容` 或 `未执行；原因：具体内容`，不能保留多个状态或省略号占位。

## CLC 全表检索

CLC 是所有检索、命名和导航的第一路由，不是几个常用方向的候选清单。

1. 逐一排查 A、B、C、D、E、F、G、H、I、J、K、N、O、P、Q、R、S、T、U、V、X、Z 22 个基本大类。
2. 对可能相关的大类展开到 bundled `classification.md` 所列最细正式类号；按 atom 回答的调用问题分类，而不是按来源出现的词分类。
3. 在主类与邻接类中检索标准术语、别名和旧 atom；跨学科术语必须映射回正式 CLC 类号。
4. 本地表未列到更深类号时，回退到真实上位类，不得猜码或自创码名。
5. `CLC方向` 正文必须写明 `CLC全表扫描：已检查22个基本大类`、与 frontmatter 完全一致的 `主类：...`，以及实际展开的邻接方向和排除理由。

## 证据与领域绑定

每个领域绑定必须标记一种验证状态：

- `来源支持`：当前来源直接支持该领域实例。
- `独立材料验证`：另一份独立来源支持相同函数签名和机制；该表行必须在“证据定位”或“来源”列以 frontmatter source 的精确 locator 或无歧义 basename 开头，再给出段落定位。
- `机制推演`：只通过角色映射和逻辑演绎得到，尚未独立验证。

推荐表格：

```markdown
| 领域 | 角色映射 | 结果 | 验证状态 | 来源 |
|---|---|---|---|---|
```

规则：

- `机制推演` 不是验证，不能单独支持 `stable`。
- `来源支持` 与 `独立材料验证` 必须解析到不同 source identity；同一文件的两种路径、内容哈希相同的副本和未抓取 URL 都不构成独立来源。
- 评论性来源中的事实断言仍是“来源支持的断言”，不是外部确认事实。
- AI 必须区分来源原话、编译器推导和外部证据。
- 跨领域输出必须保持相同函数签名；如果输出从“吸引”变成“谨慎、服从、学习”等不同结果，默认是假迁移或机制族，需要拆分。

## 关系语义

frontmatter 只存目标 ID；正文 `依赖` 和 `组件` 用 wikilink 加语义：

```markdown
- [[atom-id]] - 关系类型：理由
```

### `components` 允许

- `composed_of`
- `implements`
- `instantiates`

### `depends_on` 允许

- `prerequisite`
- `defined_by`
- `grounded_in`

逻辑关系 `supports`、`constrains`、`explains`、`contrasts_with` 只写在区分说明或必要说明，不进入 frontmatter，除非目标同时是真正前置依赖。

正文依赖/组件集合必须与 frontmatter 完全一致。

## 全自动编译协议

### 阶段 1：来源扫描与还原建模

完整阅读，再逐段扫描。先生成还原稿，记录核心结论、论证顺序、关键概念、建议、适用边界、来源定位、转写问题、内部矛盾和事实断言。不能成为 atom 的内容只从 atom 候选中丢弃，不能从还原稿丢弃。

### 阶段 2：独立来源复核

优先使用独立 subagent，只给原始来源，不泄露第一遍候选。复核者独立判断核心机制、粒度上限、证据不足和丢弃项。

### 阶段 3：CLC、命名、复用与图审查

先完成 CLC 22 大类全表路由与术语核验，再独立检索 `atoms/`：

1. 正式 CLC 主类、邻接类与对应专业术语。
2. ID、标题和 aliases。
3. 函数签名中的输入、操作和输出。
4. 中介机制、角色和成立条件。
5. 相邻机制和可能系统 atom。
6. 跨领域同构。

即使依赖零重叠，也必须完成名称、别名、机制和跨域检索。

### 阶段 4：主控裁决

主 agent 自动比较各结果。争议时默认：

- 少建而不是多建。
- 收窄边界而不是强行脱域。
- 降低证据强度而不是学术化包装。
- 保留来源原貌而不是静默修复。
- 丢弃建议和话术而不是把它们命名成机制。

### 阶段 5：整批建模

先完成批内去重、components、depends_on、逻辑关系和领域绑定。写入前保存所有待修改旧文件的原始字节，先在同目录临时文件中生成并校验完整批次，再原子替换目标。不能先写一批孤立 atom，事后补图。

### 阶段 6：双层验证与自动返工

写入后运行：

1. 原文替代盲测：只看原文的审题者提出覆盖核心主张、论证链、关键概念、建议与适用范围、证据边界与文本歧义的问题；只看还原稿的答题者回答；裁判对照原文。所有核心问题必须通过，总体语义正确率至少 90%，反向理解、把作者断言当事实、静默修复矛盾均为 0。
2. atom 污染门：检查文章观点、案例、话术和只有建议没有结果证据的内容没有被包装成通用机制。
3. atom 语义检查和确定性 validator。准备升级 `stable` 时，必须由未参与成稿的 subagent 独立复核来源忠实、术语命名、机制身份、粒度、复用、假迁移和证据门，并记录复核范围。

失败时自动修复并重跑，最多两轮；仍失败时恢复被修改旧文件的原始字节并删除本轮新增，不触碰无关用户文件。

## 复用与重构

旧 atom 准确承载候选必须同时满足：

- 函数签名兼容。
- 中介机制相同。
- 边界兼容。
- 调用功能相同。
- 依赖结构兼容。

处理顺序：

1. 精确承载：复用并追加来源、证据和新绑定。
2. 同机制但旧边界过窄：升级旧 atom。
3. 候选混入多个函数：拆分。
4. 已有 atom 实质重复：合并，保留追溯。
5. 无法承载且候选独立可复用：新建 draft。
6. 没有独立复用价值：丢弃，不建文件。

## 状态与自动维护

### draft

任一情况成立即保持 draft：

- 只有一份来源。
- 存在非空待验证项。
- 只有机制推演，没有独立材料验证。
- 语义审查或硬校验未全部通过。

### stable

必须同时满足：

- 硬校验与独立语义审查通过。
- 机制身份、函数签名和边界无未决冲突。
- `待验证：无`。
- 至少两个路径 identity 与内容哈希都不同、且本地可解析的来源文件支持同一机制，且至少一个绑定得到带证据定位的独立材料验证；或存在可核验权威原理来源并有一份实际领域材料验证。

### deprecated / merged

保留来源和去向。合并目标与原因写在必要说明，不新增 frontmatter 字段。

每次 compile 后，AI 自动重新检查相关 draft 的证据、状态、重复和边界，不需要用户维护。

没有未参与成稿的独立审查者时，不得把 atom 升级为 `stable`。

## 校验与失败恢复

确定性 validator 检查：

- `reconstructions/` 的唯一来源、source SHA-256、固定六节、替代验收状态、atom 链接与来源反向一致性。
- CLC 表至少覆盖 22 个基本大类，tag 必须为正式 `CODE/类名`，不接受裸学科词。
- 七字段、字段类型、ID/文件名、status、source、tags、aliases。
- 依赖/组件目标、允许的关系类型、非空理由、正文同步、自环和联合循环。
- 章节名称、顺序、函数签名、机制链、角色表、成立条件、迁移三边界和调用方式。
- 必要说明八个标签、命名依据、别名同步、独立语义复核和领域绑定验证状态。
- stable 是否仍有待验证项、重复/未解析来源或缺少带证据定位的独立材料绑定。
- deprecated/merged 是否保留可解析去向与迁移理由。
- merged 去向链是否无环并最终终止于活动 atom。

validator 不能证明内容为真。语义真实性由独立来源复核和主控裁决负责。

失败恢复：

- 自动修复当前批次引入的机械错误。
- 不用 `git reset --hard`、`checkout --` 或覆盖无关改动。
- 修复失败时恢复当前批次修改过的旧文件原始字节并删除无效新增，保留写入前完整状态，并在最终回复说明。

## 用户可见输出

最终只向用户报告：

- 读取的来源。
- 生成或更新的来源还原稿及原文替代盲测结果。
