# 原子名称

## 场景用途

它在什么具体场景中使用。回答什么问题。不要写"这是一个关于XX的概念"——写"当需要判断/解释/处理XX时用到它"。

这是维特根斯坦原则的核心：**意义在于用法。** 同样的词在不同场景中是不同的知识对象——"成本"在企业决策中属于经济学（F0），在注意力分配中属于心理学，在候鸟迁徙能量预算中属于生态学（Q4）。场景用途决定了它属于哪个学科方向。

## CLC方向

按场景用途归入的学科方向。取自 `docs/12-foundation-grounding.md` 分类框架。

例如：`F0 经济学`、`Q14 生态学`、`系统论`（跨学科）、`领域节点（不脱域）`

如果不确定，写判断依据——这个词在这个场景中回答哪个学科的核心问题？

## 定义

按场景用途写——不是词典定义，而是"在这个场景中它是怎么被用的"。

定义 section 的内部结构：

1. **一句定义正文**（不加粗）：用一句话说清它是什么。不要写"这是一个关于XX的概念"——写它做什么、在什么条件下、产生什么效果。
2. **"与XX的区别"**（按需）：逐个列举与相近概念的边界——"与'低调'的区别：..."。每个区别解释为什么看似相同实则不同。
3. **"排除："收尾**：一句话总结什么不属于此概念的范畴。可以以逗号分隔多项，也可以编号列表。

好的定义回答四件事：
1. **在什么场景里用**
2. **和什么对立**（边界由和它相邻但不等于它的概念定义）
3. **排除什么**（什么看起来像但不属于）
4. **一句话说清**（让没读过原文的人能理解）

## 边界

它不是什么。容易和什么混淆。在当前的场景用途下，它排除什么。什么看起来像它但实际不属于。

## 机制核心（如果是机制原子）

它到底发生了什么。写清因果链或结构约束。

**格式要求：机制核心正文包裹在 ` ```text ``` ` 代码块中。** 代码块内为纵向叙事流——用 `→` 箭头串联因果，缩进表示层级，读者从头读到尾一口气读完。正文中的因果关系通过缩进和箭头自然展开，不使用 Markdown 加粗标题（如 `**因果链：**`）将内容切分为独立段落。

**核心原则：**
- `→` 箭头串联因果，缩进表示层级
- 叙事自上而下推进——先发生什么、接着什么、最终什么
- 以下为常见模式参考，AI 根据机制的实际形态选择或创造合适的结构

参考示例（非穷举）：

*线性因果 — 机制是从因到果的单一链条：*
```text
前置条件：<...>
  → 关键操作：<...>
    → 阶段一：<...>
      → 可观察结果：<...>
```

*信号合成 — 机制涉及多方信号接收、认知合并、并行效应：*
```text
前置条件：<...>
  → 关键操作：<...>
    → 观察者同时接收两种信号：
        信号A（...）：<...>
        信号B（...）：<...>
      → 认知合成："A" + "B" = "合成的感知"
        → 效应展开：
            1. 效应名（解释）→ 表现
            2. 效应名（解释）→ 表现
          → 可观察结果：<...>
```

*条件分叉 — 机制在某个节点根据条件走向不同路径：*
```text
触发条件：<...>
  → 条件判断：<...>？
    ├── 路径A → ... → 结果A
    ├── 路径B → ... → 结果B
    └── 路径C → ...
      → 后续展开：<...>
```

以上模式可组合、可嵌套、可按需创造新结构。所有结构共同遵循：纵向叙事流、`→` 缩进层级串联因果、正文自然展开不分段加粗标签。

写法规则：
- 用 `→` 箭头串联因果，缩进表示层级
- 条件分叉使用树状字符 `├──` 和 `└──`，分叉后继续缩进
- 多信号用"信号A""信号B"分列，各附来源
- 合成瞬间用引号包裹公式（如 `"有能力" + "不释放" = "被收束的能量"`）
- 并行效应编号列出，每个效应配一句解释
- 单线程因果链可简化，但仍保持缩进叙事流

概念原子和领域节点不写此 section。

## 参与角色（如果是机制原子）

哪些东西参与了这个机制。每个角色承担什么功能。**必须用表格。**

| 角色 | 说明 |
|------|------|
| 角色A | 它在机制中起什么作用 |

概念原子和领域节点不写此 section。

## 成立条件（如果是机制原子）

这个机制在什么条件下运作。

- 条件1
- 条件2

概念原子和领域节点不写此 section。

## 迁移边界

**必须包含三层，缺一不可：**

**可迁移到：** 正面迁移示范。列出至少2个可迁移的领域，每个给出：领域名 → 角色映射 → 迁移后的效果。这一层告诉读者"这个机制可以怎么用"。

**不能迁移到：** 负面边界。列出什么领域/场景不适用，每种情况说明为什么。特别警惕"句式相同但机制不同"的假迁移。

**假迁移警惕：** 防误用警告。给出一个具体的错误示例——读者最容易把这个机制误用到什么地方，以及为什么那是错的。用一句话给出判断标准。

概念原子和领域节点不需要此三层格式，写"（概念原子，不适用）"或"（领域节点，不适用）"。

## 复用价值

**第一段：机制价值。** 这是原文中"XX"表述的机制化表述。它将"XX"这一直觉提炼为可操作的因果链。说明它可以被哪些类型的分析调用。

**第二段：调用方式（必写）。** 以"调用方式："开头，给出具体的、可操作的检查清单。格式为：当我要判断/解释"XX"时，检查 (a) ... (b) ... (c) ...。让未来读者（人或者 AI）知道在什么场景下、用什么 checklist 来调用这个机制。

## 领域绑定（如果是脱域机制原子）

列出它在不同领域中的角色映射。

| 领域 | 角色A 对应 | 角色B 对应 |
|------|-----------|-----------|
| 领域1 | 具体对象 | 具体对象 |
| 领域2 | 具体对象 | 具体对象 |

如果暂时只有一个领域绑定，写清"当前仅在X领域中验证"。

## 脱域说明

如果是从领域内容中提取的脱域原子：
- 原领域是什么
- 提取中去掉了哪些领域专属内容
- 保留了哪些跨领域机制
- 为什么可以迁移（不只是换词，而是共享机制）

如果当前是领域绑定节点：写"当前未脱域"。

## 区分说明

为什么不是已有原子。列出检查过的旧原子或本批候选，说明它们能不能承载，为什么不能。

## 必要说明

**必须包含以下子项：**

**学科来源：** 说明该机制的学术脉络。列出相关的学者、理论、研究（如"代价信号理论，Zahavi 1975""自我决定理论，Deci & Ryan 1985"）。如果由当前材料触发的概念补齐，写"由当前材料触发的概念补齐"。如果无明确学科来源，写"无"。

**常见别名：** 列出该机制/概念在原文或其他语境中的同义表达。至少列出2个。格式为逗号分隔列表。

其他可按需补充：必要例子、限制条件、来源补充、迭代入口、本批新建术语声明。

## 依赖

> 此节必须与 frontmatter `depends_on` 一致。使用 `[[atom-id]] - 语义关系类型：理由` 格式。即使为空也必须写"无"。

- 无
- 如果有依赖，写成：`[[依赖atom-id]] - prerequisite：为什么必须先理解它`
- 语义关系类型：prerequisite / defined_by / grounded_in / supports / constrains / explains

## 组件

> 此节必须与 frontmatter `components` 一致。使用 `[[atom-id]] - 语义关系类型：理由` 格式。即使为空也必须写"无"。

- 无
- 如果有组件，写成：`[[组件atom-id]] - composed_of：它承担当前原子的哪一部分`
- 语义关系类型：composed_of / implements / instantiates
- 复合原子的组件链必须能追溯到 `components: []` 的基础原子
